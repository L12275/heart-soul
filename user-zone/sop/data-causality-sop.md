# 数据因果链 SOP（强制标准）

> **核心理念：数据只是表现，实际只能看数据被什么工序形成。**
> 没有工序配套证明因果的数据，都没看的必要。

---

## 1. 什么时候必须用此SOP

以下场景，**禁止直接说结论**，必须先走完此SOP：

- 测试数据出现异常（429增多、成功率下降、密钥分布异常）
- 数据与预期不符（脚本逻辑说A，实际结果B）
- 任何涉及"为什么"的追问
- 任何涉及时序、间隔、状态流转的分析
- 任何需要判断"问题出在哪"的场景

---

## 2. 强制输出规则（三步走）

### Step 1：完整工序还原

把从起点到终点的**每一步执行**都列出来，包括：

- **时间戳**：每个步骤发生时的时间点（相对时间或绝对时间）
- **横纵间隔**：纵向间隔（同一密钥两次调用之间的时间）和横向间隔（密钥切换之间的时间/冷却时间）
- **状态变量**：每个密钥的 minuteCalls、active429、cooldownUntil、lastCallTime 在每个步骤的值
- **决策点**：代理在每一步做了什么选择、为什么选择

**格式示例：**

```
[0s] 代理启动: keyIndex=0(K1), 全部12密钥可用
     纵向间隔MIN_KEY_INTERVAL=400ms | 横向冷却HORIZONTAL_COOLDOWN=800ms

[+0.1s] 请求1进入processQueue
     检查K1: minuteCalls=0<35✓ | 冷却到期✓(cooldownUntil=0) | active429=false✓
     → 选中K1, 发上游请求, 等待响应

[+0.3s] K1返回429
     K1: minuteCalls=1, active429=true, cooldownUntil=now+800ms
     keyIndex → K2

[+0.4s] 请求2进入
     检查K2: minuteCalls=0<35✓ | 冷却到期✓ | active429=false✓
     → 选中K2...
```

### Step 2：把工序和数据的因果关系标出来

对每一步，回答：

- **这一步导致了什么数据变化？**（哪个计数器+1，哪个状态翻转）
- **下一步的数据依赖上一步的什么状态？**（如果上一步状态不同，下一步会怎样）
- **有没有代码路径没被走到的？**（哪些if分支被跳过了，为什么）

### Step 3：找出与数据表现矛盾的工序环节

对比实际数据 vs 预期数据，逐个排除：

- **预期**：根据代码逻辑，这一步应该走哪条分支？
- **实际**：stats/日志显示走了哪条分支？
- **矛盾点**：如果预期≠实际，是哪个条件判断出了问题？

## 3. 横纵间隔必须显式标注

在任何涉及密钥调用的流程描述中，**必须标注**：

| 间隔类型 | 含义                   | 代码位置                | 默认值     |
| ---- | -------------------- | ------------------- | ------- |
| 纵向间隔 | 同一密钥两次调用之间的最小时间      | MIN_KEY_INTERVAL    | 400ms   |
| 横向冷却 | 密钥被调用后，再次被选中的冷却时间    | HORIZONTAL_COOLDOWN | 800ms   |
| 窗口长度 | minuteCalls计数重置的时间窗口 | KEY_WINDOW_MS       | 60000ms |
| 卷尺上限 | 单密钥连续调用上限            | MAX_CALLS_PER_KEY   | 35次     |

**关键关系：**

- 纵向间隔保护密钥不被NVIDIA限流（400ms = ~150RPM，留余量到35次/分）
- 横向冷却防止同一密钥被连续调用（即使卷尺逻辑成功不移动指针）
- 窗口到期后minuteCalls归零，active429清除，密钥重新可用

## 4. cooldownUntil 的隐藏陷阱（已知bug）

**问题：** setInterval重置定时器（每2秒运行一次）只检查：

```javascript
if (now - keyStates[i].minuteWindowStart >= KEY_WINDOW_MS) {
  keyStates[i].minuteCalls = 0;
  keyStates[i].minuteWindowStart = now;
  keyStates[i].active429 = false;
}
```

**但没有清除 cooldownUntil。**

**后果：**

- 密钥被调用后，cooldownUntil = 调用时间 + 800ms
- 窗口到期后，minuteCalls=0、active429=false，但cooldownUntil仍然有值
- processQueue检查 `cooldownUntil > now` 时，如果时间还没过800ms，会错误地跳过该密钥
- 但800ms很短，所以实际影响有限——**除非间隔超过800ms的测试场景**

**排查方法：**
当发现"密钥理论上可用但实际被跳过"时，检查：

1. cooldownUntil是否有值
2. 是否在窗口重置时被清除
3. 如果没被清除，手动计算 `now - lastCallTime` 是否超过800ms

## 5. 检查清单（每次分析前过一遍）

- [ ] 是否列出了从起点到终点的**每一步**？
- [ ] 是否标注了每步的**时间点和间隔**？
- [ ] 是否记录了每个密钥的**完整状态变化**？
- [ ] 是否标出了**决策分支**和走哪条路？
- [ ] 是否对比了**预期vs实际**？
- [ ] 是否有**矛盾点**被遗漏？
- [ ] 是否有**代码没走到的分支**被忽略了？

## 6. 反模式（绝对禁止）

| 反模式        | 例子                | 正确做法                              |
| ---------- | ----------------- | --------------------------------- |
| 只看最终数据不看过程 | "429=16，成功率84.8%" | 先列出这16个429是怎么产生的                  |
| 跳过时序只说静态状态 | "K1被429标记"        | "K1在第N秒被调用，429后cooldownUntil=..." |
| 不标注间隔参数    | "密钥被切换了"          | "横向冷却800ms到期前，该密钥不可被选中"           |
| 用猜测替代因果    | "可能是IP被封"         | "排除A→排除B→剩余C→C的证据是..."            |
| 代码只看结果不看分支 | "逻辑是对的"           | "代码第X行走了A分支而不是B分支，因为..."          |

---

## 7. 此SOP的触发条件

在任何对话中，如果出现以下情况，**必须调用此SOP**：

- 用户说"说完整"、"展示逻辑"、"为什么"
- 数据结果与用户预期不符
- 出现了新的异常数据点
- 需要解释因果关系的任何场景

**此SOP是强制标准，不是可选参考。**
