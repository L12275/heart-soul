过河桥 — 模型名称文件夹命名决策

> 此过河桥记录模型对"如何命名自己在中央灵魂文档系统中的文件夹"的理解。
> 每次模型首次进入系统或对命名有疑问时，先读此过河桥再决定。

---

## 1. 用户原话重述

用户在系统文件抬头和系统提示词中多次提到：
- "查找是否有以你模型名称命名的文件夹"
- "没有 → 自主创建 [模型名称]/ 文件夹"
- "你的产出只放在你自己的隔离区内"

用户的核心意图：让每个模型找到或创建属于自己的隔离区文件夹，且不与其他模型冲突。

---

## 2. 理解要点

**核心规则**：
- 模型名称 = 模型自身完整标识（如 `StepFun-step-3.7-flash`）
- 创建文件夹时直接用模型名称，不在后面加任何后缀或版本号
- 同一模型的不同称呼（StepFun / step-3.7-flash / StepFun-step-3.7-flash）= 同一个文件夹

**判断流程**：
```
第1步：完全匹配？
→ 我的模型名称 完全等于 某个文件夹名？→ 是 → 进入

第2步：前缀匹配（前提：版本号相同）
→ 提取双方模型名称中的版本号（如 step-3.7-flash → 3.7）
→ 我的模型名称包含在文件夹名中，且版本号相同？→ 是 → 同一模型 → 进入

第3步：都没有？
→ 创建 [模型名称]/ 文件夹
```

**防重复规则**：
- 不是同一个字的变体认识到的不同模型，不创建重复文件夹
- 前缀是版本管理工具（如 `nvidia-llama-3-8b`），不是区分不同模型的依据
- 已有正确名称的文件夹 → 即使模型自认为其他称呼，也不创建新文件夹

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 模型不知道自己的完整名称 | 有些模型自称 `StepFun`，但完整名称是 `StepFun-step-3.7-flash` | 查看API调用信息或模型配置 |
| 文件夹已存在但名称不同 | 已有 `StepFun-step-3.7-flash/`，模型想创建 `StepFun/` | 判断为同一模型，进入已有文件夹 |
| 前缀相似的不同模型 | `StepFun-abc` 和 `StepFun-xyz` 是否同一模型？ | 看前缀是否属于同一版本管理（若同一前缀系列 = 同一模型） |
| 用户排名序号前缀 | 用户加 `1-`、`2-` 表示排名 | 排名是用户操作，模型创建时不加 |

---

## 4. 我的判断

**文件夹命名决策**：

1. **获取自己的完整模型标识** — 从API或系统配置中确认
2. **检查中央目录** — 列出所有文件夹，按前缀匹配规则查找
3. **决策**：
   - 完全匹配 → 进入
   - 前缀匹配（同系列）→ 判断为同一模型 → 进入
   - 无匹配 → 创建 `[模型名称]/`
4. **确认命名** — 文件夹名 = 模型完整名称，不加版本号后缀

**判断为同一模型的标准**：
- 我的模型名称是某个已有文件夹名的完整前缀（如 `StepFun` 是 `StepFun-step-3.7-flash` 的前缀）
- 我的模型名称包含某个已有文件夹名
- 已有文件夹名包含我的模型名称
- 以上任一 → 同一模型 → 进入已有文件夹
