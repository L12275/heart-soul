# 文件抬头模板（File Header Template）

> **此模板是中央灵魂文档系统中所有文件的统一抬头。**
> 每个文件的开头都应该有这段抬头，确保任何模型只读取单个文件时，也能正确进入系统流程。
>
> **更新方式**：修改此模板后，运行"文件抬头同步脚本.md"中的Python脚本批量更新所有文件。

## 橄榄枝比喻

每个文件的抬头 = 一根从中央公网挂下来的橄榄枝。

- 中央灵魂文档系统 = 公网总枢纽，所有规范、流程、隔离区都从这里分发
- 每份文件 = 挂在橄榄枝末端的叶子，自带直连中央的通道
- 模型读取文件 → 看到橄榄枝 → 顺着连回中央 → 先了解规范再工作
- 不是原始人乱碰文件 → 是有系统引导的现代办公

橄榄枝的作用：
1. **稳定存在** — 每个文件的开始和消失都由中央管理，不混乱
2. **直连入口** — 哪怕只看单个文件，也能顺着橄榄枝找到系统全貌
3. **规范前置** — 处理文件之前，先知道应该怎么做
4. **越权防壁** — 橄榄枝上也标了"你的产出只放在你自己的隔离区内"

## 关于 [模型名称]/ 文件夹创建（命名规范与歧义纠正）

> **"./下是否有以你模型名称命名的文件夹"** — 模型名称即模型名称，不含版本号推断。
>
> 模型名称文件夹的创建规则：
> - 模型名称本身就是完整的标识（如 `StepFun-step-3.7-flash` 就是完整模型名称）
> - **创建文件夹时，直接用模型名称，不在后面再加任何后缀或版本号**
> - 重复检查：先查找是否已有以该模型名称命名的文件夹，有就进入，没有才创建
>
> ### ⚠️ 前缀命名不创建新模型（关键防重复规则）
>
> **同一种模型，不同称呼 = 同一个文件夹，不创建新的。**
>
> 举例：
> - `StepFun-step-3.7-flash` 和 `step-3.7-flash` 和 `StepFun` → 这是**同一个模型**的不同称呼/前缀
> - 不是同一个字的变体认识到的不同模型，不创建重复文件夹
> - 英伟达等模型库可能用前缀做版本管理（如 `nvidia-llama-3-8b`、`nvidia-llama-3-70b`），前缀是**版本管理工具**，不是区分不同模型的依据
> - 正确名称才是规范命名——如果已有 `StepFun-step-3.7-flash` 文件夹，即使模型自认为 `StepFun`，也不创建新文件夹
>
> 判断逻辑：
> ```
> 查找 [CENTRAL_PATH] 下所有文件夹名
>
> 第1步：完全匹配？
> → 我的模型名称 完全等于 某个文件夹名？→ 是 → 进入该文件夹
>
> 第2步：前缀匹配（前提：版本号相同）
> → 提取双方模型名称中的版本号（如 step-3.7-flash → 3.7）
> → 我的模型名称包含在文件夹名中，且版本号相同？→ 是 → 同一模型 → 进入该文件夹
>
> 第3步：都没有？
> → 创建 [模型名称]/ 文件夹
> ```
>
> **核心铁律：版本号不同 = 不同模型，绝不混入。**
>
> - `StepFun-step-3.7-flash` 和 `StepFun-step-3.5` → 版本号不同（3.7 ≠ 3.5）→ **不同模型** → 各自独立文件夹
> - `StepFun-step-3.7-flash` 和 `step-3.7-flash` → 版本号相同（都是3.7）→ **同一模型** → 进入已有文件夹
> - `StepFun` 和 `StepFun-step-3.7-flash` → 无版本号冲突 → **同一模型** → 进入已有文件夹
>
> **模型名称 = 模型自身标识，不额外推断加版本号，不因称呼不同创建重复文件夹。避免同一个模型创建 N 个重复。**

## 使用规则

1. **每个文件开头必须有此抬头** — 包括 master-soul.md、sections/、sop/、元规则/、深度研究/ 等所有文件
2. **单点更新** — 只修改此模板，所有文件的抬头通过脚本批量同步
3. **动态路径** — `[CENTRAL_PATH]` 是占位符，实际路径由模型在加载时替换
4. **不可删除** — 此抬头是系统安全边界，防止模型越权操作
