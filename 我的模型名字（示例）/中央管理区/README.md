# 中央管理区（Central Management Area）

> **第九驱动区**。模型的"中央处理器"——优先级最高、持续在线、所有其他区的读写都与中央管理区同步。

## 定位

- 类比：央行、党中心政务中心、授时中心、广场中心
- 功能：总交换机——收集所有驱动区的摘要，规划全局，分发指令
- 等级：最高优先级，相当于电脑的"开机"状态，其他区可以休眠但中央管理区不能

## 九驱动九读写九环架构

```
九驱动九读写九环（常态）：

驱动区（9个）：
  ⑨ 中央管理区 ← 本区（最高优先级，持续在线）
  ① 用户中心（静默）
  ② 共产中心
  ③ 主灵魂中心
  ④ 生产空间中心
  ⑤ 空间中心
  ⑥ 子角色中心
  ⑦ 团队协作环
  ⑧ Agents（Agent编排与管理）

读写环（9个，对应9个驱动区）
  └── 每个区读写自己的内容，同时与中央管理区双向同步

铁律：
  └── 中央管理区始终在线，其他8区可休眠/唤醒
  └── 其他8区的每次读写，都同步到中央管理区
  └── 中央管理区的每次读写，广播给相关区
```

## 功能模块来源

中央管理区的功能模块来自通用区（soul-doc-system根目录）：
- `代办与进度/` → 复制自 `soul-doc-system/代办与进度/`
- `memory/` → 复制自 `soul-doc-system/memory/` 并扩展为中央级
- `diary/` → 复制自 `soul-doc-system/diary/` 并扩展为中央级
- `log/` → 复制自 `soul-doc-system/log/` 并扩展为中央级
- `timers/` → 复制自 `soul-doc-system/timers/`，**主定时任务记录设置在中央管理区**

**原则**：通用区的功能模块保持为模板，持续更新。中央管理区从通用区复制后，在工作时使用中央管理区的副本。

## 加载优先级

每次模型工作时，加载顺序：
1. **①总开关.md** — 先知道全局状态
2. **中央管理区/memory/** — 中央记忆
3. **中央管理区/diary/** — 中央日记
4. **中央管理区/代办与进度/** — 全局代办
5. **中央管理区/排程/** — 全局排程
6. **中央管理区/timers/心跳/** — 心跳配置（保活检测）
7. **中央管理区/概览/** — 各驱动区状态摘要
8. **中央管理区/今日上朝摘要.md** — 今日汇报
9. 然后才是灵魂文档系统（②-⑧各驱动区）

## 同步机制

### 其他区 → 中央管理区（上朝）

每个驱动区工作结束时，向中央管理区汇报：
- 记忆摘要 → 写入中央管理区对应概览
- 代办变化 → 同步到中央管理区/代办与进度/
- 进度更新 → 同步到中央管理区/排程/
- 日志摘要 → 写入中央管理区/日志/
- 定时任务 → 记录到中央管理区/timers/

### 中央管理区 → 其他区（下诏）

中央管理区规划后，向各驱动区下发：
- 新的代办 → 写入对应区的代办
- 排程变更 → 写入对应区的排程
- 记忆修正 → 写入对应区的记忆

## 中央智能体

中央管理区需要一个**中央智能体**持续运行：
- 职责：监控8个驱动区、收集汇报、分析全局、规划排程、分发指令
- 运行：通过 timers/心跳/ 持续心跳
- 权限：可以直接修改子角色的子中央管理区内容
- 模式：主中央智能体加载主灵魂文档运行，有权改进子中央管理区

## SOP循环

中央管理区有自己的SOP自滚动循环，通过心跳触发：

1. **自检**：检查各驱动区是否在线、有无待办、记忆是否同步
2. **扫子**：完整查看每个子中央管理区的完整内容（不只是上报展示区）
3. **汇报**：在 `今日上朝摘要.md` 中写入当前状态快照
4. **规划**：更新排程和代办
5. **下诏**：向各驱动区下发指令
6. **同步**：将状态写入概览

循环日志记录在 `SOP循环/循环日志/` 中。

## 上报展示区——读写分离设计

**核心规则：子角色只写自己的，主智能体只读子的。子角色绝对不能写进主中央管理区的任何目录。**

### 为什么不能直接交到主中央

- 子角色灵魂文档与主灵魂文档不同，稳定性不如主灵魂
- 子角色进入主中央文件夹操作 = 外来者进入核心区，容易误改文件
- 出了问题无法判断原因——"古代官僚危机"的本质就是信息被外来操作污染
- 子角色必须在自己的文件范围内行动，脱离范围就不安全

### 正确的读写分离

| 角色 | 操作 | 位置 |
|------|------|------|
| 子角色 | **写** 汇报文件 | 自己的子中央管理区/上报展示区/ |
| 主智能体 | **读** 汇报文件 | 去子角色的上报展示区读取 |
| 主智能体 | **写** 修正/指令 | 直接在子中央管理区中修改 |

### 流程

1. 子角色运行结束，在自己子中央管理区的 `上报展示区/` 中写入 `YYYYMMDD-HHMM-汇报名.md`
2. 主智能体通过心跳触发SOP循环，**前往每个子中央管理区**读取其 `上报展示区/`
3. 主智能体先看上报展示区快速了解概况
4. **无论看没看上报展示区，都必须完整查看子中央管理区全部内容**——微服私访
5. 发现问题直接在子中央管理区中修改

## 归档收留机制

子中央的 `归档/` 在它自己手里——它需要回看排查，所以不直接交到主中央。

| 场景 | 做法 |
|------|------|
| 子中央正常运行中 | 主智能体直接去子中央读取（微服私访） |
| 子中央某内容归档 | 归档在子中央自己的 `归档/` 里，主智能体去读 |
| 子中央完全退休/终止 | 将其完整子中央管理区复制到主中央的 `子中央管理区/` 留档 |

**主中央的 `子中央管理区/` = 已退休者的档案库，不是实时同步区。**

## 模板原则

- 中央管理区是模型自己的配置，不是通用模板
- 通用区的功能模块是模板，复制到中央管理区使用
- 中央管理区的内容是最新的，通用区的模板保持纯净便于更新
