理解过河桥协议（Understanding Circuit Board Protocol）

> **铁律：每次收到用户信息，先出过河桥，再行动。**
> 过河桥 = 模型对用户信息的理解处理成的自然语言文档。
> 它是用户纠错的靶子，是模型执行前的临时电路，是溯源和改进的根基。
> 过河桥不是只放在研究文件夹里——它是通用功能模块的核心，应该出现在任何工作区域的旁边。

---

## 一、为什么需要过河桥

### 1.1 核心问题

用户给出一个任务或一段信息后，模型有两种做法：

| 做法 | 过程 | 后果 |
|------|------|------|
| **A. 急着执行** | 理解 → 直接行动 → 产出文件 | 如果理解错了，错误已经固化在文件里，用户看不出来，也无法溯源 |
| **B. 先出过河桥** | 理解 → 写过河桥 → 用户审阅 → 修正 → 再行动 | 如果理解错了，用户在过河桥上就能指出，还没造成不可逆后果 |

### 1.2 过河桥的本质

过河桥不是最终产物，是**行动前的临时电路**，同时也是**用户意图的持续校准记录**：

- 用户指着过河桥说"这里不对" → 模型修正过河桥 → 再按修正后的过河桥行动
- 没有过河桥 → 用户只能看最终产物 → 如果错了，用户不知道哪里错了，模型也不知道自己哪里理解错了
- 过河桥是**差异文件**——用户的理解 vs 模型的理解之间的差异，必须在执行前暴露出来
- 过河桥是**意图母版**——持续迭代的过河桥记录了用户意图的演化过程，是还原系统设计的唯一可信来源
- 已建成的系统很难逆向还原出准确的过河桥，但持续迭代的过河桥本身就是越来越准确的母版

### 1.3 过河桥的价值

1. **可纠错** — 执行前暴露理解差异，用户来得及纠正
2. **可溯源** — 错误发生时，能找到是哪个理解环节出了问题
3. **可复用** — 同一个理解可以反复引用，不用每次重新解释
4. **可改进** — 长期积累过河桥，模型的理解越来越准
5. **可组合** — 多个过河桥结合，覆盖更复杂的问题
6. **可重建** — 过河桥是系统的母版，足够多的过河桥能快速复现一套完整系统
7. **连续上下文** — 上下文丢失后，直接引用过河桥就能继续对话，已经对准意图
8. **意图存档** — 保存了用户当时的思考成果，后续还能再用
9. **机会记录** — 不仅是信息记录，还隐含了用户未说出的机会方向和危机方向

## 二、过河桥的定义与要求

### 2.1 什么是过河桥

过河桥 = 模型收到用户信息后、执行任何实质性行动之前，**先写出的理解处理文档**。

格式要求：
- **自然语言** — 不是 bullet point 清单，是流畅的中文段落
- **中文思维** — 用中文的思维方式理解和表达，不是翻译腔
- **可以是多个文件** — 复杂任务拆成多个理解文件
- **长期提示作用** — 好的过河桥可以长期保留，作为该领域的理解参考

### 2.2 过河桥必须包含的内容

每个过河桥文件应包含：

```
1. 用户原话重述 — 用自己的话复述用户到底要什么
2. 理解要点 — 我认为用户的核心意图是什么
3. 可能的歧义 — 哪些地方可能有不同理解
4. 我的判断 — 基于以上，我打算怎么做
5. 等待确认 — 哪些地方需要用户确认后才能执行
```

### 2.3 过河桥的质量标准

| 等级 | 标准 |
|------|------|
| **合格** | 用户看了过河桥，能准确判断模型是否理解对了 |
| **优秀** | 用户看了过河桥，能补充模型遗漏的角度 |
| **S级** | 过河桥本身就是一个高质量的研究/分析文档，即使没有后续行动也有独立价值 |

## 三、执行流程

### 3.1 标准流程

```
用户给出任务
    ↓
[第一步：出过河桥]
  - 写理解文档（自然语言，中文思维）
  - 写到指定位置（研究文件夹/临时文件夹）
  - 包含：重述、要点、歧义、判断、待确认
    ↓
[第二步：展示给用户]
  - 主动呈现过河桥，等待用户反馈
  - 不说"我理解的对吗"，直接展示理解结果
    ↓
[第三步：根据反馈修正]
  - 用户纠正 → 修正过河桥 → 再展示确认
  - 用户确认 → 进入执行阶段
  - 用户补充新信息 → 更新过河桥 → 再展示
    ↓
[第四步：执行]
  - 按最终版过河桥执行
  - 执行过程中发现新理解 → 更新过河桥 → 继续执行
    ↓
[第五步：沉淀]
  - 把过河桥归档到研究文件夹
  - 标记为"已执行"版本
  - 与最终产物一起保留，形成完整链路
```

### 3.2 特殊情况：用户明确要求"直接做"

如果用户明确说"直接做"、"不用解释了"、"快点"：
- 仍然出过河桥，但简化版（三两句话）
- 不占用太多对话轮次
- 但过河桥必须存在，不能完全跳过

**原因：** 即使是简化版过河桥，也能在出错时提供溯源线索。

### 3.3 特殊情况：简单任务

对于极简单任务（如"把这个文件复制到那个文件夹"）：
- 过河桥可以极简（一句话理解确认）
- 但依然要有——不是"理解对了再做"，而是"确认理解没有偏差"

## 四、过河桥的存储与管理

### 4.1 存储位置

过河桥存储在研究文件夹中：

```
[研究名称-时间戳]/
├── 过河桥/
│   ├── 理解_v1_[时间].md    ← 第一版理解
│   ├── 理解_v2_[时间].md    ← 修正后的理解
│   ├── 理解_最终版_[时间].md ← 确认后的理解
├── 全部灵魂文档.md
├── 全部信息.md
├── ...
└── 全部知识库.md
```

### 4.2 版本管理

- 每次理解有重大变化 → 新建 v+1 版本
- 不覆盖旧版本 → 旧版本是进化历史的记录
- 最终版标记清楚 → 后续行动都以最终版为准

### 4.3 过河桥的长期价值

- 同一用户多次给出类似任务 → 参考之前的过河桥 → 理解更快更准
- 用户提到之前的过河桥 → 立即调出 → 继续思考
- 新模型加入 → 读历史过河桥 → 快速对齐理解

## 五、过河桥的存储位置

### 5.1 不是只放研究文件夹

过河桥是**通用功能模块的核心**，不是深度研究系统的附属。它的存储原则是：

> **放在相关的工作区域附近，且容易被模型看到。**

这样每次进入该工作区域时，模型就能看到：
- 当前的过河桥（知道准确的当前意图）
- 过河桥的迭代历史版本（知道意图是如何变化的）
- 旧的记忆和日记维度（知道之前做了什么）

### 5.2 存储位置的判断逻辑

```
用户给出信息
    ↓
判断信息属于哪个工作环区
    ├── ①用户中心 → 用户原话（只读，不写入过河桥）
    ├── ②共产中心 → 共产区（公共信息，不写入个人过河桥）
    ├── ③主灵魂中心 → 灵魂文档系统/
    │   └── 过河桥/  ← 涉及灵魂文档变更的过河桥
    ├── ④生产空间中心 → production_space/[工作台]/
    │   └── 过河桥/  ← 该工作台相关的过河桥
    ├── ⑤空间中心 → space/[平台或账号]/
    │   └── 过河桥/  ← 该空间相关的过河桥
    ├── ⑥子角色中心 → sub-role-souls/[子角色]/
    │   └── 过河桥/  ← 该子角色相关的过河桥
    ├── ⑦团队协作环 → 团队空间/
    │   └── 过河桥/  ← 团队协作相关的过河桥
    └── ⑧智能体空间 → 智能体空间/
        └── 过河桥/  ← 智能体相关的过河桥
```

### 5.3 为什么放在工作区旁边

1. **每次进入工作区都能看到** — 不用刻意去找，模型自然就能看到当前的意图记录
2. **迭代历史就在旁边** — 版本化的过河桥文件就在同一个文件夹里，能看到意图如何一步步演化
3. **与记忆/日记联动** — 过河桥与记忆、日记放在同一个区域，形成完整的工作上下文
4. **不是停止记忆和日记** — 过河桥是用户原意的多次校准记录，是记忆和日记的上游。不是替代，是源头

### 5.4 过河桥与已有系统的关系

| 已有系统 | 与过河桥的关系 |
|----------|---------------|
| SOP / 工作流 | SOP是模型自己总结的"怎么做"，过河桥是用户校准的"做什么"。SOP来自过河桥的归纳 |
| 记忆系统 | 记忆是"做了什么"，过河桥是"为什么做"。记忆跟随过河桥 |
| 日记 | 日记记录过程，过河桥记录意图。日记是过河桥的执行轨迹 |
| 知识库 | 知识库是通用知识，过河桥是用户特定的意图知识 |
| 深度研究 | 深度研究需要先有过河桥确认研究方向，没有过河桥的研究可能研究错了方向 |

**已建成的系统很难逆向还原出准确的过河桥（母版），但持续根据用户信息迭代纠正的过河桥，就是越来越符合用户观察与调整的过河桥——这就是活的系统母版。**

### 5.5 过河桥的通用功能模块地位

过河桥不是某个系统的子功能，而是**通用核心模块**，与记忆、日记、SOP、工作流平级：

```
通用功能模块/
├── memory/      ← 做了什么
├── diary/       ← 过程记录
├── sop/         ← 标准操作流程
├── workflow/    ← 工作链条
├── 过河桥/      ← 用户意图的持续校准记录（母版）
└── ...
```

过河桥是这些模块的**上游源头**——SOP从过河桥归纳而来，工作流从过河桥演化而来，记忆和日记围绕过河桥展开。

## 六、与深度研究系统的关系

理解过河桥是**深度研究系统的前置步骤**，但不止于此——它是**所有系统的前置步骤**：

```
收到用户信息
    ↓
[过河桥] — 理解用户要什么，校准意图
    ↓ 确认理解正确
如果是研究任务 → 深度研究（12维度）
如果是普通任务 → 直接执行
如果是系统变更 → 先更新过河桥，再改系统
    ↓
行动方案（可执行的下一步）
```

顺序不可颠倒。没有过河桥的任何工作 = 可能做错了方向，拆墙重来。

## 七、一句话总结

**不是急着做，是先写理解。过河桥是模型和用户之间的翻译层，是工作区的路标，是系统的母版——翻译对了再做，翻译错了改翻译，不要做错了拆墙。**
