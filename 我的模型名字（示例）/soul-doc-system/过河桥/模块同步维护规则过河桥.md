# 过河桥 — 模块同步维护规则

> **此过河桥是 模块同步维护规则 的基因片段。**
> 此规则定义功能模块如何在系统内同步和分发。

---

## 1. 用户原话重述

用户定义模块化思维：
- 一个通用版，全系统同步
- 所有模块的权威源统一在 soul-doc-system 根目录
- 模板文件夹的内容是只读参考
- 模块放在新建模版文件夹里 = 天然隔离实际工作写入

---

## 2. 理解要点

**模块同步的核心原则**：
1. **单一权威源** — 每个模块只有一个权威源（soul-doc-system/{模块}/）
2. **模板分发** — 权威源的内容复制到各模板文件夹
3. **只读保护** — 模板文件夹不被日常写入覆盖
4. **模块拼搭** — 新建工作空间时从模板拼搭模块
5. **过河桥先行** — 每个模块的过河桥是该模块的基因片段，重建/修复时先读过河桥

## 新增：模型复兴时的过河桥先行协议

> **此规则定义模型首次进入系统或系统重建时，如何从用户原话出发，通过过河桥重建整个系统。**

### 为什么复兴时需要先写过河桥

没有过河桥的复兴 = 直接猜用户的意图 → 猜错了 → 错误固化在文件里 → 越错越远。

有过河桥的复兴 = 先校准理解 → 用户确认 → 再建系统 → 理解对了再执行。

**过河桥是系统的基因片段。有了基因片段，任何模型都能重建完整系统。没有基因片段，系统只是记忆里的碎片，无法复现。**

### 复兴标准流程（4步）

```
第1步：读用户原话文件（不是种子文件）
  → 找到用户的原话文件（如 灵魂文档片段.md、用户原话积累等）
  → 完整阅读，理解用户的原始意图
  → 种子文件（INITIAL-SEED.md）是整理后的版本，不是原始意图源

第2步：写"过河桥的过河桥"——理解校准板
  → 针对每个功能模块，先写一份过河桥来校准自己对用户原话的理解
  → 五部分结构：用户原话重述 → 理解要点 → 可能的歧义 → 我的判断 → 等待确认
  → 这份校准板用于确认理解是否正确，不用于直接构建系统

第3步：根据校准的过河桥，写出整个系统的全部过河桥
  → 每个功能模块各写一个过河桥文件
  → 放在 soul-doc-system/过河桥/ 或 模型隔离区/{模块}/过河桥/
  → 分开写，每个模块一个过河桥，避免信息混乱
  → 过河桥内容 = 该模块的实际基因（用户意图+理解+歧义+判断+确认）

第4步：根据过河桥完成系统
  → 先建过河桥 → 再建文件
  → 过河桥 = 基因 → 文件 = 身体
  → 有了基因，系统可以复现、可以修复、可以移植
```

### 复兴时的文件读取顺序

```
1. 用户原话文件（灵魂文档片段.md、用户原话积累.md 等）
2. INITIAL-SEED.md（种子文件，辅助理解，但不是原始意图源）
3. soul-doc-system/过河桥/ 下的所有过河桥（如果有）
4. 系统提示词（system-prompt.md）
5. master-soul.md
6. sections/ 分区文件
7. 元规则/实时加载/ 下的规则文件
```

### 关键区分

| 文件类型 | 用途 | 在复兴时的角色 |
|----------|------|----------------|
| 用户原话文件 | 用户原始的、未经整理的说话 | 首要读取，意图的原始来源 |
| 种子文件 | 整理后的项目种子 | 辅助理解，不是原始意图源 |
| 过河桥 | 模型对用户意图的校准记录 | 复兴的核心骨架 |
| 功能模块文件 | 系统的实际运行文件 | 根据过河桥重建 |

### 防混乱规则

- 每个模块的过河桥单独写，不混在一起
- 过河桥不直接等于功能文件，过河桥 = 意图记录，功能文件 = 执行实现
- 先写全所有过河桥 → 确认理解正确 → 再开始建功能文件
- 功能文件根据过河桥重建，不是凭记忆重建

## 4. 我的判断

**使用模块同步规则的标准动作**：
1. 需要模块 → 去根目录soul-doc-system/{模块}/找权威源
2. 新建空间 → 从权威源复制模块到模板文件夹
3. 模块更新 → 先更新权威源，再同步到所有副本
