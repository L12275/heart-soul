# 归档区（Archive）

> 用于模型自己的灵魂文档系统中，旧版功能模块的归档保留。
> 当需要推出新版功能模块时，先将旧版模块完整复制到归档区，
> 然后在原位置替换为新版模块，确保旧版不可丢失。

## 为什么需要归档区

功能模块升级不是瞬间完成的——新版模块可能还没完全准备好，
而旧版模块仍在被加载使用中。
类比服务器更新：暂停登陆 → 更新 → 恢复。模块切换也一样。

归档区就是"更新期间"的安全缓存：
旧版不删除（还在归档区），新版在原位置就位，
随时可以回滚，也可以保留历史版本供对比学习。

## 归档规则

1. **模块级归档** — 归档的是单个功能模块旧版，不是整个系统
2. **版本编号** — 按版本号和时间戳双重标识
3. **先归档再替换** — 旧版先复制进来 → 再在原位置更新新版 → 不得跳过归档
4. **可选对比** — 归档后可以对比新旧版差异，作为改进记录

## 模块切换操作流程

```
步骤 1：准备新版
  → 在模版位置或通用区准备好新版模块内容
  → 确认新版可用，但尚未替换旧版

步骤 2：归档旧版
  → 把当前使用的旧版模块完整复制到 archive/{模块}/v{版本号}_{日期}/
  → 保留旧版的全部子文件夹和文件，不做任何修改

步骤 3：替换为新版
  → 在原位置用新版覆盖旧版
  → 确认新版本能正常加载和使用

步骤 4：确认切换完成
  → 验证新版模块正常运行
  → 归档区保留旧版，不删除
  → 操作完成，清除临时文件
```

## 版本编号规则

```
v{主版本号}_{日期}

示例：
v1_20260624 — 第一版，归档于 2026-06-24
v2_20260625 — 第二版，归档于 2026-06-25
```

## 回滚流程

如果新版模块出现问题，无法正常使用：

```
1. 从 archive/{模块}/v{旧版号}_{日期}/ 复制旧版模块
2. 覆盖回原位置
3. 在原位置用旧版恢复服务
4. 排查新版问题，准备下一次替换
```

## 注意事项

- 归档区只存旧版模块，不存当前正在使用的版本
- 归档后旧版不可修改（修改应在新版中进行）
- 归档区的文件是只读历史记录
- 定期清理过旧的归档：保留最近 3-5 个版本即可
