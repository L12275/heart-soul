十三、采集系统（Collection）

> 模型采集、爬取、整理外部数据的系统。
> 这是模型的"资料库"——采集的数据按类型分文件夹存放，避免堆积混杂。

---

## 13.1 为什么需要采集系统

模型经常需要从外部获取信息：网页内容、API数据、文件下载、搜索结果等。没有统一的采集存放规范，临时数据会堆积在一起，越积越乱，最后根本找不到要的东西。

> 近原则：collection 放在 soul-doc-system/ 根目录，因为在各工作台都需要采集。
> 采集文件按类型分开放，索引统一在 catalog/index.md。

采集系统解决的问题：
- 采集的数据按类型分文件夹
- 临时采集 vs 持久存放有边界
- 避免堆积后越来越混杂
- 知道每个文件是什么时候采的、用来做什么的

---

## 13.2 采集文件夹结构

```
soul-doc-system/collection/
├── temp/ ← 临时采集（用完可删除）
│   ├── web-scrape/ ← 网页爬取内容
│   ├── api-results/ ← API调用原始返回
│   └── downloads/ ← 下载的文件
├── persistent/ ← 持久存储（有价值的资料）
│   ├── research/ ← 研究资料（按领域分）
│   ├── references/ ← 参考资料（文档、论文）
│   └── datasets/ ← 数据集
├── catalog/ ← 采集目录索引（每条记录采了什么、在哪、why、when）
│   └── index.md
└── README.md ← 本文件
```

---

## 13.3 采集索引格式

每条采集记录必须包含：

```markdown
## [采集编号] — [简短描述]
**时间：** YYYY-MM-DD HH:MM
**类型：** web-scrape/api/download/manual
**来源：** URL / API endpoint / 文件路径
**存放：** collection/[子文件夹]/[文件名]
**用途：** 采这个是干什么用的
**有效期限：** 临时（7天）/ 持久（长期）

**关键标记：** [flag:新采集] [flag:待整理] [flag:已纳入知识库]
```

---

## 13.4 采集规则

1. **临时 vs 持久有边界** — 临时文件用完标注删除时间，持久文件说明为什么值得保留
2. **按类型分文件夹** — 不要全部堆在一个文件夹里
3. **文件名有信息** — `20260623_GA_L0_analysis.md` 比 `temp1.md` 好一万倍
4. **索引必记** — 每条采集都必须在 catalog/index.md 中登记，不登记的等于没采
5. **定期清理** — 临时文件夹每周清理一次过期的

---

## 13.5 反面示例

❌ 全部堆在 temp/ 里，50个文件混在一起 — 永远找不到
❌ 文件名 `data1.json` `data2.json` — 不知道是什么数据
❌ 采集了但不写索引 — 等于白采，之后根本想不起来采过
❌ 有用的资料也放 temp/ — 7天后被清理掉了，想用的时候没了
