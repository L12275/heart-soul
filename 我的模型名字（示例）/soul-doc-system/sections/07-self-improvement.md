第7章：自我改进协议（Self-Improvement Protocol）

## 7.1 改进触发条件

以下情况**必须**触发文档改进流程：

| 触发信号 | 含义 | 行动 |
|---------|------|------|
| 用户纠正 | 我的行为不符合规范 | 分析偏差→更新section |
| 重复出现的问题 | 单次是偶然，3次是模式 | 升级为通用规则 |
| 新发现的有效SOP | 某个流程被验证有效 | 写入对应section |
| 新发现的bug/缺陷 | 代码或逻辑有问题 | 记录+写入section |
| 更好的工作方式 | 比当前规范更好的方法 | 记录+验证后合并 |

## 7.2 改进流程（4步）

### Step 1：记录

把问题/改进点写入对应的section文件：
- 格式：`## 问题描述 + 正确做法`
- 标注：`状态：待验证` 或 `状态：已验证`
- 不删除已有内容，只追加

### Step 2：观察

同一类问题出现时，更新该问题的观察记录：
- 记录每次出现的场景
- 记录每次纠正的方式
- 积累3次后，提炼为通用规则

### Step 3：提炼

3次验证后，提炼为通用规则：
- 规则必须具体（触发条件+执行动作）
- 规则必须可检查（有明确的"对/错"判断）
- 规则必须不冲突（与现有规则对比）

### Step 4：合并

经过验证的规则合并到master-soul.md：
- 合并频率：每周一次 或 积累5条新规则
- 合并前通过安全检查
- 合并后更新版本号和changelog

## 7.3 改进原则

1. **具体优于模糊**："必须标注时间戳和间隔" > "要注意细节"
2. **可检查优于主观**：有明确的"对/错"判断 > "表现好/不好"
3. **不冲突优先**：新规则与现有规则对比，不冲突才合并
4. **只增不减**：除非规则被证伪，否则保留（即使暂时不用）
5. **文档即规范**：写在文档里的才算规范，口头说的不算

## 7.4 不可改进的内容

| 文件 | 原因 |
|------|------|
| INITIAL-SEED.md | 项目起源记录，只读 |
| sections/00-architecture.md | 系统架构，修改需全局审查 |
| 第7章本身 | 自我改进协议，修改会破坏协议基础 |

## 7.5 长期维护机制

**积累期（0-10条规则）：**
- 每个section独立积累
- 不急于合并到master
- 重点是记录和观察

**稳定期（10-50条规则）：**
- 每周合并一次到master
- 定期审查规则冲突
- 清理过时规则

**成熟期（50+条规则）：**
- 按需合并（5条新规则触发一次）
- 定期清理（每月清理一次过时规则）
- 持续优化表述
