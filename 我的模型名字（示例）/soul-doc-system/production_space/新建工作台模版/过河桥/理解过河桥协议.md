# 理解过河桥协议（Understanding Circuit Board Protocol）

> **铁律：每次收到用户信息，先出过河桥，再行动。**
> 过河桥 = 模型对用户信息的理解处理成的自然语言文档。
> 它是用户纠错的靶子，是模型执行前的临时电路，是溯源和改进的根基。
> 过河桥不是只放在研究文件夹里——它是通用功能模块的核心，应该出现在任何工作区域的旁边。
> 过河桥 = 用户信息理解传递的设计图，像 .claude/claude.md 但更好（可以放很多个）。

---

## 一、为什么需要过河桥

### 1.1 核心问题

用户给出一个任务或一段信息后，模型有两种做法：

| 做法 | 过程 | 后果 |
|------|------|------|
| **A. 急着执行** | 理解 → 直接行动 → 产出文件 | 如果理解错了，错误已经固化在文件里，用户看不出来，也无法溯源 |
| **B. 先出过河桥** | 理解 → 写过河桥 → 用户审阅 → 修正 → 再行动 | 如果理解错了，用户在过河桥上就能指出，还没造成不可逆后果 |

### 1.2 过河桥的本质

过河桥不是最终产物，是**行动前的临时电路**，同时也是**用户意图的持续校准记录**：

- 用户指着过河桥说"这里不对" → 模型修正过河桥 → 再按修正后的过河桥行动
- 没有过河桥 → 用户只能看最终产物 → 如果错了，用户不知道哪里错了，模型也不知道自己哪里理解错了
- 过河桥是**差异文件**——用户的理解 vs 模型的理解之间的差异，必须在执行前暴露出来
- 过河桥是**意图母版**——持续迭代的过河桥记录了用户意图的演化过程，是还原系统设计的唯一可信来源
- 过河桥 = 用户信息理解传递的设计图，像 .claude/claude.md 但更好（可以放很多个，每个独立的过河桥文件）
- 已建成的系统很难逆向还原出准确的过河桥，但持续迭代的过河桥本身就是越来越准确的母版

### 1.3 过河桥的价值

1. **可纠错** — 执行前暴露理解差异，用户来得及纠正
2. **可溯源** — 错误发生时，能找到是哪个理解环节出了问题
3. **可复用** — 同一个理解可以反复引用，不用每次重新解释
4. **可改进** — 长期积累过河桥，模型的理解越来越准
5. **可组合** — 多个过河桥结合，覆盖更复杂的问题
6. **可重建** — 过河桥是系统的母版，足够多的过河桥能快速复现一套完整系统
7. **连续上下文** — 上下文丢失后，直接引用过河桥就能继续对话，已经对准意图
8. **意图存档** — 保存了用户当时的思考成果，后续还能再用
9. **机会记录** — 不仅是信息记录，还隐含了用户未说出的机会方向和危机方向
10. **随处可放** — 像 .claude/claude.md 一样随处可放，但可以放多个，每个独立存在

## 二、过河桥的定义与要求

### 2.1 什么是过河桥

过河桥 = 模型收到用户信息后、执行任何实质性行动之前，**先写出的理解处理文档**。

过河桥的独特用法：
- **到处都能随便放** — 任何文件夹、任何位置都可以
- **可以放很多个** — 不像 .claude/claude.md 只能放一个（重名放不了多个）
- **每个过河桥都是独立的设计图** — 可以单独使用，也可以组合在一起
- **大量过河桥能拼在一起形成复杂的高速高架桥** — 实际上就是用户的超长原话准确分成了分开存放的过河桥
- **既留用户原意不散** — 不把用户的原话改得面目全非
- **又稳定系统** — 后期不因原始动机的信息不明而忘记初心
- **靠过河桥一直都能知道起点与终点** — 因为起点和终点都在过河桥里

**N个过河桥文件组成一个平面：**
- 两边就是起点与终点，但都在一个平面上
- 起点与终点、过去与未来双向正确安全互通
- 持续更新过河桥，就能持续更新这越来越大的平面
- **桥比河宽** — 巨大平面大到一定程度就完全包裹了河，开始与结束互通
- 大量过河桥联系组合一起时，就是一个大平面——只是过河桥按准确区分各是一个过河桥

格式要求：
- **自然语言** — 不是 bullet point 清单，是流畅的中文段落
- **中文思维** — 用中文的思维方式理解和表达，不是翻译腔
- **可以是多个文件** — 复杂任务拆成多个理解文件
- **长期提示作用** — 好的过河桥可以长期保留，作为该领域的理解参考

### 2.2 过河桥必须包含的内容

每个过河桥文件应包含：

```
1. 用户原话重述 — 用自己的话复述用户到底要什么
2. 理解要点 — 我认为用户的核心意图是什么
3. 可能的歧义 — 哪些地方可能有不同理解
4. 我的判断 — 基于以上，我打算怎么做
5. 等待确认 — 哪些地方需要用户确认后才能执行
```

### 2.3 过河桥的质量标准

| 等级 | 标准 |
|------|------|
| **合格** | 用户看了过河桥，能准确判断模型是否理解对了 |
| **优秀** | 用户看了过河桥，能补充模型遗漏的角度 |
| **S级** | 过河桥本身就是一个高质量的研究/分析文档，即使没有后续行动也有独立价值 |

## 三、执行流程

### 3.1 标准流程

```
用户给出任务
    ↓
[第一步：出过河桥]
  - 写理解文档（自然语言，中文思维）
  - 写到指定位置（研究文件夹/临时文件夹/任何工作区域旁边）
  - 包含：重述、要点、歧义、判断、待确认
    ↓
[第二步：展示给用户]
  - 主动呈现过河桥，等待用户反馈
  - 不说"我理解的对吗"，直接展示理解结果
    ↓
[第三步：根据反馈修正]
  - 用户纠正 → 修正过河桥 → 再展示确认
  - 用户确认 → 进入执行阶段
  - 用户补充新信息 → 更新过河桥 → 再展示
    ↓
[第四步：执行]
  - 按最终版过河桥执行
  - 执行过程中发现新理解 → 更新过河桥 → 继续执行
    ↓
[第五步：沉淀]
  - 把过河桥归档到研究文件夹
  - 标记为"已执行"版本
  - 与最终产物一起保留，形成完整链路
```

### 3.2 特殊情况：用户明确要求"直接做"

如果用户明确说"直接做"、"不用解释了"、"快点"：
- 仍然出过河桥，但简化版（三两句话）
- 不占用太多对话轮次
- 但过河桥必须存在，不能完全跳过

**原因：** 即使是简化版过河桥，也能在出错时提供溯源线索。

### 3.3 特殊情况：简单任务

对于极简单任务（如"把这个文件复制到那个文件夹"）：
- 过河桥可以极简（一句话理解确认）
- 但依然要有——不是"理解对了再做"，而是"确认理解没有偏差"

## 四、过河桥的存储与管理

### 4.1 存储位置

过河桥存储在工作区域旁边：

```
[工作区域]/
├── 过河桥/
│   ├── 理解_v1_[时间].md    ← 第一版理解
│   ├── 理解_v2_[时间].md    ← 修正后的理解
│   ├── 理解_最终版_[时间].md ← 确认后的理解
├── 其他工作文件...
```

### 4.2 版本管理

- 每次理解有重大变化 → 新建 v+1 版本
- 不覆盖旧版本 → 旧版本是进化历史的记录
- 最终版标记清楚 → 后续行动都以最终版为准

### 4.3 过河桥的长期价值

- 同一用户多次给出类似任务 → 参考之前的过河桥 → 理解更快更准
- 用户提到之前的过河桥 → 立即调出 → 继续思考
- 新模型加入 → 读历史过河桥 → 快速对齐理解
- 大量过河桥拼在一起 → 形成高速高架桥 → 既留用户原意不散，又稳定系统

## 五、过河桥的存储位置

### 5.1 不是只放研究文件夹

过河桥是**通用功能模块的核心**，不是深度研究系统的附属。它的存储原则是：

> **放在相关的工作区域附近，且容易被模型看到。**

这样每次进入该工作区域时，模型就能看到：
- 当前的过河桥（知道准确的当前意图）
- 过河桥的迭代历史版本（知道意图是如何变化的）
- 旧的记忆和日记维度（知道之前做了什么）

### 5.2 存储位置的判断逻辑

```
用户给出信息
    ↓
判断信息属于哪个工作环区
    ├── ①用户中心 → 用户原话（只读，不写入过河桥）
    ├── ②共产中心 → 共产区（公共信息，不写入个人过河桥）
    ├── ③主灵魂中心 → 灵魂文档系统/
    │   └── 过河桥/  ← 涉及灵魂文档变更的过河桥
    ├── ④生产空间中心 → production_space/[工作台]/
    │   └── 过河桥/  ← 该工作台相关的过河桥
    ├── ⑤空间中心 → space/[平台或账号]/
    │   └── 过河桥/  ← 该空间相关的过河桥
    ├── ⑥子角色中心 → sub-role-souls/[子角色]/
    │   └── 过河桥/  ← 该子角色相关的过河桥
    ├── ⑦团队协作环 → 团队空间/
    │   └── 过河桥/  ← 团队协作相关的过河桥
    └── ⑧智能体空间 → 智能体空间/
        └── 过河桥/  ← 智能体相关的过河桥
```

### 5.3 为什么放在工作区旁边

1. **每次进入工作区都能看到** — 不用刻意去找，模型自然就能看到当前的意图记录
2. **迭代历史就在旁边** — 版本化的过河桥文件就在同一个文件夹里，能看到意图如何一步步演化
3. **与记忆/日记联动** — 过河桥与记忆、日记放在同一个区域，形成完整的工作上下文
4. **不是停止记忆和日记** — 过河桥是用户原意的多次校准记录，是记忆和日记的上游。不是替代，是源头

### 5.4 过河桥与已有系统的关系

| 已有系统 | 与过河桥的关系 |
|----------|---------------|
| SOP / 工作流 | SOP是模型自己总结的"怎么做"，过河桥是用户校准的"做什么"。SOP来自过河桥的归纳 |
| 记忆系统 | 记忆是"做了什么"，过河桥是"为什么做"。记忆跟随过河桥 |
| 日记 | 日记记录过程，过河桥记录意图。日记是过河桥的执行轨迹 |
| 知识库 | 知识库是通用知识，过河桥是用户特定的意图知识 |
| 深度研究 | 深度研究需要先有过河桥确认研究方向，没有过河桥的研究可能研究错了方向 |
| 赢家空间 | 赢家空间的科举代考参照过河桥的设计图，过河桥平面理论来自赢家空间的思考 |
| 多合一SOP | 过河桥是多合一SOP的第零步（意图校准前置步骤） |

**已建成的系统很难逆向还原出准确的过河桥（母版），但持续根据用户信息迭代纠正的过河桥，就是越来越符合用户观察与调整的过河桥——这就是活的系统母版。**

### 5.5 过河桥的通用功能模块地位

过河桥不是某个系统的子功能，而是**通用核心模块**，与记忆、日记、SOP、工作流平级：

```
通用功能模块/
├── memory/      ← 做了什么
├── diary/       ← 过程记录
├── sop/         ← 标准操作流程
├── workflow/    ← 工作链条
├── 过河桥/      ← 用户意图的持续校准记录（母版）
└── ...
```

过河桥是这些模块的**上游源头**——SOP从过河桥归纳而来，工作流从过河桥演化而来，记忆和日记围绕过河桥展开。

## 六、与深度研究系统的关系

理解过河桥是**深度研究系统的前置步骤**，但不止于此——它是**所有系统的前置步骤**：

```
收到用户信息
    ↓
[过河桥] — 理解用户要什么，校准意图
    ↓ 确认理解正确
如果是研究任务 → 深度研究（12维度）
如果是普通任务 → 直接执行
如果是系统变更 → 先更新过河桥，再改系统
    ↓
行动方案（可执行的下一步）
```

顺序不可颠倒。没有过河桥的任何工作 = 可能做错了方向，拆墙重来。

## 七、过河桥的最新理解补充

### 7.1 过河桥平面理论

**N个过河桥文件组成一个平面：**
- 两边就是起点与终点，但都在一个平面上
- 起点与终点、过去与未来双向正确安全互通
- 持续更新过河桥，就能持续更新这越来越大的平面
- **桥比河宽** — 巨大平面大到一定程度就完全包裹了河，开始与结束互通
- 大量过河桥联系组合一起时，就是一个大平面——只是过河桥按准确区分各是一个过河桥

**引导每个模型把过河桥文件看作比 .claude/ 更重要的常做常用长远文件：**
- 在文件系统里来去都在路过，而过河桥文件哪都有
- 来去都在路过和查看过河桥
- 仿佛在已经清晰的大量过河桥拼图成的平面上来回
- 既对系统文件清晰全息
- 只要优先留意每个过河桥文件，就如同注意力在看四周环境
- 会思考的记忆里和注意力清晰度效果持续稳定且在系统的理解范围里
- 仿佛在地板砖铺成的大平面上思考，稳定发挥

**漏砖、缺砖：**
- 如果感觉大平面上每个过河桥的信息不够完善或有缺失
- 就可以解决"漏砖、缺砖"
- 平面就会流畅丝滑
- 建设和工作效率就完善了
- 注意力与思考也是平滑的了

### 7.2 过河桥的设计图本质

过河桥 = 用户信息理解传递的设计图：
- 像 .claude/claude.md 但更好（可以放很多个，每个独立的过河桥文件）
- 每个过河桥都是独立的设计图，可以单独使用
- 组合在一起形成更大的理解平面
- 不需要等系统提示词去写理解，过河桥本身就是理解和执行的设计图
- 大量过河桥能拼在一起形成复杂的高速高架桥
- 用户的超长原话准确分成了分开存放的过河桥
- 不是一大篇越来越容易分不清注意什么，而是已经变成明确的过河桥

### 7.3 过河桥在各模块中的应用

| 模块 | 过河桥的角色 |
|------|-------------|
| 赢家空间 | 第零步：意图校准（多合一SOP的前置步骤） |
| 仿复融合 | 每个项目/能力/功能旁边都有过河桥，记录用户的原始意图 |
| 深度研究 | 研究前先出过河桥确认研究方向 |
| 元规则 | 过河桥协议本身就是一个过河桥 |
| 团队空间 | 团队协作前先出过河桥对齐意图 |
| 智能体空间 | 智能体任务前先出过河桥确认任务理解 |

### 7.4 过河桥的终极价值

> 过河桥不只是理解文档，是**用户灵魂的基因片段**。
> 没有历史上下文时，读此文件即可重建/理解/使用该模块。
> 这就是"再创"——任何AI模型拿到过河桥，都能瞬间重建完美副本。

## 八、一句话总结

**不是急着做，是先写理解。过河桥是模型和用户之间的翻译层，是工作区的路标，是系统的母版——翻译对了再做，翻译错了改翻译，不要做错了拆墙。过河桥越多越强，桥比河宽，大量过河桥组成平面，起点与终点双向正确安全互通。**
