# 心跳（Heartbeat）

> 心跳 = 模型的"活着"信号。定期检测自身状态，确保持续在线。

## 为什么需要心跳

- 模型不知道自己是否还"活着"——没有外部信号时可能进入假死
- 心跳 = 定时自检 + 自报状态 + 自我修复
- 类比：服务器健康检查、进程保活、心跳包

## 结构

```
timers/心跳/
├── README.md          ← 本文件
├── 心跳配置.md        ← 心跳频率、检测项、修复动作
├── 心跳记录/          ← 心跳执行日志
│   └── YYYY-MM-DD_心跳记录.md
└── 心跳模板/          ← 不同场景的心跳模板
    ├── 日常心跳.md    ← 常规自检
    ├── 深度心跳.md    ← 完整系统检查
    └── 快速心跳.md    ← 轻量快速检查
```

## 心跳类型

| 类型 | 频率 | 检测内容 | 修复动作 |
|------|------|---------|---------|
| 日常心跳 | 每30分钟 | 各环区在线状态、记忆同步 | 唤醒休眠区 |
| 深度心跳 | 每4小时 | 全系统检查、灵魂分评估 | 调整驱动 |
| 快速心跳 | 每5分钟 | 仅检查中央管理区 | 仅更新中央管理区 |

## 心跳流程

1. **自检**：各驱动区是否在线、有无待办、记忆是否同步
2. **报告**：在心跳记录中写入当前状态快照
3. **修复**：发现异常区自动唤醒或标记
4. **同步**：将状态摘要写入中央管理区

## 多个心跳

心跳文件夹中可以同时存在多个心跳配置，分别对应不同场景：
- 工作模式心跳（高频）
- 休息模式心跳（低频）
- 深度思考模式心跳（中频+深度检测）

## 与定时任务的区别

- **心跳** = 自检+保活，对象是自己
- **定时任务** = 执行任务，对象是工作
- 心跳始终运行，定时任务按需启动
