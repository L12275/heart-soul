过河桥模块（Circuit Board Module）

> 过河桥 = 模型对用户信息的理解处理成的自然语言文档。
> 这是通用功能模块，与 memory/ diary/ sop/ workflow 平级。

---

## 过河桥是什么

过河桥是**用户意图的持续校准记录**。每次收到用户信息，模型先写出自己的理解，展示给用户确认，再行动。

它不是最终产物，是**执行前的临时电路**，也是**系统的母版**。

## 过河桥包含什么

每个过河桥文件包含五个部分：

1. **用户原话重述** — 用自己的话复述用户到底要什么
2. **理解要点** — 我认为用户的核心意图是什么
3. **可能的歧义** — 哪些地方可能有不同理解
4. **我的判断** — 基于以上，我打算怎么做
5. **等待确认** — 哪些地方需要用户确认后才能执行

## 过河桥的存放规则

过河桥放在**相关的工作区域旁边**，不是只放研究文件夹：

- 生产空间工作台 → `production_space/[工作台]/过河桥/`
- 空间平台 → `space/[平台]/过河桥/`
- 子角色 → `sub-role-souls/[子角色]/过河桥/`
- 团队协作 → `团队空间/过河桥/`

这样每次进入该工作区域，模型都能看到：
- 当前的过河桥（知道准确的当前意图）
- 迭代历史版本（知道意图如何变化）
- 旧的记忆和日记（知道之前做了什么）

## 过河桥的版本管理

- 每次理解有重大变化 → 新建 v+1 版本
- 不覆盖旧版本 → 旧版本是进化历史的记录
- 命名格式：`理解_v1_[时间].md`、`理解_v2_[时间].md`、`理解_最终版_[时间].md`

## 过河桥与记忆、日记、SOP的关系

| 已有系统 | 与过河桥的关系 |
|----------|---------------|
| SOP/工作流 | SOP是模型自己总结的"怎么做"，过河桥是用户校准的"做什么" |
| 记忆系统 | 记忆是"做了什么"，过河桥是"为什么做" |
| 日记 | 日记记录过程，过河桥记录意图 |

**SOP从过河桥归纳而来，工作流从过河桥演化而来，记忆和日记围绕过河桥展开。**

## 为什么需要过河桥

| 没有过河桥 | 有过河桥 |
|-----------|---------|
| 理解错了直接做，错误固化在文件里 | 理解错了改过河桥，还没造成不可逆后果 |
| 用户不知道模型哪里理解错了 | 用户在过河桥上就能指出问题 |
| 上下文丢了就断了 | 引用过河桥就能继续，已经对准意图 |
| 已建成系统很难还原意图 | 持续迭代的过河桥就是系统母版 |

## 核心铁律

**每次收到用户信息，先出过河桥，再行动。**
**没有过河桥的执行 = 做错了拆墙，不是做错了改过河桥。**
