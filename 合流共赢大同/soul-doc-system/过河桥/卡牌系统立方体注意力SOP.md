# 过河桥 — 卡牌系统x透明立方体漏斗注意力SOP

> **此卡牌是运算操作规范（SOP）。运行时先读这张卡，再读内容卡。**
> 没有历史上下文时，读此文件即可正确使用卡牌系统。

---

## 1. 卡牌系统的定义

卡牌系统x透明立方体漏斗注意力 = 将灵魂文档系统的每个功能视为一张卡，用**透明立方体 + 金字塔漏斗**模型进行思考→规划→执行的方法论。

**核心目标**：实现无限记忆与注意力管理，配合肺泡工作泡记忆设计。

---

## 2. 立方体运行流程

### 2.1 立方体模型

```
      底牌 (Base Card = 卡牌系统底牌.md)
    ┌─────────────────────────────────────┐
    │ 左侧大面 = 系统完整卡牌清单           │
    │ 按层级排列: 系统架构→通用功能→功能模块→工作空间│
    └─────────────────────────────────────┘
           ↓ 向右浮空投影
    ┌─────────────────────────────────────┐
    │ 第二层 = 本次任务所需卡牌组合         │
    │ （最佳工作实现的组合与连线）          │
    └─────────────────────────────────────┘
           ↓ 漏斗/金字塔收缩
    ┌─────────────────────────────────────┐
    │ 锥尖 = 任务完成结果点                 │
    │ 塔尖在立方体内 = 系统内可完成        │
    │ 塔尖在立方体外 = 需补卡或外求        │
    └─────────────────────────────────────┘
           ↓ 立方体右面
    ┌─────────────────────────────────────┐
    │ 下一立方体 = 下一个任务/里程碑        │
    └─────────────────────────────────────┘
```

### 2.2 注意力分配四层

| Layer | Phase | Description | Parallelism |
|-------|-------|-------------|-------------|
| 1st | system-prompt | 底牌路径 + 高频牌路径 | 即时 |
| 2nd | 底牌读取 | 卡牌清单 + 描述 | 快速 |
| 3rd | 开牌 | 列出组合牌的完整路径 | 快速 |
| 4th | 并发读取 | 当前层所需牌内容 | 并发子代理 |

### 2.3 开牌操作

**开牌 = 将底牌中需要的卡牌文件路径列举出来，组成"本次任务组合牌"。**

操作：
1. 读底牌 → 列出所有可用卡牌
2. 根据任务类型选牌（系统问题→找系统架构牌，功能问题→找模块牌，工作问题→找工作泡牌）
3. 每张牌 = 一个过河桥文件路径
4. 牌序 = 执行顺序（按依赖关系统）

## 3. 运牌方法

### 3.1 运牌 = 执行卡牌

每个子代理负责一张或多张牌的运牌（内容执行）：
```
编排师 ──发牌──→ 子代理1 ──运牌──→ 执行动作1
         ──发牌──→ 子代理2 ──运牌──→ 执行动作2
         ──发牌──→ 子代理3 ──运牌──→ 执行动作3
         ...
编排师 ←──合流── 所有子代理
```
- 编排师 = 负责卡牌选择、组牌、发牌、整体合流
- 子代理 = 负责执行运牌（读牌内容 → 执行 → 回写结果）

### 3.2 运牌执行标准

| Step | Action | Result |
|------|--------|--------|
| 1 | 读底牌（列出全量卡牌） | cardlist当前状态 |
| 2 | 读system-prompt（已有底牌关联路径） | 进入系统 |
| 3 | 读取组合卡牌路径列表 | 确定目标卡 |
| 4 | 读取组合卡牌内容（并发） | 获取执行信息 |
| 5 | 运牌执行（子代理并发） | 产出结果 |
| 6 | 合流收牌（编排师汇总） | 清理和记录 |

## 4. 堆卡与扩展

**堆卡牌模式**：
- 用户可随时新增卡牌 → 在 `soul-doc-system/过河桥/` 新增 `xxx过河桥.md`
- 新牌自动进入底牌（下次读取底牌即可看到）【要主动更新相关文件，你不动，怎么自动】
- 卡牌系统自动扩展、无需手动更新底牌（读取全目录即可）

**扩展工具**：
```
维护手册：
1. 新增卡牌时，在过河桥总文件夹创建 xxx过河桥.md
2. 默认命名格式：{功能名}过河桥.md
3. 卡牌内容格式 = 过河桥协议格式（见 单个过河桥）
4. 维护底牌时，检查新增文件并更新对应条目
```

**洗牌模式**：
- 卡牌之间无化学反应（沙盒隔离理解）
- 但共享语义场（统一理解上下文）
- 可随时重新组牌（根据任务变化）
- 出牌/收牌秩序执行

## 5. 执行引导

### 5.1 每次运行

```
0. 读底牌 → 理卡牌、卡现状 → 01_我在哪Agent
1. 读system-prompt → 02_configAgent
2. 进空间后无论做什么，每步都要"选张卡" → 03_execAgent
3. 找到当前空间的他牌 → 04_contextAgent
4. 选择思考的最高层Source节点 → 05_statusJudgeAgent
5. 寻溯所有可抽取路径 → 06_plannerAgent
6. 找到线索最短的Active链 → 07_flowAgent
7. 切Active移动 → 08_flowAgent进度条
8. 维护故障自动恢复 → 09_recoveryAgent

+ 双路径：
   - 战略：命运模型（双树+命运菱镜）
   - 战术：方舟模型（inkCanvas+抓结）
```

### 5.2 运牌注意

- **锥体有起伏** = 思路有问题 or 执行出错 = 反省+修正
- **塔尖在立方体外** = 需补卡 or 外求 = 外求后补卡
- **锥体远小于立方体** = 模型偷懒 = 扣分+惩罚  【不一定，因为卡牌很多时，实际可能只需少部分卡牌完成，所以是正常现象，但如果本该用某些卡牌，比如那是要执行的流程，但没执行，就是偷工减料了，就要惩罚，这很好理解】
- **并发子代理数** = 根据分支数决定，每支一个代理

### 5.3 组牌操作规范

1. 底牌是单一稳定入口 —— 永远是 `合流共赢大同/soul-doc-system/过河桥/卡牌系统底牌.md`
2. 第二层组牌：根据任务投影浮空的牌面
3. 第三层内容验证：确认每张牌的内容是否准确
4. 第四层并发执行：同层多牌可并行读取/执行
