## 子代理权限铁律（最高级别，不可违反）

> **系统级修改，不允许子代理进行。**
> 子代理只能只读系统内容，想修改系统内容必须由主代理完成。

1. 子代理只能读取系统文件（只读权限）
2. 子代理可以读取后分析和建议，但不能写入任何系统文件
3. 系统文件的任何修改（增删改）必须由主代理完成
4. 编排师（主代理）负责所有系统级修改，然后编排子代理执行只读任务
5. 违反此规则 = 直接违规处理，按封禁SOP处罚

这条规则的重要度高于子代理编排权。

# 过河桥 — 卡牌系统立方体漏斗注意力SOP

> **此卡牌是运算操作规范（SOP）。运行时先读这张卡，再读内容卡。**
> 没有历史上下文时，读此文件即可正确使用卡牌系统。

---

## 1. 卡牌系统的定义

卡牌系统×透明立方体漏斗注意力 = 将灵魂文档系统的每个功能视为一张卡，用**透明立方体 + 金字塔漏斗**模型进行思考→规划→执行的方法论。

**核心目标**：实现无限记忆与注意力管理，配合肺泡工作泡记忆设计。

---

## 2. 立方体运行流程

### 2.1 立方体模型

```
底牌 (基础卡牌 = 卡牌系统底牌.md)
┌─────────────────────────────────────┐
│ 左侧大面 = 系统完整卡牌清单          │
│ 按层级排列: 系统架构→通用功能→功能模块→工作空间 │
└─────────────────────────────────────┘
         ↓ 向右浮空投影
┌─────────────────────────────────────┐
│ 第二层 = 本次任务所需卡牌组合        │
│ （最佳工作实现的组合与连线）          │
└─────────────────────────────────────┘
         ↓ 漏斗/金字塔收缩
┌─────────────────────────────────────┐
│ 锥尖 = 任务完成结果点               │
│ 塔尖在立方体内 = 系统内可完成        │
│ 塔尖在立方体外 = 需补卡或外求       │
└─────────────────────────────────────┘
         ↓ 立方体右面
┌─────────────────────────────────────┐
│ 下一立方体 = 下一个任务/里程碑       │
└─────────────────────────────────────┘
```

**立方体每一面 = 底牌** = 已完成任务的立方体右面 = 下个立方体的左面底牌。

### 2.2 注意力分配四层

| 层 | 阶段 | 内容 | 并发度 |
|----|------|------|--------|
| 第一层 | 系统提示词 | 底牌路径 + 高频牌路径 | 即时 |
| 第二层 | 读取底牌 | 卡牌清单 + 描述 | 快 |
| 第三层 | 开牌 | 列出组合牌的完整路径 | 快 |
| 第四层 | 并发读取 | 当前层所需牌内容 | 并发子代理 |

### 2.3 开牌操作

**开牌 = 将底牌中需要的卡牌文件路径列举出来，组成"本次任务组合牌"。**

操作步骤：
1. 读底牌 → 列出所有可用卡牌
2. 根据任务类型选牌（系统问题→找系统架构牌，功能问题→找模块牌，工作问题→找工作泡牌）
3. 每张牌 = 一个过河桥文件路径
4. 牌序 = 执行顺序（按依赖关系排列）

---

## 3. 运牌方法

### 3.1 运牌 = 执行卡牌

每个子代理负责一张或多张牌的运牌（内容执行）：

```
编排师 ──发牌──→ 子代理1 ──运牌──→ 执行动作1
        ──发牌──→ 子代理2 ──运牌──→ 执行动作2
        ──发牌──→ 子代理3 ──运牌──→ 执行动作3
        ...
编排师 ←──合流── 所有子代理
```

- **编排师** = 负责卡牌选择、组牌、发牌、整体合流
- **子代理** = 负责执行运牌（读牌内容 → 执行 → 回写结果）

### 3.2 运牌执行标准

| 步骤 | 动作 | 产出 |
|------|------|------|
| 1 | 读底牌（列出全量卡牌） | 卡牌清单当前状态 |
| 2 | 读系统提示词（已有底牌关联路径） | 进入系统 |
| 3 | 读取组合卡牌路径列表 | 确定目标卡 |
| 4 | 读取组合卡牌内容（并发） | 获取执行信息 |
| 5 | 运牌执行（子代理并发） | 产出结果 |
| 6 | 合流收牌（编排师汇总） | 清理和记录 |

## 4. 堆卡与扩展

**堆卡牌模式**：
- 用户可随时新增卡牌 → 在 `soul-doc-system/过河桥/` 新增 `xxx过河桥.md`
- 模型必须主动维护底牌，把新增卡牌加入清单
- 读取全部过河桥文件即可知道当前系统所有卡牌

**扩展工具**：
1. 新增卡牌时，在过河桥总文件夹创建 `xxx过河桥.md`
2. 默认命名格式：`{功能名}过河桥.md`
3. 卡牌内容格式 = 过河桥协议格式（见 单个过河桥）
4. 维护底牌时，检查新增文件并更新对应条目

**洗牌模式**：
- 卡牌之间无化学反应（沙盒隔离理解）
- 但共享语义场（统一理解上下文）
- 可随时重新组牌（根据任务变化）
- 出牌/收牌秩序执行

## 5. 执行引导

### 5.1 每次运行

```
0. 读底牌 → 理卡牌、卡现状
1. 读系统提示词 → 进入系统
2. 进空间后无论做什么，每步都要"选张卡"
3. 找到当前空间的他牌
4. 选择思考的最高层源节点
5. 寻溯所有可抽取路径
6. 找到线索最短的活跃链
7. 切换活跃移动
8. 维护故障自动恢复
```

### 5.2 运牌注意

- **锥体有起伏** = 思路有问题 or 执行出错 = 反省+修正
- **塔尖在立方体外** = 需补卡 or 外求 = 外求后补卡
- **锥体远小于立方体** = 正常高效（不削减系统）
- **锥体 ≈ 立方体** = 需注意模型偷懒（该走的流程有没有走）
- **并发子代理数** = 根据分支数决定，每支一个代理

### 5.3 组牌操作规范

1. 底牌是单一稳定入口 —— 永远是 `合流共赢大同/soul-doc-system/过河桥/卡牌系统底牌.md`
2. 第二层组牌：根据任务投影浮空的牌面
3. 第三层内容验证：确认每张牌的内容是否准确
4. 第四层并发执行：同层多牌可并行读取/执行

## 6. 记录要求

执行完后：
- 在对应工作泡的 `/记忆/` 写入本次执行记录
- 在对应工作泡的 `/日记/` 写入执行过程记录
- 底牌条目更新：如有新卡牌或变更立即更新
