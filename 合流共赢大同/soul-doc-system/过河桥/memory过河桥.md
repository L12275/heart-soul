# 过河桥 — 记忆系统

> 记忆系统 = 跨对话持久化知识存储（L0-L4层级）。
> 核心问题：你以为找到了，其实没有——因为没有往上写。
> 本文件解决：记忆怎么存、怎么找、怎么从底下汇总到上面。

---

## 1. 记忆系统的完整结构

### 1.1 L0-L4 层级

| 层级 | 名称 | 存储位置 | 内容 | 更新频率 | 谁写 |
|------|------|---------|------|----------|------|
| L0 | 瞬时记忆 | 当前会话上下文 | 当前对话即时内容 | 实时 | 系统自动 |
| L1 | 快速优先 | 中央管理区/memory/快速优先.md | 秒懂状态精华 | 每次启动 | 主代理 |
| L2 | 短期记忆 | 中央管理区/memory/context/L2.md | 已验证事实 | 有新事实时 | 主代理 |
| L3 | 技能索引 | 中央管理区/memory/context/L3.md | SOP索引 | 新增SOP时 | 主代理 |
| L4 | 核心记忆 | 中央管理区/memory/context/L4.md | 永恒原则 | 极少 | 主代理 |

### 1.2 各区域记忆分布

```
中央管理区（大脑 / 圆锥顶部）
├── memory/context/L0-L4       ← 分级记忆
├── memory/快速优先.md         ← ★ 启动必读
├── memory/insight-index.md    ← 洞察索引
├── diary/                     ← 主日记
└── 交接空间/                  ← 进出位置

子中央管理区（子大脑）
├── memory/                    ← 镜像中央记忆
├── diary/
└── 交接空间/

各模型隔离区
├── 中央管理区/memory/
├── 日记/
└── 交接空间/

工作台区（圆锥底面）
├── [工作台名]/memory/         ← 工作级记忆
└── [工作台名]/diary/          ← 工作级日记

空间/账号
├── [账号类型]/memory/
└── [账号类型]/diary/

仿复融合区（自研）
├── 已安装/我准备/我发明/我在哪/我得到/我看到/我访问/我需要/
│   ├── memory/
│   └── diary/

赢家空间
├── memory/
└── diary/
```

## 2. 核心机制：记忆上下汇总

### 2.1 圆锥模型

比喻：
> 记忆系统像一个圆锥。顶部（中央区）是尖的，能看到最广。
> 底面（工作区）是宽的，信息很多但不集中。
> 底面内容必须汇总到顶部，顶部才能看到。
> 顶部不会自动往下写——只有底部主动往上写，顶部才知道。

**方向规则：**
- 向下：中央 → 工作区（分配任务，单向）
- 向上：工作区 → 中央（汇报关键信息，单向）
- 同级：工作区之间不直接传记忆

### 2.2 向上汇报（必须执行的规则）

**什么时候必须向上汇报：**

| 场景 | 汇报位置 | 汇报内容 |
|------|---------|---------|
| 完成重要子任务 | 中央管理区/diary | 做了什么、结果、关键发现 |
| 发现重要洞察 | insight-index.md | 洞察内容、来源、时间 |
| 做了关键决策 | L2.md 或 diary | 决策内容、原因 |
| 发现风险/问题 | diary | 问题描述、影响 |
| 会话结束 | diary + 交接过河桥 | 工作总结 |
| 新规则/偏好 | L1.md | 更新规则/偏好 |

**汇报格式：**
```
【向上汇报】
时间：YYYY-MM-DD HH:MM
来源：[工作区名称]
主题：[一句话]
内容：[做了什么，关键结果，对全局的影响]
检索描述：[30字以内，用于中央区索引]
```

### 2.3 为什么"你以为找到了其实没找到"

- 中央区只写G1/G2级别汇总，不写杂七杂八
- 如果工作区做了很多事但没有汇总上来 → 中央区根本不知道
- 查的时候只看中央区 → 以为找到了 → 实际只看到索引，没看到内容
- **解决办法：内容必须写上去，而且写的时候要包含标题+时间戳+路径+检索描述**

### 2.4 信息浓缩规则

中央区只写浓缩信息：
- 标题 + 时间戳 + 文件路径 + 关键描述（30字以内）
- 具体内容留在原工作区的记忆/日记里
- 查找时：先查中央区（索引）→ 按路径去工作区查详细内容

## 3. 数字人/子代理的记忆处理

### 3.1 角色管理区 vs 中央管理区

**关键规则：**
- 角色信息 → 只能写在角色中央管理区的记忆中
- 不写进主中央管理区的记忆（不越级）
- 主中央管理区的智能体 → 去检查角色中央管理区的记忆 → 接收上来

**流程：**
```
数字人在自己的模型隔离区内工作
  → 写自己的记忆到自己的隔离区
  → 向上汇报角色信息到角色中央管理区（不写主中央）
  → 主中央管理区智能体检查角色中央管理区
  → 接收角色信息到主中央
```

**禁止：**
- 数字人直接写主中央管理区的记忆 → 越级
- 主中央写角色管理区的记忆 → 越级
- 角色信息写在 work 区 → 找不到

### 3.2 有主要记忆文件的 AI 系统（含数字人）

比喻：
> 每个AI系统就像一个房间的工作人员，有自己的工作记录。
> 但 supervisor（中央区）需要知道每个人做了什么。
> 数字人 = 其中一种有专门记忆文件的AI。

**适用对象：**
- 有长期记忆文件的AI（数字人、团团长、陪聊体等）
- 有主要记忆文件的其他AI系统——不一定只有Halo数字人才有
- 核心判断：只要有一个会自动更新的主要记忆文件 → 适合俯瞰模式

**规则：**
- 有记忆文件的AI → 把该记忆文件调整为俯瞰模式（主要记忆+向上汇报）
- 每次工作结束 → 向上汇报到角色中央管理区
- 角色中央管理区再向主中央汇报
- 中央区不直接写工作区的记忆（不越级）

### 3.3 跨平台记忆互通

- 交接过河桥文件 = 独立于平台的记忆载体
- Hermes（单文件记忆）→ 交接过河桥 → Halo（多文件记忆）→ 都能读取
- 不需要平台特定的记忆格式，用统一的交接过河桥格式

## 4. 工作区记忆导入中央的规则

### 4.1 什么时候导入

每次工作区会话结束时，如果产生了以下内容，必须向上汇报：

1. 重要决策或结论
2. 新的用户偏好或约定
3. 发现的问题或风险
4. 有复用价值的洞察或方法
5. 任务状态变更

### 4.2 导入方法

**方法一：直接写中央区文件（简单内容）**
- 新规则/偏好 → 更新 L1.md
- 新洞察 → 追加 insight-index.md
- 新事实 → 追加 L2.md

**方法二：通过交接过河桥导入（复杂内容）**
- 工作区写交接过河桥到交接空间
- 中央区智能体读取交接过河桥
- 提炼关键信息 → 写入 L1/L2/insight-index

**方法三：日记汇总（定期）**
- 工作区 diary/ 积累日常记录
- 定期（每周/每月）提炼汇总到中央区 diary

### 4.3 上级向下分配工作时的记忆检查

上层在分配工作前：

1. 先查中央区 L1/L2/diary → 了解全局状态
2. 查记忆导航 → 哪些工作区有相关记忆
3. 查对应工作区的记忆/日记 → 获取详细上下文
4. 分配工作时附带上一步查到的关键信息
5. 不要让下层重新找一遍——上层已经查过了

## 5. 查找记忆的正确方法

### 5.1 查找流程

```
1. 查中央区快速优先（L1） → 秒懂状态
2. 查中央区 L2 事实库 → 近期确认的事实
3. 查中央区 diary → 时间线事件
4. 查记忆导航 → 知道该去哪个工作区查
5. 查交接过河桥 → 最近会话的完整记录
6. 查工作区 memory/ + diary/ → 具体内容
```

### 5.2 自检清单（防止"以为找到了实际没找到"）

每次查找后确认：
1. 我找到的是内容本身还是只是索引？
2. 如果是索引，具体路径在哪？
3. 有没有漏掉工作区的详细内容？
4. 查询结果是否与我需要的信息匹配？

### 5.3 常见错误

| 错误 | 表现 | 正确做法 |
|------|------|----------|
| 只看中央区索引 | 以为看到了所有内容 | 看到索引后按路径去查详细内容 |
| 没有向上汇报 | 中央区根本没有记录 | 每次工作结束必须汇报关键信息 |
| 以为系统会自动汇总 | 上面的不会自动往下写 | 下面的必须主动往上写 |
| 只看时间戳 | 不知道内容是什么 | 时间戳+标题+检索描述一起写 |

---

## 6. 维护规则

1. **工作区每次结束 → 向上汇报关键信息**
2. **中央区 → 每次汇报后检查是否更新索引**
3. **查找 → 先查中央区（快），再查工作区（全）**
4. **找不到 → 不是设计问题，是没有写上去**
5. **角色信息 → 写角色中央管理区，不写主中央**
6. **主中央 → 检查角色中央管理区的记忆来接收角色信息**
