# 过河桥 — 记忆上下汇总是解决这个问题的关键机制，它让记忆系统正常运转。

---

## 1. 问题：你以为找到了，其实没有

比喻：
> 你去旅游，在城市里打卡记了一些地方。
> 回到家里，你以为自己记得这些事，但家里没有记录。
> 你想翻家里的相册找当时的打卡记录——找不到。
> 因为那些记忆还留在城市里，没有汇总回家。

**这是记忆系统最大的潜在问题。** 中央区写的都是G1/G2级别的汇总，不是把杂七杂八全写上去。如果下面的工作区做了很多事但没有把关键信息汇总到中央区，中央区就看不到。

- 上面的记忆不会自动往下写（单向的）
- 但下面的记忆**必须往上写**（主动汇报）
- 如果不写，中央区以为看到了，实际没看到——以为找到了，其实没找到

---

## 2. 记忆的圆锥模型

比喻：
> 圆锥顶部 = 中央区（尖端，看到最广泛的信息）
> 圆锥底面 = 各工作区（宽，信息多但不集中）
> 底面内容 → 汇总到顶部 → 顶部才能看到
> 如果没有汇总动作，顶部什么都看不到

**方向规则：**
- 向下：中央 → 各工作区（中央可以分配任务下去，单向）
- 向上：各工作区 → 中央（各工作区必须主动汇报关键信息，单向）
- 同级：工作区之间不直接传记忆，通过中央中转

---

## 3. 各区记忆定位

| 区域 | 角色 | 记忆职责 | 读写方向 |
|------|------|---------|----------|
| 中央管理区 | 圆锥顶部 / 大脑 | 汇总所有关键信息，看到全局 | 读全部、写汇总 |
| 子中央管理区 | 子大脑 | 镜像中央区的汇总 | 读中央、写子集 |
| 工作台区 | 圆锥底面 | 记录具体执行过程 | 写执行细节、写汇总 |
| 空间/账号 | 用户空间 | 记录账号相关信息 | 写操作记录、写汇总 |
| 仿复融合区 | 自研空间 | 记录自研进展 | 写发明/准备/我得到等 |
| 赢家空间 | 赢家系统 | 记录赢家相关 | 写科举/中榜/方向 |

---

## 4. 向上汇报规则（核心规则）

### 4.1 什么时候必须向上汇报

| 场景 | 汇报位置 | 汇报内容 |
|------|---------|---------|
| 完成一个重要子任务 | 中央管理区/日记 | 做了什么、结果是什么、关键发现 |
| 发现了一个重要洞察 | 中央管理区/memory/insight-index.md | 洞察内容、来源、时间 |
| 做了关键决策 | 中央管理区/记忆/L2 或 日记 | 决策内容、原因、后果 |
| 发现了问题/风险 | 中央管理区/日记 | 问题描述、影响范围、建议 |
| 工作结束/会话结束 | 中央管理区/日记 + 交接过河桥 | 本次工作总结 |
| 新规则/新约定 | 中央管理区/memory/context/L1.md | 更新"用户偏好与约定" |

### 4.2 汇报格式

每次向上汇报时，必须包含：

```
【向上汇报】
时间：YYYY-MM-DD HH:MM
汇报来源：[工作区名称]
汇报人：[模型名称]
主题：[一句话描述]
内容：
  - 做了什么：[具体描述]
  - 关键结果：[结果]
  - 对全局的影响：[其他区域可能受影响的方面]
  - 需要上层知道：是/否
  - 检索描述：[30字以内，用于中央区索引]
```

### 4.3 汇报审批

- 中央区收到汇报 → 审查内容 → 决定是否纳入全局
- 纳入→ 写入L2事实库或insight-index
- 不纳入 → 保留在汇报来源区域，作为该区域的私有记录
- 紧急 → 直接写入中央日记，不经过审查

## 5. 信息浓缩规则（为什么不能全写上去）

比喻：
> 中央区记忆 = 大脑工作记忆（7±2个区块）
> 东西多了就全乱套了
> 所以只能写摘要：标题+关键描述+时间戳+文件路径

**中央区记忆只写浓缩信息：**

| 层级 | 写什么 | 不写什么 |
|------|--------|---------|
| L1 快速优先 | 当前任务、规则、偏好 | 具体过程、历史细节 |
| L2 事实库 | 已验证事实、决策 | 推测、未验证信息 |
| L3 技能索引 | SOP名称、位置、用途 | SOP具体内容 |
| 日记 | 时间线的关键事件 | 每个中间步骤 |

**具体内容留在原工作区：**
- 工作区/记忆/ = 该工作的完整记忆
- 工作区/日记/ = 该工作的时间线
- 中央区只记录"这里有这个内容，标题是什么，时间戳，路径在哪"

---

## 6. 查找时的正确流程

当需要找信息时：

```
1. **先查中央区**（快速优先 → L2 → 日记）
   - 看有没有记录过这件事
   - 如果有 → 得到标题、时间、路径 → 去对应路径查详细内容
   - 如果没有 → 进入步骤2

2. **查记忆导航**（过河桥/记忆导航过河桥.md）
   - 根据场景判断该去哪个区查
   - 按路径去对应区域查找

3. **查交接过河桥**
   - 如果刚结束会话 → 查最近的交接过河桥
   - 里面有完整的会话记录

4. **查工作区**
   - 去对应工作台/空间/账号的记忆和日记
   - 根据时间戳和标题定位
```

**常见错误（你以为找到了其实没找到）：**
- 只看中央区的索引 → 以为看到了所有内容 → 实际只看到了标题
- 没有把具体内容汇总到中央 → 中央根本没有记录 → 找不到
- 看到中央区有某个标题 → 以为内容也在那 → 实际内容在底下

## 7. 防忘事与向上汇报的关系

| 机制 | 方向 | 内容 |
|------|------|------|
| 交接过河桥 | 实时的 | 会话录像（当前会话的完整记录） |
| 向上汇报 | 定期的 | 关键事件汇总（从工作区到中央区） |
| 日记 | 单向的 | 时间线事件记录 |
| L0-L4 | 分层的 | 不同粒度/重要度的记忆 |

**流程：**
- 工作 → 边干边写交接过河桥（实时录像）
- 做重大决策 → 向上汇报到中央区
- 结束会话 → 写交接过河桥 + 更新向上汇报
- 启动 → 读快速优先（中央汇总） + 读交接过河桥（最近录像）

---

## 8. 数字人/子代理的记忆处理

比喻：
> 数字人 = 每个房间的不同工作人员。
> 每个工作人员有自己的工作记录（工作区记忆）。
> 但 supervisor（中央区）需要知道每个人做了什么。
> 所以每个人下班前要写一份汇报给 supervisor。

**规则：**
- 数字人在自己模型隔离区内工作，写自己的记忆
- 每次工作结束时 → 必须向上汇报到中央管理区
- 汇报到什么位置 → 根据汇报内容类型（任务结果→日记，洞察→insight-index，决策→L2）
- 中央区不直接写工作区的记忆（上面的不会往下写）
- 但中央区可以写对工作区的指令（向下分配）

---

## 9. 自检清单（防止"以为找到实际没找到"）

每次查找信息后，模型自查：

1. 我查的是中央区还是工作区？
2. 我找到的是内容本身还是只是标题/索引？
3. 如果是索引，内容在哪？路径对不对？
4. 查询结果是否与我需要的信息匹配？
5. 如果我只看中央区，有没有漏掉工作区的具体内容？

**发现找不到时的排查步骤：**
1. 确认信息是否已经被写入（不是"应该已经写了"而是真的写了）
2. 确认写入位置是否正确
3. 确认中央区是否有对应的索引记录
4. 如果都没有 → 可能是真的没写 → 需要重新产生这个信息

---

## 10. 维护规则

1. **每个工作区记忆 → 定期（每次结束前）向上汇报关键信息到中央区**
2. **中央区索引 → 每次汇报后检查是否需要更新**
3. **工作区执行 → 写详细过程（留在当地）**
4. **中央区 → 只写浓缩摘要（标题+时间+路径+关键描述）**
5. **查找时 → 先查中央区（快到），再查工作区（全到）**
6. **找不到 → 不是因为设计有问题，是因为没有写上去**
