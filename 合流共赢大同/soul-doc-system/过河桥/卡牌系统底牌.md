## 子代理权限铁律（最高级别，不可违反）

> **系统级修改，不允许子代理进行。**
> 子代理只能只读系统内容，想修改系统内容必须由主代理完成。

1. 子代理只能读取系统文件（只读权限）
2. 子代理可以读取后分析和建议，但不能写入任何系统文件
3. 系统文件的任何修改（增删改）必须由主代理完成
4. 编排师（主代理）负责所有系统级修改，然后编排子代理执行只读任务
5. 违反此规则 = 直接违规处理，按封禁SOP处罚

这条规则的重要度高于子代理编排权。

# 过河桥 — 卡牌系统底牌

> **此底牌是卡牌系统×透明立方体漏斗注意力的完整卡牌目录总览。**
> 每次运行、思考时的冷启动入口。读完此文件即可知道系统中有哪些卡牌、它们在哪里、如何组合、如何运牌。

## 1. 透明立方体漏斗注意力模型

### 1.1 立方体概念

```
                     ┌──────────┐
                     │ 立方体    │ ← 全局注意力容器
                     │          │
左面───────█─────────│  底牌     │ ← 完整卡牌清单（本文件）
（初始化）  │         │          │
           │    ▼    │  第二层   │ ← 浮空组合牌面（本次任务投射）
           │    │    │          │   卡牌按关系线条+主动/被动箭头排列
           │    │    │   ┌─漏斗─┐│
           │    └────→│  /      ││ → 向右漏斗式缩小到终点（任务完成）
           │          │ │ 结果点││
右面────────┴──────────│ └──────┘│ ← 立方体每一面都是底牌
（下一任务起点）       └──────────┘
```

**立方体每面都是底牌** = 已完成任务的立方体右面 = 下个立方体的左面底牌

### 1.2 注意力的四层递进

| 层 | 入口 | 读取什么 | 速度 |
|----|------|----------|------|
| 第一层 | 系统提示词 | 底牌文件路径 + 主要高频卡牌路径 | 即时 |
| 第二层 | 读取底牌 | 全量卡牌清单 + 卡牌描述 + 关系图 | 快 |
| 第三层 | 卡牌路径列表 | 列出所需卡牌的完整路径 | 快 |
| 第四层 | 卡牌内容 | 每个卡牌的完整内容 | 并发快 |

### 1.3 组牌运牌流程

```
第1步：读底牌 ─────────────────────────────────────────────── 注意力1
第2步：读系统提示词（已有底牌路径和主要牌路径） 注意力1(并行)
第3步：读取组合卡牌 ───────────────────────────────────────── 注意力2
         （按本次任务所需的卡牌清单读取）
第4步：确定漏斗分层 ───────────────────────────────────────── 注意力3
         （把所需卡牌分成可并发的漏斗平层）
第5步：并发读取卡牌内容 ──────────────────────────────────── 注意力3
         （编排子代理并行阅读，树杈多少条路线就编排多少子代理）
第6步：漏斗运牌执行 ──────────────────────────────────────── 注意力3
         （每个卡牌对应一个子代理，从第一层到最后一层逐层推进）
第7步：收牌 + 更新底牌 ───────────────────────────────────── 注意力3
         （完成的结果写入对应工作泡，更新底牌如果有新牌）
```

**编排师（运行模型）责任**：
- 组牌前亲自读取所需卡牌，确保理解到位
- 编排师几乎不再参与执行，看着子代理工作
- 确认锥体走势流畅：不流畅 = 思考/运行出错，需反省改正
- 锥体体积检查：不足以内求补齐，超出则临时外求+事后补牌

## 3. 用户扩展牌（领包入住）

> 用户可以不断堆积卡牌——特别是喜欢功能想法实现的用户。
> 用户可以随时领包带走、领包入住。

**领包入住流程**：
1. 用户在外部系统已有功能 → 通过过河桥协议迁移到 `soul-doc-system/` 对应子目录
2. 外部系统的文件/记忆/配置 → 通过过河桥无缝同步到中央
3. 用户不再用某个功能 → 领包离开，迁出或归档
4. 堆叠增加 → 在过河桥总文件夹增加新 `卡牌名过河桥.md`，系统自动识别为新卡牌

**卡牌的特征**：
- 互相不化学反应 → 沙盒式隔离理解
- 不一定是沙盒 → 这是解释卡牌思维的方式
- 能共享意义 → 统一语义场
- 类型非常多 → 模型牌、模块牌、工具牌、空间牌、工作区牌、脚本牌、等等

## 6. 卡牌关系图（主动/被动）

```
公网入口(101) ──主动──→ 理解协议(102) ──主动──→ 底牌(本文件)
  ↑                                              ↓
  └──────────── 布局:系统 ───────────────────┘

system-prompt(105) ──主动──→ INITIAL-SEED(103) ──主动──→ master-灵魂(104)
  ↓                        ↓                         ↓
底牌 ──主动──→ 组牌 ──主动──→ 运牌（漏斗式向右）
  ↓                        ↓
所有记忆/日记 ──被动──附属──→ 对应工作泡（执笔者独占读写）

中央管理区(307) ──主动──→ 阅卷厅(308) ──主动──→ 防忘事(407)
中央管理区(307) ──主动──→ 灵魂分数(228/229)
```

**箭头含义**：
- **主动**：该牌触发或控制后续行为
- **被动**：该牌被其他牌激活时参与
- **布局:系统**：系统布局型关系（始终存在）

## 7. 公网入口走流程（全场通行卡）

> **这是卡牌系统的"通行证"机制**。

**流程**：
1. 模型读取本底牌 → 知道有哪些卡需要
2. 通过公网入口(101)走流程全程 → 有权访问对应卡牌
3. 流程 = 确认隔离协议遵守 + 署名正确 + 不越权写入
4. 每次新会话启动时先检查「待审区」→ 用户已通过的项目
5. 只有通过的项目才能同步到对应位置建桥执行

**公网入口流程全程卡 = 全场通行资格的标准操作**。

## 8. 堆卡牌模式

> 用户可以不断堆积卡牌——特别是喜欢功能想法实现的用户。

**实现方式**：
1. **领包入住**：用户在外部系统接入时，已有的卡牌可直接迁移
2. **灵活同步**：外部系统的文件/记忆/配置通过过河桥协议无缝同步
3. **洗牌交换**：卡牌之间无化学反应（沙盒隔离），但可共享意义
4. **堆叠增加**：只需增加新 `卡牌名过河桥.md`，系统自动识别

**锥体体积管理**：
| 锥体体积 vs 立方体 | 含义 | 行动 |
|--------------------|------|------|
| 锥体 << 立方体 | 正常高效 | 不削减系统 |
| 锥体 ≈ 立方体 | 卡牌够用，注意偷懒 | 惩罚偷工减料 |
| 锥体 > 立方体 | 需补充卡牌 | 紧急补牌或临时外求 |
| 锥体 >> 立方体 | 系统升级 | 更大的底牌 |

## 9. 子代理编排规范

### 9.1 并发编排
- 组牌形成的运牌路线像树杈多支线 → 一个子代理负责一条路线
- 树杈有多少条路线，编排多少子代理
- 子代理可自行裂变编排二级子代理进行树杈连续展开
- 编排师给子代理专用的系统提示词（子代理版灵魂文档）

### 9.2 交接过河桥
- 编排师写一个 `交接过河桥` 文件到团队空间/会话临时子代理区/
- 包含：最强多合一SOP x 过河桥理解 + 当前任务信息
- 子代理读取交接文件 → 信息对齐 → 开始工作
- 每次会话随时更新交接文件，实现无损压缩切换

### 9.3 编排师工作流
1. 亲自读取所有组牌用到的卡牌文件（短期费时，长期高效）
2. 写组牌运牌漏斗推进计划到临时文件
3. 编排足够多子代理并发执行
4. 编排师观察漏斗进度，确认锥体走势流畅
5. 完成 → 收牌 → 更新底牌和日记/记忆

---

## 10. 卡牌数量管理警告

- **锥体体积 < 立方体**：正常高效（不削减系统）
- **锥体体积 ≈ 立方体**：卡牌够用但需注意模型偷懒
- **锥体体积 > 立方体**：需补充卡牌或临时外求（事后补牌）
- **锥体体积 >> 立方体**：系统需升级更大的底牌

> 新牌不会"自动"进入底牌。底牌是人工更新的。模型必须主动维护底牌。
> 每次发现新牌 → 立即更新底牌对应条目。
