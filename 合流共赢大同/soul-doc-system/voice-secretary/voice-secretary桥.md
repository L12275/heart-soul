# 过河桥 — 语音翻译秘书

> 语音输入 → 转写文字（可能乱）→ 秘书理解重组 → 精准指令 → 模型执行
> 用户只管说方向，秘书管理解+传达，模型拿到清晰任务直接干。

---

## 1. 用户原话重述

用户用语音输入法说话，语音转写为文字后有两个问题：
- 转写不准（发音、方言、口音导致错字）
- 口语化表达（口水话、语序乱、信息碎片化）

需要一个"秘书"层：
1. 听到/读到用户的原始语音转写
2. 理解用户真正想表达的意思
3. 重新组织成精准、清晰、不含冗余的指令
4. 把整理后的指令发给模型执行

秘书的身份 = 用户身边的红人+销冠+管理高手：
- 懂老板（用户）的心声，说一点就全理解
- 知道怎么给下属（模型）安排工作，不同岗位用不同风格
- 确保结果好、对方不生气、用户满意

核心目标：解决语音→AI的最后一公里质量问题。

---

## 2. 理解要点

**三层架构**：

| 层 | 角色 | 做什么 | 输出 |
|----|------|--------|------|
| 用户层 | 老板 | 说方向、提要求（语音） | 语音 |
| 秘书层 | 翻译+编排 | 听懂意图、重新组织、传达指令 | 结构化任务 |
| 执行层 | 下属 | 收到清晰指令、直接执行 | 结果 |

**为什么不直接语音→模型**：
- 模型拿到的是一团话，容易漏解、误解、顺序混乱
- 秘书脑中听到全貌，理解透，重新组织 → 精准表达
- 秘书保护双方：用户不被误解、模型不被乱指挥

**秘书的两大压力**：
1. 用户不满意 → 第一个骂秘书（传话的）
2. 秘书输出不好 → 模型乱干 → 用户追溯回秘书
→ 秘书必须说清：用户意图是什么、模型该干什么、结果预期是什么

**与子代理编排的关系**：
- 子代理编排师 = 也是一种秘书，写指令、选人员
- 站在用户和子代理之间，是第三者
- 没有秘书时，用户和模型"打手语"
- 有了秘书，用户只管说方向，秘书理解+安排

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 秘书是独立AI还是prompt层 | 独立系统更灵活，prompt层更简单 | 先prompt层快速落地，再评估独立系统 |
| 语音转写用什么工具 | 用户已有转写工具或需要推荐 | 秘书接收转写结果，不关心转写工具 |
| 秘书是否执行任务 | 只翻译+传达，还是也干活 | V1只做翻译传达，成熟后加执行 |
| 秘书是否自累积记忆 | 是否记住历史转写模式 | V1不自带记忆，靠system-prompt规则 |
| 粗口/情绪传达 | 用户的情绪语气要不要原样传递 | 秘书传达意图和指令，情绪过滤 |

## 4. 执行方法（V1快速落地）

### 第一步：建立prompt

将用户的秘书prompt优化为系统级prompt，写入 `voice-secretary/USER_SE_prompt.md`
关键特性：
- 听懂语音转写的真实意图（识别谐音、语序乱、碎信息）
- 重新组织为结构化任务指令
- 按不同模型/岗位调整表达风格
- 保护用户信息不扭曲

### 第二步：接入流程

```
用户语音说话
    ↓
语音转写工具 → raw_text（可能乱）
    ↓
秘书prompt处理 → structured_task（精准指令）
    ↓
子代理编排 → 选模型、派任务
    ↓
模型执行 → 结果
    ↓
回收验证 → 用户满意
```

### 第三步：优化迭代

- 每次转写后对比：用户原始意图 vs 秘书输出 vs 模型执行结果
- 哪里理解错了 → 调整prompt
- 哪里传达歪了 → 调整风格
- 积累快箭 → 下次更快更准
