# 全部过河桥 — 深度研究总览

---

## 一、过河桥到底是什么

先回答最根本的问题：过河桥是什么？

从用户的原话说起。用户建这套系统的时候，发现了一个很真实的问题：模型在理解用户信息的时候，经常会"想当然"。用户说了一个意思，模型理解成了另一个意思，然后直接就去执行了——等到产出文件出来了，用户一看，"这不是我要的"。这时候再改，已经晚了，因为错误已经固化在文件里了。

过河桥就是解决这个问题的。

过河桥 = 模型在收到用户信息之后、执行任何实质性行动之前，先写出来的理解处理文档。

它不是最终产物，是行动前的"临时电路"。就像你在盖房子之前先画一张设计图——设计图不是房子，但设计图错了，房子一定错。过河桥就是那张设计图。

更准确地说，过河桥有三个身份：

第一，它是**用户意图的校准记录**。模型收到用户信息，用自己的话重述一遍用户到底要什么，列出可能的歧义，说出自己的判断，标出哪些地方需要确认。用户看了过河桥，能直接判断模型理解对了没有——错了就指出来，还没造成不可逆的损失。

第二，它是**溯源工具**。当系统越来越复杂，一个模型做了某个决定，后来出了问题——翻过河桥，就能看到当时模型是怎么理解的，哪里出了偏差。

第三，它是**系统的母版**。一个积累了足够多过河桥的系统，理论上任何一个AI模型拿到这些过河桥文件，就能瞬间重建整个系统，不需要再看原话文件。过河桥本身就成了活的系统文档。

用户用了很多比喻来形容过河桥：像 .claude/claude.md 但更好（可以放很多个，每个独立的过河桥文件）；像一根橄榄枝从中央挂到每个文件；像一座桥连接河的两岸；N个过河桥拼在一起形成一个平面，桥比河宽——平面大到一定程度就完全包裹了河，开始与结束互通。

这些比喻都在说同一件事：过河桥是让模型和用户之间"翻译对了再做"的基础设施。

## 二、过河桥的五段结构

每个标准的过河桥文件，包含五段内容。这是用户定义的固定结构，不是模型自己想出来的。

**第一段：用户原话重述**

用自己的话复述用户到底要什么。不是逐字抄写，是用自己的理解重新组织一遍。这段的目的是让用户看到：模型确实理解了用户的话，不是对着关键词在回复。

**第二段：理解要点**

模型认为用户的核心意图是什么。这是模型对用户目标的解读，通常比重述更深入——重述是"用户说了什么"，要点是"用户真正要什么"。

**第三段：可能的歧义**

哪些地方可能有不同理解。模型主动列出自己不确定的地方，告诉用户"这里有几种可能的理解方向"。这是过河桥最有价值的部分之一——模型主动暴露自己的不确定性，而不是假装全懂了然后执行错了。

**第四段：我的判断**

基于以上理解，模型打算怎么做。这是模型对"接下来要执行的动作"的说明，让用户在行动之前就能看到模型的工作计划。

**第五段：等待确认**

哪些地方需要用户确认后才能执行。模型主动标出"这里我还不太确定，你想让我怎么做"，把决策权交还给用户。

这个五段结构的核心精神是：**翻译对了再做，翻译错了改翻译，不要做错了拆墙**。

用户原话中强调的另一个关键点：过河桥不是只放在研究文件夹里的研究工具。它是通用功能模块的核心，应该出现在任何工作区域的旁边。跟记忆放一起、跟日记放一起、跟SOP放一起——哪里最容易被模型看到，就放在哪里。

## 三、过河桥的存储规则

过河桥的存储有三层逻辑：系统级、模块级、隔离区级。

**系统级**：过河桥协议本身和处理规范放在 soul-doc-system/过河桥/ 目录下。每个系统模块（如 memory、diary、sop 等）都有对应的一个过河桥文件，记录该模块的理解。这些系统级过河桥是所有模型都能读取的参考文件。

**模块级**：soul-doc-system/ 下的每个子目录（如 sections/、memory/、diary/、sop/ 等）内部可以放置与本模块相关的过河桥文件。这些过河桥文件是该模块的"基因片段"——没有历史上下文时，读这个过河桥就能重建和理解整个模块。

**隔离区级**：每个模型在自己的隔离区（如 合流共赢大同/）内也有过河桥文件夹。这是模型对自己工作的理解记录，只属于该模型，不对外公开。

还有一个特殊的存储位置：**中央管理区/交接空间/**。这是交接过河桥的专用存放位置，用于跨会话、跨平台的记忆恢复。每个AI平台、每个项目、每个会话主题都有独立的子文件夹，里面放着对应的交接过河桥文件。

存放路径的选择原则：**放在相关的工作区域附近，且容易被模型看到。** 每次进入该工作区域时，模型自然就能看到当前意图记录和迭代历史版本。

## 四、过河桥的命名规范

过河桥文件的命名遵循一个简单的规则：`[过河桥名称].md`，其中过河桥名称直接对应它所描述的对象。

从50个过河桥文件中可以归纳出以下几种命名模式：

**模式一：模块名 + 过河桥**

最常见。直接在被描述的模块名后面加上"过河桥"三个字。例如：
- memory过河桥.md（描述记忆系统的过河桥）
- diary过河桥.md（描述日记系统的过河桥）
- skills过河桥.md（描述技能系统的过河桥）
- sop过河桥.md（描述SOP系统的过河桥）

这种命名模式适用于系统功能模块。

**模式二：具体文件名 + 过河桥**

用于描述一个具体文件的过河桥。例如：
- master-soul过河桥.md（描述 master-soul.md 的理解）
- system-prompt过河桥.md（描述 system-prompt.md 的理解）
- INITIAL-SEED过河桥.md（描述 INITIAL-SEED.md 的理解）

注意这里有时用全角字符"·"或"过河桥"直接附件在文件名后面，没有分隔符。

**模式三：系统名/概念名 + 过河桥**

用于描述一个系统或概念的过河桥。例如：
- 理解过河桥协议.md（过河桥协议本身的说明书——有点递归的感觉，一个过河桥写了过河桥本身）
- 元规则总过河桥.md（元规则系统的总览过河桥）
- 元规则过河桥.md（元规则系统中单个模块的过河桥）
- 合体过河桥.md（元规则/合体/归档区的过河桥）

**模式四：功能描述 + 过河桥**

描述一个功能或流程的过河桥。例如：
- CENTRAL_PATH解析过河桥.md（描述 CENTRAL_PATH 解析流程的理解）
- 交接过河桥-过河桥.md（描述交接过河桥机制本身的过河桥——又一个递归）
- 模块同步维护过河桥.md（描述模块同步维护的理解）
- 文件抬头同步脚本过河桥.md（描述文件抬头同步脚本的理解）

**模式五：项目/空间名 + 过河桥**

描述一个项目或工作空间的过河桥。例如：
- 赢家空间过河桥.md
- 科举过河桥.md
- 仕途过河桥.md
- 团队空间过河桥.md
- 智能体空间过河桥.md

**交接过河桥的特殊命名规则**：

交接过河桥有更严格的命名格式：`交接过河桥-YYYYMMDD HHMM-[桥名].md`

其中：
- 时间戳精确到分钟
- 桥名是AI主动总结的，浓缩会话主题
- 如有并行分支，后缀加 `-并行分支`

这个规则的目的是让交接过河桥在文件名上就能看出时间顺序，方便跨会话恢复时查找。

**命名规范的核心原则**：过河桥的命名应该让任何模型看到文件名就能知道这份过河桥描述的是什么，不需要打开文件才知道。

## 五、过河桥与SOP、记忆、日记的关系

这是过河桥和其他系统文档之间的核心关系。用一句话说：**过河桥是源头，SOP/记忆/日记是下游。过河桥是用户原意的持续校准记录，它们是从过河桥中归纳出来的。**

具体来说：

**过河桥 vs SOP**：SOP是模型自己总结的"怎么做"——它是从过河桥的"做什么"中归纳出来的标准操作流程。SOP来自过河桥的归纳，反过来SOP又能帮助模型更快地写过河桥（因为模型已经知道这个模块的标准做法了）。

**过河桥 vs 记忆**：记忆系统记录"做了什么"，过河桥记录"为什么做"。记忆是跟随过河桥的——没有过河桥校准的意图，记忆就变成了没有背景的孤零零的事件记录。

**过河桥 vs 日记**：日记是过河桥的"执行轨迹"。日记记录过程，过河桥记录意图。侧记是过河桥执行时的时间线。反过来，日记中的事件如果能反推出意图，就能帮助模型完善对应的过河桥。

**过河桥 vs 知识库**：知识库是通用知识（所有人共享的），过河桥是用户特定的意图知识（只有该用户/系统才有）。

**过河桥 vs 深度研究**：深度研究需要先有过河桥确认研究方向。没有过河桥的研究可能在错误的方向上越跑越远。过河桥是深度研究的第零步。

**过河桥 vs 多合一SOP**：过河桥是多合一SOP的前置步骤。流程顺序是：用户给出信息 → 先出过河桥（意图校准）→ 用户确认 → 再执行多合一SOP。

**过河桥 vs 领域模块（如科举、仕途）**：每个领域模块的过河桥记录的是该领域的设计意图。比如"科举过河桥"记录的是用户对科举考核系统的原始设计意图——如果后来要修改科举规则，先读科举过河桥，知道用户最初为什么要这样设计，再加修改，才不会偏离初心。

在看过的所有过河桥文件中，几乎每个功能模块的过河桥都会提到一句话："没有历史上下文时，读此文件即可重建/修复/使用该模块。"这说明过河桥的最终目标是让系统具备自解释能力——任何一个后来加入的模型，看到一个过河桥，就能明白这个模块是做什么的、为什么这么做、怎么维护。

## 六、过河桥的版本管理规则

版本管理在过河桥体系中不是一个"可选项"，而是基本规则。

**基本规则：每次理解有重大变化，就新建 v+1 版本。不覆盖旧版本。**

这是因为旧版本记录了意图的进化历史。模型靠旧版过河桥能看到"用户最初是这样想的"，靠新版能看到"后来变成了那样"，中间的变化过程本身就是宝贵的系统记忆。

标准的过河桥版本管理结构：

```
[工作区域]/过河桥/
├── 理解_v1_[时间].md        ← 第一版理解
├── 理解_v2_[时间].md        ← 修正后的理解
├── 理解_v3_[时间].md        ← 再修正
└── 理解_最终版_[时间].md    ← 确认后的理解（最终标记）
```

最终版是明确标记的——后续所有行动都以最终版为准。最终版和旧版本并列存放，不覆盖、不删除。

**交接过河桥的版本管理更严格**。交接过河桥文件一旦创建就不会被修改，新的会话产生新的交接过河桥文件。这是因为交接过河桥的本质是"实时录屏"——录像是不可篡改的历史记录，改录像等于改历史。

在交接过河桥的结构中，每个会话主题文件夹内：
- 最新时间戳的文件 = 主过河（当前有效）
- 历史时间戳的文件 = 旧版主过河（被取代但仍保留）
- 次并行过河桥/ 子文件夹 = 副线程记录
- 暂停并行过河桥/ 子文件夹 = 已不活跃的并行记录（可随时取回使用）

**过河桥的长期价值被用户明确描述过**：

> 同一用户多次给出类似任务 → 参考之前的过河桥 → 理解更快更准
> 用户提到之前的过河桥 → 立即调出 → 继续思考
> 新模型加入 → 读历史过河桥 → 快速对齐理解

## 七、过河桥的核心铁律

从50个过河桥文件中提取的核心铁律，大约有下面几条。每一条都是用户的原话或强频率出现的规则。

**铁律一：每次收到用户信息，先出过河桥，再行动。**

这是最高级别的规则。没有过河桥就执行 = 可能做错了方向，拆墙重来。例外只有一个：用户明确要求"直接做"或"不用解释了"——但即便如此，过河桥还是要写，只是可以简化版（两三句话）。

**铁律二：过河桥是自然语言，不是bullet point清单。**

模型很容易把过河桥写成项目符号列表。用户明确反对：要用流畅的中文段落，用中文思维理解和表达，不是翻译腔。

**铁律三：过河桥不放在一个地方——它出现在任何工作区域旁边。**

这是过河桥和 .claude/claude.md 的关键区别。.claude/claude.md 只能放一个，而且只能放一个位置。过河桥可以放很多个，每个工作区旁边都可以有自己的过河桥。目的是让模型路过的时候自然就能看到，不需要刻意去找。

**铁律四：SOP从过河桥归纳。记忆和日记从过河桥展开。**

过河桥是源头，其他是下游。不要反过来——不要先写SOP再过河桥，也不要先写记忆再过河桥。

**铁律五：已经建成的系统很难逆向还原出准确的过河桥（母版），但持续迭代纠正的过河桥就是越来越准确的母版。**

这个铁律是理解过河桥终极价值的钥匙。过河桥不是事后补写的说明文档，是系统运行的实时校准器。有了它，系统是活的——每次理解偏差都被及时纠正，过河桥本身就越发接近用户的真实意图。

**铁律六：过河桥越多越强。桥比河宽。**

N个过河桥拼在一起形成一个平面。平面大到一定程度，就完全包裹了河。起点与终点双向正确安全互通。这既是方法论（多建过河桥），也是哲学（系统足够完善后，一切皆可通达）。

**铁律七：过河桥的用户回应机制。**

用户用 `【】` 大括号在过河桥上回应。半边 `【` = 用户正在写，模型等待。完整 `【回应内容】` = 看到后立即处理。行内的 `【xxx】` 也是用户补充说明。模型看到用户批注要立即处理，不能忽略。

**铁律八：交接过河桥的防忘事红线。**

没写交接过河桥就结束会话 = 扣25分。说"后续处理"然后消失 = 扣25分。让用户帮你记进度 = 扣25分。换对话窗口不先读过河桥 = 扣10分。

**铁律九：子代理只能读，不能写系统文件。**

过河桥作为系统文件，子代理只读不写。系统文件的任何修改必须由主代理完成。子代理可以写自己的工作泡过河桥，但不能动系统级的。

## 八、从所有过河桥文件中提取的共性模式

通读了约50个过河桥文件后，我看到了几个反复出现的结构性模式。这些模式就是写过一个河桥文件时应该遵循的"模板"。

**模式一：五段式 + 元数据尾注**

几乎所有过河桥文件都遵循这个结构：
1. 文件头部（标准橄榄枝抬头）
2. 【用户回应区域】
3. 创建时间/更新时间
4. 五段内容（原话重述、理解要点、歧义、判断、等待确认/维护规则）
5. 元数据尾注（创建时间 + 关联文件列表）

**模式二：标准橄榄枝抬头**

几乎所有有机系统的文件都带同样的9行橄榄枝抬头，内容包括：系统所属声明、公网入口说明、隔离区规则。这是整个系统的"文件甲"（passport），让每个文件都能追溯到中央。

这个橄榄枝抬头有一个有趣的设计细节：它的第4行引用了 `[CENTRAL_PATH]/system-prompt.md`。这意味着任何一个模型拿到任何一份过河桥文件，顺着橄榄枝就能找到系统入口——不需要预先知道系统的存在。

**模式三：歧义-判断-确认三角**

大部分过河桥的2-5段构成了一个三角形关系：歧义（问题的空间）→ 判断（模型的答案）→ 等待确认（留给用户的决策权）。这个三角形是过河桥的"处理循环"——模型把不确定性暴露出来，给出自己的倾向性判断，把最终的决策权留给用户。

**模式四：模块级过河桥的通用模板**

大部分系统模块的过河桥有非常相似的内部结构。以下是其通用骨架：

```
## 1. 用户原话重述
用户在系统文件中提到：[模块描述]。用户的核心意图：让模型在需要时能正确使用该模块。

## 2. 理解要点
核心机制 + 模块路径（规范位置/隔离区内/模板位置）

## 3. 可能的歧义
（歧义表格：说明 + 解决）

## 4. 我的判断
使用该模块的标准动作（4-5步）
重建该模块的标准动作（4-5步）

## 5. 等待确认
- [ ] 检查项...
```

这个模板的优点是通用性强——任何新模块都能套用。缺点是有些模块（如理解过河桥协议本身、交接过河桥机制、灵魂分数白盒存亡系统）的内容远超这个模板的承载能力，需要写得更深入。

**模式五：关联文件引用**

每个过河桥文件末尾都有"关联"字段，列出与当前过河桥相关的其他文件。这个设计的作用是构建过河桥之间的"关联网络"——一个过河桥写完，自动把读者导向相关的其他过河桥，形成完整的理解路径。
