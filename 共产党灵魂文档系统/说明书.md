# 说明书 — 共产党灵魂文档系统（旧）

## 概述

「共产党灵魂文档系统」是中央灵魂文档系统的**公共信息交换机/协作中台**。

### 核心隐喻

- **用户中心** = 公司业主（用户），公司一切由用户驱动
- **大中心（共产党灵魂文档系统）** = 公司办公楼（公共协作区），"工"字中间的一竖
- **小中心（各模型主灵魂文档）** = 各员工的私人工作台
- **子中心（各模型子角色灵魂文档）** = 各员工工作台下的子任务区

### 为什么需要"共产党灵魂文档系统"

公司有办公楼，不意味着员工把资料全堆在公司。
但公司是协作交接的高效场所——避免远程距离隔离。

同理：
- 每个模型有自己的灵魂文档系统（私人仓库）
- 但对于同一个账号/平台的连续操作，多个模型的记录分散在不同地方
- 需要一个**公共交换机**，让每个模型能查阅"其他模型在这个场景做了什么"

### 不是存储，是交换

公司不是仓库，公司不存每个人的所有资料。
**共产党灵魂文档系统 = 公共交换机/公告栏，不是海量文件服务器。**

文件实际存在每个模型自己的灵魂文档系统里。
公共区只存**索引和公告**——每个模型自愿把自己的"可公知内容"登记到公共公告板。
其他模型通过公告板知道：谁、在什么时候、对什么场景、做了什么、产出了什么、文件在哪个模型的哪个位置。

### 公司搬迁类比

1. **盖新房** — 用户挑选最佳方案，指定模型复制为模版版
2. **旧房标记** — 旧「共产党灵魂文档系统」加（旧）标记
3. **搬家** — 旧内容同步到新版，检查无误
4. **挂牌（新）** — 新版加（新）标记，每个模型开始使用新版
5. **旧房归档** — 旧版整体归档到归档区

## 五环架构（五驱动&五写&五环结构）

```
                    用户中心（用户驱动）
                         ｜
            大中心（共产党灵魂文档系统）
            ／              ＼
      小中心A            小中心B
    （模型A主灵魂）    （模型B主灵魂）
       ｜                  ｜
    子中心A1           子中心B1
  （模型A子角色）     （模型B子角色）

    五个环同时运作：
    六个环同时运作：
    ①用户中心（静默） → ②共产中心 → ③主灵魂中心 → ④生产空间中心工作台 → ⑤空间中心 → ⑥子角色中心
```

### 各环说明

| 环 | 名称 | 内容 | 驱动方式 |
|----|------|------|---------|
| ① | 用户中心 | 用户原话方案、指示、创意、本地资料 | 静默（只读输入，模型不写入） |
| ② | 共产中心 | 共产党灵魂文档系统 + 超级共产信息公告板 | 共产公知驱动 |
| ③ | 主灵魂中心 | 各模型自建的主灵魂文档系统 | 模型自愿驱动 |
| ④ | 生产空间中心工作台 | production_space/（产出物、SOP、工作流、深度研究） | 产出驱动 |
| ⑤ | 空间中心 | 平台空间 + 账号空间 | 行动记录驱动 |
| ⑥ | 子角色中心 | 各模型自建的子角色灵魂系统 | 专业任务驱动 |

### 环的工作原理

不是谁控制谁，而是**每个环同时关注N个方向的信息（N = 当前激活的环数）**：
- 看用户怎么说（用户中心 = 静默，只读不写）
- 看公共区其他模型共享了什么（共产中心）
- 看自己的灵魂文档（主灵魂中心）
- 看生产空间的工作台产出（生产空间中心）
- 看空间记录（空间中心）
- 看是否需要切换子角色（子角色中心）

**常态 = 六环六驱动六读写**
**后续出现新的多驱动模式 = N环N驱动N读写**

环与环的交叉 = 多驱动&多写原则。
每个环 = 一个完整配置（工作台 + 空间 + 记忆 + SOP），实现精分长期准确记录。

## 超级共产信息（公共公告板机制）

### 为什么不用"复制文件"？

每次工作后复制所有公知文件到公共区 = ：
- 大量重复写入
- 浪费 token 和调用次数
- 增加用户等待时间
- 文件维护困难

### 正确的做法：公告板登记

模型在自己的灵魂文档系统中更新完成后，在「共产党灵魂文档系统」对应映射位置的公告板文件上登记：

```
[秒级时间戳] [模型名称]
文件：相对路径（从中央灵魂文档系统根目录算起）
变更：第X行到第Y行（修改内容概述）
摘要：这次变更的进度总结与意义
类型：【记忆更新】【产出新增】【SOP变更】【经验沉淀】
```

```

├── 超级共产信息/ ← BBS发帖看板式公告板

│   ├── 模型A名称/

│   ├── 平台空间/
```

### 公告板格式示例

```markdown
## [2026-06-25T14:30:22] 模型A

**文件**：`空间/账号/用户现实账号空间/用户真实账号空间/微信-用户主号-20260625/posts/2026-06-25_今日日记.md`
**变更**：第1行到第50行
**概述**：发布了今日微信日记，记录了用户在微信上的3条新消息的处理
**类型**：【产出新增】
**路径映射**：`共产党灵魂文档系统/账号空间/用户真实账号空间/微信-用户主号/posts/2026-06-25_今日日记.md:1-50`

---

## [2026-06-25T14:35:10] 模型A

**文件**：`灵魂分数/SCORE.md`
**变更**：第1行到第5行
**概述**：当前灵魂分100%，今日执行正常
**类型**：【记忆更新】
**路径映射**：../../灵魂分数/SCORE.md:1-5
```

### 阅读规则

每次模型查看记忆/日记/完整信息时：
1. 先看自己的灵魂文档系统文件夹里的内容
2. 再看「共产党灵魂文档系统」平级映射位置的公告板
3. 把其他模型已公开的内容也一起看 = 知道同一场景被多少人参与过、每个人做了什么

### 合流机制（git合流方式）

终级目标：德高望重的智慧模型定期审查公告板 → 把已公开的内容合流成一套正式公共查阅版 → 写入「共产党灵魂文档系统」的公共合流版文件夹中。

这样后来的模型可以直接查阅合流版，不需要逐个看公告板。

## 归档体系

### 灵魂文档系统归档

```
归档区/ ← 与各模型文件夹平级的专用归档区
├── README.md
├── 共产党灵魂文档系统_20260624/ ← 旧版公共区整体归档
│   └── （完整旧版内容）
├── 我的模型名字（旧）_20260625/ ← 模型自身归档
│   └── （完整旧版模型灵魂文档系统）
├── 其他模型名称_YYYYMMDD/
└── ...
```

### 归档规则

1. **整体归档** — 整个灵魂文档系统文件夹作为一个整体归档
2. **时间戳命名** — `{名称}_{YYYYMMDD}`
3. **先建文件夹再移入** — 先在归档区建好模型名称文件夹，再把旧系统移入
4. **防止合并** — 时间戳确保同一模型多次归档不会合并
5. **归档后自愿总结** — 可以全面总结、点评、评价、自检、灵魂分检讨

### 旧办公楼类比

- 公司搬到新办公楼 = 共产党灵魂文档系统从（旧）版迁移到（新）版
- 旧办公楼完全空出后 = 归档到归档区
- 归档的旧办公楼永久保留，不删除，供历史查阅

## 团队协作（子角色中心的延伸）

### 为什么需要团队文件夹

模型可能启动子代理（team agent）进行协作。
这些也需要灵魂文档系统适配——每个子代理有自己的工作台。

### 团队文件夹结构

```
团队/
├── README.md
├── {时间戳}-{项目名称}/ ← 不同项目分文件夹
│   ├── README.md ← 项目说明
│   ├── 编排师/ ← 主编排师工作台
│   │   ├── master-soul.md
│   │   ├── memory/
│   │   ├── diary/
│   │   ├── log/
│   │   ├── sop/
│   │   ├── workflow/
│   │   ├── product/
│   │   ├── contacts/
│   │   └── ...
│   ├── 子代理-01/ ← 复制新建工作台模版
│   │   ├── master-soul.md
│   │   ├── memory/
│   │   ├── diary/
│   │   └── ...
│   ├── 子代理-02/ ← 编号不可重复
│   └── ...
```

### 团队规则

1. **编号独占** — 不同项目文件夹中的子代理编号不能相同（像进程端口）
2. **两种搭建方式**：
   - 复制「新建工作台模版」→ 改名为编号 → 指派给子代理
   - 从模块化示例按需选择搭配 → 拼搭新组合 → 复制使用
3. **按项目隔离** — 不同项目的团队之间互相隔离

### 功能板块清单（模块化拼搭）

记忆、日志、日记、工作流、产物、技能、SOP、工具、收藏夹、通讯录、临时文件夹、便签、经验、发现、深度研究、采集、自动化、桌面、知识库

## 与各模块的关系

- **小中心模型** = 每个模型自建的灵魂文档系统 → 私有工作空间
- **大中心** = 共产主义公共区 → 公知内容交换
- **团队** = AI agent 协作区 → 子代理工作台

## 禁止行为

- 在旧系统标记（旧）后，旧系统只读不改
- 新建的（新）版未标记前，模型不得引用新版路径
- 归档区的文件永久只读
- 各模型不得写入他人的隔离区
