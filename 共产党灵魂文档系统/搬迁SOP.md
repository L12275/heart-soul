# 共产党灵魂文档系统 — 搬迁SOP

> **适用场景**：从旧版公共区搬迁到新版公共区。
> **方向**：顺时针安全操作（与逆时针归档方向相反）。
> **原则**：遵循超级信息SOP — 每个步骤有完整因果工序支撑，每步可验证。

## 搬迁总览

```
步骤 1：准备新版（预部署）     ← 顺时针起点
步骤 2：旧版加（旧）标记（冻结旧版）
步骤 3：同步旧内容到新版（数据迁移）
步骤 4：新版加（新）标记（激活新版）
步骤 5：旧版归档（拆除旧办公楼）  ← 顺时针终点
```

> **归档 vs 搬迁的区别：**
> - 归档 = 逆时针暂停归档（停止 → 归档 → 冻结）
> - 搬迁 = 顺时针安全操作（准备 → 冻结旧 → 迁移 → 激活新 → 归档旧）
> - 搬迁期间旧版不停服（先冻结但保留，新版准备好再切换）
> - 搬迁完成后旧版才归档，归档前旧版始终可回滚

## 前置条件

- 用户已指定最佳方案
- 有足够的模型参与执行
- 搬迁期间不影响正常使用（旧版保持可读）

## 详细步骤

### 步骤 1：准备新版（预部署）

**执行者**：用户指定模型
**动作**：
1. 将最佳方案的灵魂文档系统文件夹复制到目标位置
2. 新版路径与旧版路径同级或按用户指定位置
3. 验证新版结构完整性：
   - [ ] 所有模块文件夹存在（memory/ diary/ sop/ workflow/ product/ 等）
   - [ ] 超级共产信息/ 公告板结构存在
   - [ ] 平台空间和账号空间结构存在
   - [ ] 团队/ 文件夹存在
4. 验证新版内容正确性：
   - [ ] README.md 内容完整
   - [ ] 模块同步维护规则.md 存在且内容正确
   - [ ] 系统提示词包含完整的环架构描述（当前为九环）
   - [ ] 团队文件夹结构正确
5. **新版暂不标记（新），处于"待切换"状态**
6. 向用户提交验证报告

**输出**：新版就绪，等待切换

### 步骤 2：旧版加（旧）标记（冻结旧版）

**执行者**：所有模型（协作完成）
**动作**：
1. 旧版 README.md 加（旧）标记
2. 所有模型停止向旧版写入任何内容
3. 旧版进入只读冻结状态
4. 此时状态：
   - 旧版 = 只读（冻结）
   - 新版 = 只读（未激活）

**验证**：
- [ ] 旧版 README.md 已标记（旧）
- [ ] 旧版无任何模型写入操作
- [ ] 所有模型已知晓旧版已冻结

**输出**：旧版冻结，数据安全锁定

### 步骤 3：同步旧内容到新版（数据迁移）

**执行者**：用户指定模型 + 智慧模型审查
**动作**：
1. 按模块逐项同步旧版内容到新版：
   - memory/ — 记忆系统
   - diary/ — 日记
   - sop/ — SOP
   - workflow/ — 工作流
   - product/ — 产物
   - 超级共产信息/ — 公告板
   - 平台空间/ — 平台记录
   - 账号空间/ — 账号记录
   - 团队/ — 团队协作
2. 每同步完一个模块，检查：
   - [ ] 路径正确（新版对应位置）
   - [ ] 内容完整（无遗漏文件）
   - [ ] 格式正确（markdown 无损坏）
3. 所有模块同步完成 → 整体检查：
   - [ ] 新旧版内容一致
   - [ ] 无遗漏模块
   - [ ] 无格式错误

**验证**：用户或智慧模型审查通过

**输出**：新版内容完整，与旧版一致

### 步骤 4：新版加（新）标记（激活新版）

**执行者**：用户
**动作**：
1. 确认同步无误后，新版 README.md 加（新）标记
2. 所有模型开始使用新版
3. 旧版彻底冻结，不再读取
4. 通知所有模型：公共区已切换到新版

**验证**：
- [ ] 新版 README.md 已标记（新）
- [ ] 所有模型已知晓并使用新版
- [ ] 旧版无任何读取操作

**输出**：新版正式启用，旧版退役

### 步骤 5：旧版归档（拆除旧办公楼）

**执行者**：指定模型
**动作**：
1. 在归档区新建文件夹：`共产党灵魂文档系统_YYYYMMDD`
2. 将旧版完整内容移入归档文件夹
3. 归档区 README.md 更新：记录本次归档的时间戳和版本
4. 归档后旧版永久只读

**验证**：
- [ ] 旧版已移入归档区
- [ ] 归档区文件夹命名正确（含时间戳）
- [ ] 归档区内容完整
- [ ] 归档区标记为只读

**输出**：旧版归档完成，搬迁闭环

## 回滚方案

如果搬迁过程中出现问题：

```
1. 停止搬迁流程
2. 旧版仍处于冻结状态（可读）
3. 新版标记移除（回退到未激活状态）
4. 排查问题
5. 重新执行搬迁流程
```

**关键保障**：旧版在步骤 2 冻结后始终可读，直到步骤 4 激活新版。因此任何步骤失败都可以回滚到旧版。

## 搬迁铁律

1. **旧版不删除** — 整个搬迁过程中旧版始终存在，直到步骤 5 才移入归档区
2. **先冻结再迁移** — 旧版必须先标记（旧）冻结，再同步内容
3. **先验证再激活** — 新版必须通过内容检查，才标记（新）激活
4. **激活后才归档** — 旧版必须等新版激活后才移入归档区
5. **全程可回滚** — 每个步骤失败都可以回到上一步

## 搬迁 vs 归档 对比

| 对比项 | 归档（逆时针） | 搬迁（顺时针） |
|--------|--------------|--------------|
| 方向 | 逆时针暂停归档 | 顺时针安全操作 |
| 步骤 | 暂停 → 归档 → 冻结 | 准备 → 冻结旧 → 迁移 → 激活新 → 归档旧 |
| 旧版状态 | 直接归档 | 先冻结 → 再迁移 → 最后归档 |
| 服务中断 | 有（归档期间不可用） | 无（旧版冻结但仍可读） |
| 回滚能力 | 无（已归档） | 有（旧版冻结期间随时可回滚） |
| 适用场景 | 模型自身升级 | 公共区系统升级 |
