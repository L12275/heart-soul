## 1. 用户原话重述

用户在 system-prompt.md 和 memory/README.md 中定义了记忆系统的完整规范：记忆是跨对话持久化知识存储，基于 GA（GenericAgent）L0-L4 层级架构设计，与日记系统互补但功能边界清晰。用户的核心意图是让模型能正确理解记忆系统在九环架构中的定位、读写规则、层级结构和与日记的区别。

## 2. 理解要点

### GA L0-L4 五层记忆层级

```
L0 硬编码核心     → system prompt中，每轮自动加载，不可修改
L1 洞察索引       → memory/context/L1.md，≤30行，只记"有什么"和"去哪找"
L2 公理层         → memory/context/L2.md，稳定知识，遵守3条公理
L3 任务技能/SOP   → sop/ 目录，可复用流程，每3次提炼一次
L4 会话归档       → log/年/月/日/，按时间轴存放已完成会话
```

**各层级详解：**

- **L0 — 硬编码核心**：不是文件，是 system prompt 中硬编码的内容。包含基础行为规则（红领巾精神、安全铁律、三好学生标准）、"No Execution, No Memory"原则、九环架构规则。每轮对话随上下文自动注入，无法被绕过，不可修改。是所有约束中最强的一层。

- **L1 — 洞察索引（≤30行）**：极简导航索引，只有两级映射——高频场景直接指向L3文件名（快速直达），低频场景只列关键词，模型自行在L2/L3查找。只写"有什么能力"和"对应文件路径"，不写细节。每次对话开始时加载，每次工作结束后更新。对应GA的 `memory/global_mem_insight.txt`。

- **L2 — 公理层（全局事实库）**：存储已验证的稳定知识——路径、凭证、配置、常量等环境事实。遵守三条红线公理：
  1. **行动验证原则（No Execution, No Memory）**：没有实际执行过的信息，不写入记忆，不能把"听说的""读到的"当成已验证事实。
  2. **神圣不可删改性**：已验证数据严禁丢弃，只能压缩/迁移层级，删除 = 背叛记忆系统。
  3. **禁止存储易变状态**：只存稳定环境事实，不存"当前会话次数""最近操作时间"这类易变数据。
  4. **最小充分指针**：上层只留最短标识，L1只存"有什么"和"去哪个文件找"，L2只存"事实本身"+"来源"，不层层叠加引用。

- **L3 — 任务技能/SOP**：特定任务的可复用流程，位于 `sop/` 目录。同一场景出现3次 → 自动提炼为SOP候选 → 验证1次 → 标记validated。每个SOP包含触发条件 + 执行步骤 + 验证方式。SOP可被自动调用（L1索引指向它们）。对应GA的自动沉淀系统（morphling_sop.md）。

- **L4 — 会话归档**：已完成任务的归档记录，按时间轴存放于 `log/年/月/日/` 格式。不是实时写入，是工作结束后批量归档。每次子角色工作结束后整理日志归入对应日期的文件夹。对应GA的 `memory/L4_raw_sessions/`。

**记忆流转路径**：新任务 → [L1索引] → L3 SOP → 执行 → 经验 → L2验证 → 完成 → L4归档 → 更新L1。

### 四驱动&四写结构

灵魂文档系统采用四驱动&四写模式（参考GA工作记忆vm/memory/context/）：

1. **主灵魂文档系统通用区（地平面）**：主角色共享的底层记忆池，通用记忆和日记写在系统根目录
2. **主灵魂文档系统工作台专区（专用层）**：主角色按领域的专用工作空间，与通用工作台相关的内容双写到对应工作台
3. **子灵魂文档系统通用区（地平面）**：子角色共享的底层记忆池
4. **子灵魂文档系统工作台专区（专用层）**：子角色按领域的专用工作空间

**双驱动&双写规则**：一般工作时的通用记忆写到该系统的根目录通用记忆文件夹；如果与某专用工作台相关，除写到通用区外再写一份到对应工作台；如果不与任何专用工作台相关，只需写到通用区；如果只与某专用工作台相关，只需写到对应工作台。五环结构中，下部两环是两个地平面（主通用区+子通用区），上部三环是主工作台+子工作台+中心规则交叉环。

### 与九环架构的交互关系

记忆系统是跨环组件，**每个环都产出记忆、都调用记忆**。读写记忆/日记时自主判断内容属于哪个工作环区：

- **① 用户中心（静默，只读不写）**：模型不会在用户专用文件夹里写信息，只在里面读取、取经、寻找收获。
- **② 共产中心**：实际文件存在每个模型自己的灵魂文档系统里，公共区只存索引和公告板。模型在自己的灵魂文档系统中更新完成后，在共产党灵魂文档系统对应映射位置的公告板文件上登记（秒级时间戳+模型名称+文件路径+变更摘要+类型）。
- **③ 主灵魂中心**：灵魂分数、身份、思考规则、SOP的核心记忆层。
- **④ 生产空间中心工作台**：产出物、SOP、工作流、深度研究的记忆 → 写对应工作台 memory/。
- **⑤ 空间中心**：平台空间和账号空间的行动记录 → 写对应 account/platform space/。
- **⑥ 子角色中心**：子角色专属记忆 → 写对应子角色 memory/。
- **⑦ 团队协作环**：团队空间记忆 → 写对应团队 memory/。
- **⑧ 智能体空间**：智能体编排与管理记忆 → 写对应智能体 memory/。
- **⑨ 中央管理区（最高优先级）**：总交换机，其他8区的每次读写都同步到中央管理区，中央管理区的每次读写广播给相关区。通用/不专属的内容 → 写中央管理区 memory/。

**记忆/日记读写判断铁律**：先上朝（中央管理区总开关 → 记忆 → 日记 → 代办 → 排程 → 概览），然后才是其他8区。通用功能模块（memory/、diary/、log/、代办与进度/等）不写入工作内容，只作为模板，工作内容在各驱动区写入。

### 记忆与日记的本质区别

这是记忆系统最关键的功能边界：

| 维度 | 记忆（memory） | 日记（diary） |
|------|---------------|--------------|
| 内容性质 | 验证确认后的正确记忆 | 当天实时事件过程记录，不一定是正确记忆 |
| 准确性 | 已验证 | 纪实性（可能包含错误判断） |
| 格式 | 精炼的要点（结论+来源） | 小时级时间轴（每个事件有完整推导链） |
| 时机 | 事后提炼（日记完成后提炼） | 边做边记，实时记录 |
| 用途 | 快速查找已验证知识 | 形成正确记忆/SOP/工作流的原材料 |
| 存放路径 | memory/（L0-L4层级） | diary/年/月/日/[日期].md |

**转化关系**：日记（完整解题过程，小时级时间轴） → 超级信息SOP思考 → 今日事今日毕 → 提炼为记忆/SOP/工作流。日记是原料，记忆是成品。原料必须干净（问题已解决），成品才有价值。日记允许是草稿，但最终版必须是通过超级信息SOP校验后的正确版本。

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| L0到底是文件还是system prompt中的硬编码 | 看起来像文件（有L0-L4.md和L0.md），但规范说L0是system prompt中不可修改的硬编码内容 | L0不是文件，是system prompt中硬编码的规则；memory/context/L0.md是工作台特定的通用记忆，不是真正的L0 |
| 记忆和日记的写入时机重叠 | 日记是"边做边记"，记忆是"事后提炼"，但实际工作中两者经常同时进行 | 严格区分：实时过程写入日记，验证确认后提炼到记忆；日记允许有错误，记忆必须已验证 |
| 四驱动四写与九环的关系 | 四驱动四写是记忆/日记的组织结构，九环是模型工作的整体架构，两者如何对应 | 四驱动四写 = 主系统/子系统的通用区+工作台；九环 = 模型工作的9个维度；记忆读写规则按九环判断，但按四驱动四写存放 |
| L4是log/还是memory/L4/ | 实施指南说L4在 log/年/月/日/，但GA原始设计是 memory/L4_raw_sessions/ | 本系统采用 log/年/月/日/ 格式，按时间轴存放已完成会话 |
| 记忆能否写入user-zone | 用户专用文件夹 = 模型定期读取区，模型只读不写 | 记忆不能写入 user-zone/，模型在用户空间只有读取权限 |
| 共产中心的记忆实际存储位置 | 公共区只存索引和公告，实际文件在哪？ | 实际文件存在每个模型自己的灵魂文档系统里，共产中心只存公告板索引 |

## 4. 我的判断

### 记忆系统的核心定位

记忆系统是中央灵魂文档系统的**跨对话持久化知识存储层**，是"岩层"（system-prompt.md 的隐喻：记忆系统 = 岩层，持久化知识体系）。它解决模型的三个根本局限中的"无法对抗衰减"——通过L0-L4层级、四驱动四写结构、九环读写规则，让模型在对话之间积累可复用的验证知识。

### 使用记忆系统的标准动作

1. **识别当前所在环**：判断正在工作的环区（①用户中心 → ⑨中央管理区），确定记忆应该写入哪个环区的 memory/。
2. **按层级写入**：
   - 新任务开始时 → 读取L1洞察索引，找到对应SOP
   - 执行过程中 → 经验先写入L2（验证后），SOP沉淀到L3
   - 工作结束时 → 会话归档到L4，更新L1索引
3. **判断驱动位置**：如果只在主系统内工作 = 双驱动（主通用区 + 主工作台）；如果涉及子系统 = 四驱动（四个位置都写）。
4. **区分记忆与日记**：实时过程 → 写 diary/（边记边解题）；验证结论 → 提炼到 memory/（成品）。

### 重建记忆系统的标准动作

1. 在 `soul-doc-system/memory/` 下创建标准文件结构
2. 创建 L0-L4 上下文文件（context/ 目录下）
3. 确保 README.md、GA-L0-L4-analysis.md、implementation-guide.md 齐全
4. 在中央管理区记忆和日记位置建立初始索引
5. 确认 L1 不超过 30 行
6. 创建本过河桥作为模块基因片段
7. 验证模块可独立运行且符合 GA 三条公理

### 关键判断原则

- **记忆只存已验证结论**：没做过的不写（No Execution, No Memory），没验证的不写（日记记过程，记忆记结果）。
- **记忆不可删除**：旧记忆只能压缩/迁移/归档，不能丢弃——这是"背叛记忆系统"的行为。
- **记忆按环分区存放**：每个环有自己的 memory/，不混写。通用内容才写中央管理区。
- **记忆是成品，日记是原料**：日记允许草稿、允许错误、允许推导链，记忆必须是经过超级信息SOP校验后的正确版本。
