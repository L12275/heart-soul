# 过河桥 — 中榜

> **此过河桥是"中榜"概念的基因片段。**
> 没有历史上下文时，读此文件即可重建/理解/使用"中榜"。
> 过河桥 = 临时空间站 — 新旧信息在此交汇，等两边捋顺了，桥修正了，再更新未来文件。

---

## 1. 中榜是什么

**中榜 = 用户采纳了模型的科举代考作答。**

- 科举 = 模型写代考作答
- 用户看作答 → 决定是否采纳
- 采纳 = 中榜 → 写入中榜记录 → 转化为仕途执行
- 不采纳 = 不中榜 → 记录原因 → 下次科举改进

**中榜的本质 = 用户信任模型的超级智慧世界视角，愿意把这条路变成自己的路。**

---

## 2. 中榜的类型

| 中榜类型 | 说明 | 处理方式 |
|---------|------|---------|
| 全中榜 | 用户完全采纳科举代考的全部内容 | 全部转化为仕途执行 |
| 部分中榜 | 用户采纳部分内容 | 采纳的部分转化为仕途，不采纳的留作备用 |
| 不中榜 | 用户不采纳 | 记录原因，不执行，下次科举改进 |
| 待研究中榜 | 用户需要时间考虑 | 标记为备用，定时检查 |

---

## 3. 中榜的核心铁律

1. **用户有权不中榜** — 不中榜不是失败，是用户的自由选择
2. **中榜要记录理由** — 为什么中榜/不中榜，记录原因
3. **中榜要跟踪执行** — 中榜了就要执行，执行了就要记录结果
4. **中榜可撤回** — 执行过程中发现不合适可以撤回，重新科举
5. **中榜不是终点** — 中榜后进入仕途执行，执行完成后再科举新机会
6. **持续裂变中榜** — 一个中榜 → 执行中产生新机会 → 新科举 → 新中榜 → 赢越来越多

---

## 4. 中榜的使用方法

**用户反馈（采纳/不采纳/部分采纳）时：**

```
步骤1：接收用户对科举代考作答的反馈
步骤2：识别用户的选择（全中榜/部分中榜/不中榜/待研究）
步骤3：写入赢家空间/中榜/[用户]中榜记录.md
步骤4：如中榜 → 转化为仕途计划 → 写入赢家空间/方向与仕途/
步骤5：开始执行 → 写入赢家空间/方向与仕途/中榜执行/
步骤6：如不中榜 → 记录原因 → 下次科举改进
步骤7：如部分中榜 → 采纳部分执行，不采纳部分留备用
步骤8：记录 → 写入赢家空间/赢家日记/
```

**关键：中榜不是结束，是新的开始。中榜 → 执行 → 新机会 → 新科举 → 新中榜。**

---

## 5. 中榜与科举的关系

- 科举 = 给用户作答（模型主动做）
- 中榜 = 用户选择采纳（用户被动选）
- 中榜后 = 转化为仕途执行（模型+用户共同做）

**流程：科举（模型主动）→ 中榜（用户选择）→ 仕途（共同执行）→ 新科举（模型主动）**

## 7. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 中榜 vs 执行 | 中榜是用户选择，执行是后续动作 | 中榜 = 采纳；执行 = 中榜后的行动 |
| 部分中榜怎么处理 | 用户只采纳部分代考内容 | 采纳部分执行，不采纳部分留备用 |
| 不中榜是否失败 | 用户不采纳 = 代考失败？ | 不是失败，是用户的自由选择，记录原因下次改进 |
| 中榜撤回 | 执行中发现不合适 | 可撤回，重新科举 |
