# 功能模块同步维护规则

> **核心原则：一个通用版，全系统同步。所有模块的权威源统一在 soul-doc-system 根目录。**

## 模块化思维

所有「新建xx模版」文件夹（账号空间模版、工作台模版等）都是**模块拼搭**的结果。
每个功能模块只有一个"权威源"——`soul-doc-system/` 根目录下的对应文件夹。
根目录没有的模块文件夹，才在具体区域文件夹内建立，以该位置作为该模块的通用区模版。

**为什么模块只放在模版文件夹里：**
因为新建账号空间模版、新建工作台模版等是**模板**，不是实际工作空间。
模板文件夹的内容是"只读参考"，不会被日常写入。
如果模块放在实际工作空间里，会被实际工作不断覆盖写入，就无法保持通用模版的纯净性。
放在新建模版文件夹里 = 天然隔离实际工作写入 = 永远保持通用标准。

## 新增模块规则

新建立的功能模块，判断其归属位置：

### 符合根目录条件的（放在 soul-doc-system 根目录）
- 该模块在**多个区域**都会用到（工作台、账号空间、平台空间等）
- 该模块是**通用功能**，不绑定特定工作场景
- 该模块是**系统级功能**，如灵魂分数、git追踪等
- 判断：将模块放在根目录后，是否需要同步到多个模版？
  - 是 → 放在根目录
  - 否 → 放在具体区域

### 适合只在某区域文件夹的（不放根目录）
- 该模块**只在特定场景**使用（如子角色的前期经验）
- 该模块**不属于通用模版拼搭**的必要组件
- 该模块**已经被区域内的新建模版引用**作为模板
  - 此时该区域文件夹即成为该模块的"局部权威源"
  - 更新方式：编辑区域内的模块 → 复制到该区域下的各新建模版

## 模块更新流程

### 根目录模块更新（全系统同步）

```
编辑 soul-doc-system/{模块}/        ← 编辑权威源
    ↓
复制到 production_space/新建工作台模版/{模块}/    ← 覆盖工作台模版
    ↓
复制到 space/账号/各分类/新建账号空间模版/{模块}/  ← 覆盖所有账号模版
    ↓
如有平台空间：复制到 space/平台/通用-待细分/{模块}/  ← 覆盖平台模版
（仅 memory/ diary/）
```

### 区域内模块更新（局部同步）

```
编辑 区域/{模块}/                    ← 编辑权威源
    ↓
复制到 区域/新建xx模版/{模块}/        ← 覆盖该区域下的各模版
```

