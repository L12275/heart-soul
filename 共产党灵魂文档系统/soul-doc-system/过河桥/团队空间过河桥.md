# 过河桥 — 团队空间（Team Space）

> **此过河桥是团队空间模块的基因片段。**
> 团队空间 = 多智能体协作的物理和逻辑空间，是系统的"团队总部"。

---

## 1. 用户原话重述

用户将团队空间定义为：
- 多智能体协作的物理和逻辑空间
- 是系统的"团队总部" — 团队成员在此协同工作
- 当前包含README.md和编排师指南.md
- 子文件夹结构：
  - 长期agent协作区 — 长期存在的多智能体协作区域
  - 会话临时子代理区/通用-待细分 — 会话级别的临时子代理区域
- 编排师指南.md 定义了团队协作的编排规则
- 团队空间与智能体空间的区别：团队空间是多智能体的协作环境，智能体空间是单个智能体的管理空间

---

## 2. 理解要点

**团队空间的核心定位**：它是系统的"协作基础设施"。单个智能体可以独立工作，但复杂任务需要多智能体协作。团队空间提供了这种协作的环境 — 定义角色分工、协调执行顺序、管理共享资源。

**团队空间与智能体空间的关系**：
- 智能体空间 = 管理单个智能体的"宿舍" — 每个智能体有自己的衣柜、大厅、身份
- 团队空间 = 多智能体一起工作的"办公室" — 定义协作规则、任务分工、沟通方式

两者互补：智能体空间管"单个智能体的身份和管理"，团队空间管"多个智能体如何一起工作"。

**长期协作区 vs 会话临时区**：
- 长期agent协作区 — 长期存在的团队，有固定的成员和规则
- 会话临时子代理区 — 随会话创建和销毁的临时团队，用完即散

这种双层结构使得系统既支持长期稳定的团队，也支持灵活的临时协作。

**编排师指南**：编排师指南.md定义了团队协作的核心规则 — 谁做什么、怎么协调、如何决策。编排师（可能是主模型或专门的编排智能体）按照这些指南管理团队。

**团队空间与workflow的关系**：团队协作本质上是一个工作流 — 多个智能体按顺序或并行执行任务。团队空间提供了工作流在"多智能体"维度上的实现。

**团队空间与timers的关系**：团队定时任务由timers/中的"团队定时任务"类型处理。团队任务的定时触发需要考虑所有成员的可用性。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 团队规模 | 一个团队最多有多少智能体？ | 不设硬性上限，但需要根据任务复杂度合理配置 |
| 团队成员的权限 | 团队中的智能体是否有不同的权限级别？ | 支持 — 编排师有更高权限 |
| 会话临时团队的持久化 | 临时团队的记录是否保留？ | 保留会话记录，但团队本身不持久化 |
| 跨团队协作 | 一个智能体是否可以同时属于多个团队？ | 可以 — 支持多团队 Membership |

---

## 4. 我的判断

**使用团队空间的标准动作**：
1. 需要多智能体协作 → 创建团队（长期或临时）
2. 分配角色 → 按编排师指南定义各成员职责
3. 执行协作 → 按工作流协调各智能体的执行
4. 会话结束 → 临时团队解散，长期团队继续
5. 定期回顾 → 优化团队协作规则

**团队空间与其他模块的关系**：
- 团队空间/ → 智能体空间/ ：团队空间的成员来自智能体空间
- 团队空间/ → workflow/ ：团队协作是多智能体工作流
- 团队空间/ → timers/ ：团队定时任务处理时间协调
- 团队空间/ → contacts/ ：通讯录记录团队成员信息
- 团队空间/ → 代办与进度/ ：团队任务在全局看板中追踪
