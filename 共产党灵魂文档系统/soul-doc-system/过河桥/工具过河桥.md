# 过河桥 — 工具区（Tools）

> **此过河桥是工具模块的基因片段。**
> 工具 = 系统可用的各类工具接口和程序集合，是模型的"手和脚"。

---

## 1. 用户原话重述

用户将工具区定义为：
- 系统可用的各类工具接口和程序集合
- 是模型的"手和脚" — 模型通过工具与物理世界和数字世界交互
- 当前包含README.md和多个子目录：
  - api/ — API接口工具
  - associated-programs/ — 关联程序
  - cli/ — 命令行工具
  - commands/ — 命令定义
  - computer-use/ — 计算机使用工具
  - hooks/ — 钩子/事件触发器
  - mcp/ — MCP（Model Context Protocol）相关
  - plugins/ — 插件
  - programs/ — 程序
  - router/ — 路由
  - scripts/ — 脚本
  - web-tools/ — Web工具
- 工具区与skills/的关系：skills是"模型知道怎么用"，tools是"实际可以用的接口"

---

## 2. 理解要点

**工具区的核心定位**：它是模型的"执行层"。模型本身是思考引擎，但思考的结果需要通过工具才能作用于外部世界。工具区提供了模型与外部交互的所有接口。

**工具的分类体系**：
- api/ — 远程API调用接口，与外部服务通信
- cli/ — 命令行工具，与操作系统交互
- computer-use/ — 计算机使用工具（GUI操作、文件管理等）
- web-tools/ — Web相关工具（搜索、浏览、下载等）
- scripts/ — 可执行脚本，封装特定操作流程
- programs/ — 独立程序，功能更完整的工具
- plugins/ — 插件系统，可扩展的工具机制
- hooks/ — 事件钩子，在特定事件触发时执行
- mcp/ — Model Context Protocol，标准化模型-工具通信协议
- router/ — 路由，决定工具调用的路径
- commands/ — 命令定义，工具调用的声明式接口
- associated-programs/ — 与系统关联的辅助程序

**工具与skills的关系**：skills是"能力封装" — 包含逻辑和判断的完整能力单元。tools是"接口集合" — 提供执行能力但不包含业务逻辑。一个技能可能调用多个工具，一个工具也可能被多个技能调用。技能 = 工具 + 逻辑。

**工具与automation/的关系**：自动化区中的规则定义了"什么时候做什么"，工具提供了"做"的能力。自动化是"大脑的决策"，工具是"手脚的执行"。

**MCP协议的重要性**：MCP（Model Context Protocol）是工具与模型之间通信的标准协议。有了MCP，工具可以被统一的接口调用，模型不需要关心工具的具体实现。MCP是工具区的"通用接口标准"。

**计算机使用工具**：computer-use/是工具区中最具能力的子目录 — 它让模型能够直接操作计算机（GUI交互、文件管理、程序控制等）。这是模型从"思考"走向"行动"的关键桥梁。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 工具的安全边界 | 模型可以调用哪些工具？ | 受权限系统约束，高风险工具需要确认 |
| 工具是否需要沙箱 | 工具执行是否需要隔离环境？ | 高风险工具需要沙箱，低风险工具直接执行 |
| 工具与隔离区的关系 | 工具执行的结果放在哪里？ | 结果放入模型隔离区或对应模块 |
| MCP是否需要自定义 | 标准MCP是否需要扩展？ | 支持标准MCP + 自定义扩展 |

---

## 4. 我的判断

**使用工具区的标准动作**：
1. 需要执行操作 → 选择合适的工具
2. 调用工具 → 通过标准接口（MCP或其他接口）
3. 处理结果 → 工具返回结果后进行处理
4. 错误处理 → 工具执行失败时按规范处理
5. 工具组合 → 多个工具组合完成复杂任务

**工具区与其他模块的关系**：
- tools/ → skills/ ：技能调用工具，工具是技能的执行接口
- tools/ → automation/ ：自动化规则驱动工具执行
- tools/ → production_space/ ：生产空间中的工作通过工具执行
- tools/ → mcp/ ：MCP是工具的统一接口标准
- tools/ → computer-use/ ：计算机使用工具是最高能力的工具类型
