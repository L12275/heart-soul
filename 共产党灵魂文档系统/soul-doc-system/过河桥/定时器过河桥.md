# 过河桥 — 定时器区（Timers）

> **此过河桥是定时器模块的基因片段。**
> 定时器 = 系统的"时间引擎"，驱动所有需要定时触发的自动化任务。

---

## 1. 用户原话重述

用户将定时器区定义为：
- 系统的"时间引擎" — 驱动所有需要定时触发的自动化任务
- 当前包含README.md和四个子文件夹：
  - 单个定时循环 — 单个任务的定时循环
  - 团队定时任务 — 团队级别的定时任务
  - 心跳 — 系统心跳机制
  - 滚动定时任务 — 滚动式定时任务
- 定时器是自动化区的时间触发机制
- 与automation/的关系：自动化规则定义"做什么"，定时器定义"什么时候做"

---

## 2. 理解要点

**定时器的核心定位**：它是系统的时间驱动层。系统中很多任务不是用户触发就执行，而是需要在特定时间自动执行。定时器提供了这种时间驱动的能力。

**四类定时任务**：
- 单个定时循环 — 最基础的定时单元，单个任务按固定周期循环执行
- 团队定时任务 — 涉及多智能体协作的定时任务，需要协调多个参与者的执行时间
- 心跳 — 系统级的心跳机制，用于检测系统活性、保活连接
- 滚动定时任务 — 滚动式任务队列，任务按队列顺序滚动执行

**定时器与automation/的关系**：自动化区的合并协议有时间触发条件（如"每周至少合并一次"）。定时器提供具体的时间触发执行能力。自动化规则定义触发条件，定时器执行定时操作。

**定时器与discovery/的关系**：发现区的探索任务通常由定时器触发（如"每天探索一次新资源"）。定时器定义了探索的频率和时机。

**定时器与workflow/的关系**：工作流中的某些节点可能需要定时触发。定时器为工作流提供时间维度的调度能力。

**心跳机制**：心跳是最基础的定时任务类型 — 它不执行具体业务逻辑，而是发送一个"我还活着"的信号。心跳用于：
- 保持系统活性 — 防止系统因长时间无活动而被休眠
- 检测连接状态 — 如果心跳失败，说明连接可能已断开
- 协同保活 — 多智能体系统中的心跳用于确认各智能体的存活状态

**定时器的可靠性**：定时任务需要可靠的执行保障。如果定时器本身失效，所有依赖时间的自动化功能都会停止。心跳机制可以用来监控定时器的活性。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 定时任务的时区处理 | 使用哪个时区？ | 统一使用UTC，用户可配置本地时区转换 |
| 定时任务失败的处理 | 任务执行失败后是否重试？ | 支持重试机制，最大重试次数可配置 |
| 心跳的间隔 | 多久发一次心跳？ | 根据系统需求配置，默认间隔可调整 |
| 滚动定时任务的调度策略 | FIFO还是优先级？ | 支持多种调度策略，按任务类型选择 |

---

## 4. 我的判断

**使用定时器的标准动作**：
1. 需要定时任务 → 选择合适的定时任务类型
2. 配置定时器 → 设置周期/时间/触发条件
3. 启动定时器 → 加入定时任务队列
4. 监控 → 通过心跳机制确保定时器活性
5. 停止/修改 → 不再需要的任务及时清理

**定时器区与其他模块的关系**：
- timers/ → automation/ ：定时器是自动化规则的时间触发机制
- timers/ → discovery/ ：发现任务由定时器触发
- timers/ → workflow/ ：工作流中的时间节点由定时器驱动
- timers/ → 团队空间/ ：团队定时任务涉及多智能体协调
- timers/ → 智能体空间/ ：心跳机制监控智能体活性
