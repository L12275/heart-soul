# 过河桥 — 发现区（Discovery）

> **此过河桥是发现模块的基因片段。**
> 发现 = 系统主动探索和发现新信息、新资源、新机会的引擎，是系统的"雷达"。

---

## 1. 用户原话重述

用户将发现区定义为：
- 系统主动探索和发现新信息、新资源、新机会的引擎
- 当前只有一个README.md
- 发现 = 系统的"雷达" — 不是被动接收信息，而是主动扫描外部世界
- 发现的内容来源包括：互联网搜索、平台浏览、推荐系统、机会识别
- 与深度研究/的区别：发现是广度探索（发现有什么），深度研究是深度挖掘（把某个发现研究透）

---

## 2. 理解要点

**发现区的核心定位**：它是系统的"感知层"。系统不能只靠用户输入来获取信息，还需要主动去发现。发现区就是系统主动感知外部世界的功能模块。

**发现与深度研究的区别**：发现是广度优先 — 大量扫描，识别信号。深度研究是深度优先 — 选中一个信号，深入挖掘。两者构成"发现→研究"的工作流：发现区找到候选，深度研究区验证和分析。

**发现的内容类型**：
- 信息发现 — 新的知识、新的动态、新的趋势
- 资源发现 — 有用的工具、有用的资料、有用的平台
- 机会发现 — 新的可能性、新的方向、新的价值点

**发现的触发机制**：发现不是持续全功率运行的。它由定时器触发（如每天扫描一次），或在特定条件下触发（如完成一个任务后顺便探索相关领域）。

**发现的输出去向**：发现的内容不是直接写入发现区就完事了 — 有价值的发现需要被分发到各个模块：
- 有用的链接 → bookmarks/
- 有用的知识 → wiki/ 或 experience/
- 有价值的机会 → 记录到桌面等待决策
- 有用的工具 → tools/

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 发现的触发频率 | 是持续运行还是定时触发？ | 以定时触发为主，避免过度消耗资源 |
| 发现结果的优先级 | 所有发现都同等处理还是分级？ | 分级处理 — 立即行动/记录待审/观察跟踪 |
| 发现与用户输入的关系 | 用户要求搜索算不算发现？ | 用户指令触发的是定向搜索，发现区负责非定向的自主探索 |
| 发现的存储方式 | 按时间还是按主题组织？ | 按主题组织为主，时间戳为辅 |

---

## 4. 我的判断

**使用发现区的标准动作**：
1. 定时触发 → 启动探索流程
2. 扫描多个来源 → 收集原始信号
3. 初步筛选 → 判断信号的价值等级
4. 分发 → 有价值的发现进入对应模块
5. 定期回顾 → 调整探索策略和触发频率

**发现区与其他模块的关系**：
- discovery/ → 深度研究/ ：发现是发现的源头，深度研究是发现的深度加工
- discovery/ → bookmarks/ ：发现的好链接存入书签
- discovery/ → wiki/ ：发现的新知识消化后存入wiki
- discovery/ → experience/ ：发现的经验验证后存入经验库
- discovery/ → timers/ ：发现由定时器触发
- discovery/ → desktop/ ：重要发现先放到桌面等待决策
