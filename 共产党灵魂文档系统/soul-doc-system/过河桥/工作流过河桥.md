# 过河桥 — 工作流区（Workflow）

> **此过河桥是工作流模块的基因片段。**
> 工作流 = 任务执行的结构化流程定义，是系统的"任务调度蓝图"。

---

## 1. 用户原话重述

用户将工作流区定义为：
- 任务执行的结构化流程定义存储区
- 是系统的"任务调度蓝图" — 定义任务从开始到完成的完整路径
- 当前只有README.md
- 工作流从过河桥演化而来 — 过河桥定义"做什么"，工作流定义"按什么顺序做"
- 与sop/的区别：SOP是操作规范（怎么做一步），工作流是任务流程（多步的顺序和依赖）
- 工作流可能涉及多个模块的协同 — 跨越多个工作台和功能模块

---

## 2. 理解要点

**工作流的本质**：工作流是任务的"路径规划"。一个复杂任务不是一步完成的，而是分成多个步骤，每个步骤有先后顺序和依赖关系。工作流把这些步骤组织成一个可执行的流程。

**工作流与SOP的区别**：
- SOP = 操作规范 — "做这一步时应该遵循什么规范"
- 工作流 = 任务流程 — "整个任务包含哪些步骤，按什么顺序执行"

两者互补：SOP定义每一步的规范，工作流定义步骤之间的顺序和依赖。一个工作流中的每个节点可能对应一个SOP。

**工作流的层级**：
- 系统级工作流 — 跨模块的大流程（如"完成一个研究项目"的完整流程）
- 工作台级工作流 — 特定工作台内的流程（production_space/[工作台]/workflow/）
- 任务级工作流 — 单个任务的执行流程

**工作流的执行机制**：工作流不只是静态的流程文档，它应该是可执行的。系统可以根据工作流定义自动调度任务、协调步骤、管理依赖。工作流从"文档"演化到"可执行引擎"。

**工作流与过河桥的关系**：过河桥是"意图的完整创造模板"（五部分结构），工作流是"意图的执行蓝图"。过河桥定义"要什么"，工作流定义"按什么路径得到"。

**工作流与timers的关系**：工作流中的某些节点可能需要定时触发（如"等待一段时间后再执行下一步"）。定时器为工作流提供时间维度的调度能力。

**工作流与automation的关系**：自动化规则可以触发工作流的执行。例如，当某个条件满足时（自动化触发），启动对应的工作流。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 工作流是否需要可视化 | 流程图还是纯文本？ | 纯文本为主，流程图为辅 |
| 工作流是否支持分支 | 条件分支如何处理？ | 支持 — 工作流定义中可包含条件分支 |
| 工作流的执行状态 | 执行到哪一步是否需要追踪？ | 需要 — 工作流执行状态是任务管理的基础 |
| 工作流是否可以嵌套 | 一个工作流中是否可以调用另一个工作流？ | 支持 — 工作流可以组合和嵌套 |

---

## 4. 我的判断

**使用工作流的标准动作**：
1. 开始复杂任务 → 先定义工作流（或选择已有工作流）
2. 按工作流执行 → 逐步完成每个节点
3. 节点执行 → 按SOP规范执行每个步骤
4. 状态追踪 → 记录工作流的执行进度
5. 完成 → 产出结果，工作流归档

**工作流区与其他模块的关系**：
- workflow/ → sop/ ：工作流中的每个节点按SOP执行
- workflow/ → 过河桥/ ：工作流从过河桥演化而来
- workflow/ → timers/ ：工作流的时间节点由定时器驱动
- workflow/ → automation/ ：自动化规则触发工作流执行
- workflow/ → production_space/ ：每个工作台有自己的工作流
- workflow/ → 代办与进度/ ：工作流执行状态映射到代办进度
