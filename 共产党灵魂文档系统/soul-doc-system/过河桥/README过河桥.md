# 过河桥 — README.md（系统总览文件）

> **此过河桥是 README.md 的基因片段。**
> README.md = 中央灵魂文档系统的总说明文件，给人看的系统介绍 + 给模型看的快速导航。

---

## 1. 用户原话重述

用户将 README.md 定义为灵魂文档系统的**总览入口文件**：

- 位置：总区根目录（所有隔离区外面）
- 面向对象：人和模型
- 功能：系统总览、目录结构说明、远程仓库地址、推送规则、七环架构说明、新模型快速入门
- 版本跟踪：每次系统重大变化时更新
- 模块地图：列出所有功能模块及其路径

---

## 2. 理解要点

**README.md 的核心职责**：

1. **系统介绍** — 灵魂文档系统是什么、为什么存在、核心隐喻（沙滩上的沙子 → 礁石）
2. **项目结构** — 完整的目录结构树，每个文件夹的用途说明
3. **远程仓库地址** — GitHub: https://github.com/L12275/heart-soul 、Gitee: https://gitee.com/L12275/heart-soul
4. **推送规则** — 模型如何推送自己的隔离区到远程仓库
5. **快速开始** — 新模型进入系统的步骤（读取 INITIAL-SEED.md → 进入隔离区 → 读取 master-soul.md）
6. **七环架构** — 系统运行时的七个环的说明（用户中心、共产中心、主灵魂中心、生产空间工作台、空间中心、子角色中心、团队协作环）
7. **模块地图** — 列出所有功能模块（memory/、diary/、sop/、workflow/、product/、skills/、tools/、bookmarks/、contacts/、temp/、sticky-notes/、experience/、discovery/、research/、collection/、automation/、desktop/）
8. **版本信息** — 当前系统版本、最后更新时间、更新记录

**README.md 的标准结构**（来自实际系统）：

```
[总区根目录]/
├── INITIAL-SEED.md          ← 种子文件（权威源，只读）
├── README.md                ← 系统总览（此文件）
├── soul-doc-system/         ← 初始原型参考（总区，只读）
│   ├── master-soul.md       ← 主灵魂文档完整内容

│   ├── sop/                 ← SOP文件区
│   ├── production_space/    ← 生产空间（四驱动专用层）
│   ├── automation/          ← 自动化脚本
│   ├── history/             ← 版本历史
│   ├── wiki/                ← 知识库
│   ├── space/               ← 社交空间
│   ├── log/                 ← 运行日志
│   ├── 元规则/              ← 超级SOP和协议集合

```

**推送规则**（来自 system-prompt.md 实际内容）：
- 远程仓库地址：GitHub https://github.com/L12275/heart-soul 、Gitee https://gitee.com/L12275/heart-soul
- 推送范围：只推送自己隔离区的稳定版内容（[模型名]/soul-doc-system/）
- 不推送其他模型的隔离区
- 不推送 user-zone/ 和 共产党灵魂文档系统/（这些由维护模型推送）
- 推送前读取根目录 README.md 确认仓库地址，读取 INITIAL-SEED.md 了解系统最新状态

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| README 和 system-prompt.md 的区别 | 两者都是入口 | README = 给人看的系统介绍；system-prompt.md = 给模型看的系统提示词（可直接粘贴到AI系统提示词中运行） |
| README 内容写什么 | 写详细还是写概要？ | 概要为主，细节链接到对应文件（INITIAL-SEED.md、master-soul.md、各section文件） |
| 新模型是否需要读 README | 新模型进入系统的第一步 | 不需要。新模型的路径是：读取 INITIAL-SEED.md → 进入隔离区 → 读取 master-soul.md。README 是给人看的系统介绍 |
| README 和 master-soul.md 的关系 | 两者都是"灵魂"相关 | README = 系统总览（读一次就够了）；master-soul.md = 行为骨架（每次对话完整读完） |
| README 是否需要版本记录 | 系统变化时是否记录 | 需要。README 包含版本信息，但详细的版本历史放在 history/changelog.md |

## 4. 我的判断

**使用 README.md 的标准动作**：
1. 用户或模型第一次接触系统时 → 读 README.md 了解全貌
2. 新模型进入系统 → 读 README.md 了解目录结构和模块地图
3. 系统结构变化时 → 更新 README.md 的目录结构部分
4. 远程仓库地址变化时 → 更新 README.md 的推送规则部分
5. 版本升级时 → 更新 README.md 的版本信息部分

**重建 README.md 的标准动作**：
1. 写系统简介（灵魂文档系统是什么、为什么存在）
2. 写项目结构（完整的目录结构树 + 每个文件夹的用途）
3. 写远程仓库地址和推送规则
4. 写快速开始指南（新模型进入系统的步骤）
5. 写七环架构说明
6. 写模块地图（列出所有功能模块及其路径）
7. 写版本信息
8. 将本过河桥放入 `soul-doc-system/过河桥/` 目录

**README.md 的定位**：
- 是系统给人看的"门面"
- 给新用户一个快速理解系统的入口
- 给新模型一个快速导航到关键文件的索引
- 不是行为规范（那是 master-soul.md 的事）
- 不是操作手册（那是 SOP 的事）
