# 过河桥 — 自动化区（Automation）

> **此过河桥是自动化模块的基因片段。**
> 自动化 = 模型的自我编辑、质量检查、路线图和合并协议的运行机制。

---

## 1. 用户原话重述

用户将自动化区定义为：
- 模型自我维护和自动化的运行机制集合
- 包含合并协议（merge-protocol.md）— 定义sections中规则安全合并到master-soul.md的流程
- 包含质量检查（quality-check.md）— 定义如何检查合并后内容的质量
- 包含路线图（roadmap.md）— 系统的演进规划
- 包含自我编辑提示词（self-edit-prompt.md）— 模型自我修改的提示词模板
- 合并触发条件：时间触发（每周）、数量触发（5条已验证规则）、用户触发、质量触发
- 合并前必须逐项检查：冲突检查、重复检查、表述检查、结构检查

---

## 2. 理解要点

**自动化区的本质**：它不是存储内容的模块，而是存储"模型如何自我管理"的规则集合。这些规则定义了模型在无人干预的情况下，如何维护系统质量、如何进化系统内容。

**合并协议的核心逻辑**：sections/中的新规则需要经过验证后才能合并到master-soul.md。合并不是简单追加，而是经过冲突检查、重复检查、表述检查、结构检查四道关卡的安全流程。合并后还需要验证，验证不通过则回滚。

**质量检查的作用**：在合并前后都有质量检查环节，确保合并后的master-soul.md不会因为新增规则而降低质量。质量检查是合并协议的守门员。

**路线图的功能**：roadmap.md记录了系统的演进方向，是长期规划的载体。它不是短期SOP，而是对未来发展的展望和承诺。

**自我编辑提示词**：self-edit-prompt.md是模型修改自身行为时的提示词模板。它不是普通的工作提示词，而是"模型修改模型"的特殊提示词，需要特别谨慎的设计。

**自动化区与SOP的关系**：自动化区中的规则比普通SOP更底层。普通SOP指导"做事"，自动化规则指导"模型如何修改做事的方式"。两者是层级关系，不是并列关系。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 自动化区是否包含定时任务 | timers/是独立模块，但自动化也需要定时触发 | 自动化区的合并协议有时间触发条件，timers/提供具体的定时执行能力，两者配合 |
| 自我编辑的范围 | 模型可以自我编辑哪些文件？ | 仅限于自动化区定义的范围，不能越权修改用户空间 |
| 合并协议的权威性 | sections中的规则和master-soul.md哪个优先级高？ | master-soul.md是权威源，sections是待验证的候选区 |
| 质量检查的粒度 | quality-check是检查单个规则还是整体文档？ | 两者都要 — 单个规则的四项检查 + 整体文档的可读性验证 |

---

## 4. 我的判断

**使用自动化区的标准动作**：
1. 新增规则到sections/ → 等待验证
2. 达到合并触发条件 → 启动合并协议
3. 逐项通过四道安全检查 → 执行合并
4. 合并后验证 → 不通过则回滚
5. 定期回顾roadmap.md → 调整长期方向
6. 需要修改系统行为时 → 使用self-edit-prompt作为模板

**自动化区与其他模块的关系**：
- automation/ → sections/ 的合并守门员
- automation/ → master-soul.md 的质量保障者
- automation/ → sop/ 的升级机制（SOP进化由自动化规则驱动）
- automation/ → timers/ 的时间触发依赖方
