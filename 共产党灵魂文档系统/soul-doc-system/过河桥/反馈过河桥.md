# 过河桥 — 反馈区（Feedback）

> **此过河桥是反馈模块的基因片段。**
> 反馈 = 用户对系统运行结果的评价和修正意见，是系统进化的直接输入。

---

## 1. 用户原话重述

用户将反馈区定义为：
- 用户对系统运行结果的评价和修正意见的存储区
- 当前包含README.md和一个具体的反馈文件：2026-06-23_F-001_子角色模板GA修正反馈.md
- 反馈是系统进化的直接输入 — 没有反馈就没有正确的进化方向
- 反馈文件命名规范：日期 + 编号 + 主题 + "反馈"后缀
- 反馈与自动化区的关系：自动化区的合并协议中有"质量触发"和"用户触发" — 用户反馈直接触发系统调整

---

## 2. 理解要点

**反馈区的核心定位**：它是系统的"校准器"。无论模型运行得多么好，没有用户的反馈，系统就无法知道哪里需要改进。反馈区是用户声音的系统化入口。

**反馈的价值层次**：
- 纠正性反馈 — 用户指出模型做错了什么
- 方向性反馈 — 用户指出模型走错了方向
- 建设性反馈 — 用户给出改进的具体建议
- 评价性反馈 — 用户对结果的满意度评价

**反馈的处理流程**：
1. 接收反馈 → 存入feedback/，按规范命名
2. 分析反馈 → 识别反馈的类型和优先级
3. 执行修正 → 根据反馈调整系统
4. 记录结果 → 在反馈文件中记录处理结果
5. 归档 → 处理完成后的反馈移至archive/或保留在feedback/作为历史

**反馈与automation/的关系**：自动化区的合并协议中有"用户触发"条件 — 用户反馈可以直接触发系统规则的合并和调整。反馈是自动化系统的重要外部输入。

**反馈与experience/的关系**：有价值的反馈经过处理后可以转化为经验。用户纠正的错误 → 标记为suspected-hallucination的教训 → 沉淀为proven的正确经验。

**反馈与sections/的关系**：用户反馈是sections/中新增规则的重要来源。用户指出问题 → 转化为规则 → 经过验证 → 合并到master-soul.md。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 反馈的存储期限 | 处理后的反馈是否保留？ | 保留 — 反馈是进化历史的记录 |
| 反馈的优先级 | 所有反馈同等处理还是分级？ | 分级 — 纠正性最高，评价性次之 |
| 反馈是否需要回复 | 模型是否需要对每份反馈做出回应？ | 是，在反馈文件中记录处理结果 |
| 反馈与行内批注的关系 | 用户在文件中的【】批注算不算反馈？ | 算 — 行内批注是即时反馈，feedback/是结构化反馈 |

---

## 4. 我的判断

**处理反馈的标准动作**：
1. 收到用户反馈 → 立即存入feedback/，按命名规范
2. 分析反馈类型 → 纠正性/方向性/建设性/评价性
3. 制定处理方案 → 在反馈文件中记录"我的判断"
4. 执行修正 → 调整对应模块
5. 记录结果 → 在反馈文件中标记处理状态
6. 转化经验 → 有价值的反馈沉淀为经验

**反馈区与其他模块的关系**：
- feedback/ → automation/ ：反馈触发自动化调整
- feedback/ → sections/ ：反馈转化为规则候选
- feedback/ → experience/ ：反馈沉淀为经验
- feedback/ → master-soul.md ：重要反馈可能影响核心规范
- feedback/ → diary/ ：反馈处理过程记录在日记中
