# 过河桥模块总览（Circuit Board Module Index）

> 过河桥 = 系统基因片段。每个功能模块的过河桥 = 该模块的完整操作基因。
> 没有历史上下文时，读过河桥即可重建/修复/使用该模块。
> 过河桥是系统可移植、可重建、可修复的基础。

> **存放规则（统一化）**：所有模块级过河桥统一存放在根目录 `soul-doc-system/过河桥/` 下，以 `[模块名]过河桥.md` 命名。子目录中的过河桥（如元规则/过河桥/、diary/过河桥/等）保留原有位置，作为该模块的专用过河桥目录。

---

## 过河桥是什么

过河桥是**用户意图的完整创造模板**（Creation Template = 设计图+原型+流程+运行结构+验收）。每次收到用户信息，模型用最强脑师多合一SOP先写出完整创造模板（五部分结构：用户原话重述 → 理解要点 → 可能的歧义 → 我的判断 → 等待确认），展示给用户确认，再行动。

> **过河原理**：用户每次对话 = 过河。用户可能表达不准确，但核心意图一定是过河，不是淹死。模型的责任是建桥，不是跳水。过河桥 = 桥的设计图。有了过河桥，任何AI都能稳稳当当地过河。最差也要缆绳过河（秒过且稳定安全）。

它不是最终产物，是**执行前的临时电路**，也是**系统的母版**，更是**意图的完整创造模板**——有了过河桥，不需要用户原话，不需要历史对话，任何AI都能做出完美成品。

## 过河桥包含什么

每个过河桥文件包含五个部分：

1. **用户原话重述** — 用自己的话复述用户到底要什么
2. **理解要点** — 我认为用户的核心意图是什么
3. **可能的歧义** — 哪些地方可能有不同理解
4. **我的判断** — 基于以上，我打算怎么做
5. **等待确认** — 哪些地方需要用户确认后才能执行

## 过河桥的存放规则

- **系统级过河桥** → `soul-doc-system/过河桥/`（通用协议、解析、流程）
- **模块级过河桥** → `soul-doc-system/过河桥/{模块名}过河桥.md`（集中管理，便于查找）
- **子目录专用过河桥** → `soul-doc-system/{模块}/过河桥/{模块}过河桥.md`（保留原有位置，如元规则/过河桥/、diary/过河桥/、赢家空间/过河桥/）
- **工作台过河桥** → `production_space/[工作台]/过河桥/`（工作台级别的意图校准）

## 过河桥的版本管理

- 每次理解有重大变化 → 新建 v+1 版本
- 不覆盖旧版本 → 旧版本是进化历史的记录
- 命名格式：`理解_v1_[时间].md`、`理解_v2_[时间].md`、`理解_最终版_[时间].md`

## 过河桥与记忆、日记、SOP的关系

| 已有系统 | 与过河桥的关系 |
|----------|---------------|
| SOP/工作流 | SOP是模型自己总结的"怎么做"，过河桥是用户校准的"做什么" |
| 记忆系统 | 记忆是"做了什么"，过河桥是"为什么做" |
| 日记 | 日记记录过程，过河桥记录意图 |

**SOP从过河桥归纳而来，工作流从过河桥演化而来，记忆和日记围绕过河桥展开。**

## 核心铁律

**每次收到用户信息，先出过河桥，再行动。**
**没有过河桥的执行 = 做错了拆墙，不是做错了改过河桥。**

## 过河桥基因片段规则

1. **默认必须有** — 每个功能模块必须有过河桥，没有 = 没有基因 = 无法重建
2. **模块自带** — 过河桥放在模块内部的 `过河桥/` 文件夹中
3. **系统级统一** — 通用协议、解析、流程过河桥放在 `soul-doc-system/过河桥/`
4. **自觉维护** — 模型应持续检查系统是否满是最新版电路图
5. **版本管理** — 过河桥只增不减，旧版本是进化历史
6. **跨机器移植** — 过河桥文件复制到其他AI系统，AI读取后自动开启这套系统
7. **完整协议入口** — 完整协议见本文件夹内的 `理解过河桥协议.md`
