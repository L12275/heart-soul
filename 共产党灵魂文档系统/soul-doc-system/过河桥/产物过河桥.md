# 过河桥 — 产物区（Product）

> **此过河桥是产物模块的基因片段。**
> 产物 = 系统运行产生的正式产出物的统一存放区，是系统的"成果仓库"。

---

## 1. 用户原话重述

用户将产物区定义为：
- 系统运行产生的正式产出物的统一存放区
- 当前有三个子文件夹：执行类（execution outputs）、文档类（documentation）、研究类（research）
- 产物是系统工作的最终交付物 — 不是过程文件，不是草稿，而是完成品
- 与desktop/的区别：desktop是过程中的临时工作区，product是最终成果的正式存放
- 与production_space/的区别：production_space是"生产的地方"，product是"生产的结果"

---

## 2. 理解要点

**产物区的核心定位**：它是系统的"成果仓库"。系统中所有的工作最终都要产出某种形式的成果 — 执行的结果、编写的文档、研究的结论。产物区统一存放这些成果，形成系统的价值输出。

**三分类体系**：
- 执行类 — 任务执行的产出物（报告、结果、输出）
- 文档类 — 编写的文档（文章、说明、指南）
- 研究类 — 研究的产出物（分析、结论、数据）

**产物与process的关系**：产物区存放的是结果，不是过程。过程记录在diary/和memory/，产物只保留最终的交付物。但产物可以附带简短的制作说明（链接到对应的diary/记录）。

**产物的质量标准**：产物区的内容应该达到一定的质量标准才能放入 — 完成度、正确性、可用性。半成品留在desktop/或production_space/，只有达标的内容才能升级到product/。

**产物与memory/experience的关系**：产物的核心内容可以提炼为记忆和经验。例如，研究类产物的核心结论 → 进入experience/；执行类产物的关键结果 → 进入memory/。产物是"完整输出"，记忆/经验是"精华提取"。

**产物的版本管理**：产物可能有多个版本（初版、修订版、终版）。版本管理在产物区内部进行，不覆盖旧版本，保留完整产出轨迹。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 产物的分类边界 | 一个产出物可能同时属于多个分类怎么办？ | 主分类存放，其他分类建立引用 |
| 半成品是否可以放入产物区 | 未完成的工作是否允许进入？ | 不允许 — 产物区只存放完成品 |
| 产物是否需要元数据 | 是否需要记录产物的制作时间、作者、版本？ | 需要 — 每个产物应有基本的元数据 |
| 产物与wiki的关系 | 研究类产物和wiki内容是否重叠？ | 产物是"某次研究的完整输出"，wiki是"系统化的知识体系" |

---

## 4. 我的判断

**产出到产物区的标准动作**：
1. 工作完成 → 产出物达到质量标准
2. 分类 → 确定产物属于执行类/文档类/研究类
3. 存放 → 放入对应子文件夹，附带元数据
4. 提炼 → 产物的精华内容同步到memory/和experience/
5. 版本管理 → 新版本不覆盖旧版本

**产物区与其他模块的关系**：
- product/ → desktop/ ：桌面是草稿区，产物区是成品区
- product/ → production_space/ ：生产空间是生产过程，产物区是生产结果
- product/ → memory/ ：产物的核心内容提炼为记忆
- product/ → experience/ ：产物的验证经验沉淀为经验
- product/ → wiki/ ：产物的知识内容消化后进入wiki
- product/ → archive/ ：过时产物归档到archive/
