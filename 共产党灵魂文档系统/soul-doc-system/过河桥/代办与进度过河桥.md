# 过河桥 — 代办与进度（Todo & Progress）

> **此过河桥是代办与进度模块的基因片段。**
> 代办与进度 = 任务管理的看板，是系统的"任务仪表盘"。

---

## 1. 用户原话重述

用户将代办与进度区定义为：
- 任务管理的看板系统
- 是系统的"任务仪表盘" — 集中展示所有待办、进行中、已完成的任务
- 当前包含README.md
- 代办与进度是工作流执行状态的集中展示 — 工作流定义"做什么"，代办看板展示"做得怎么样了"
- 与workflow/的区别：workflow定义流程结构，代办看板展示实际执行状态

---

## 2. 理解要点

**代办与进度的核心定位**：它是系统的"任务仪表盘"。系统中有大量任务同时运行 — 有些待开始，有些进行中，有些已完成。代办看板提供了这些任务状态的集中视图。

**任务状态分类**：
- 待办（Todo）— 已规划但未开始的任务
- 进行中（In Progress）— 正在执行的任务
- 已完成（Done）— 已完成的任务
- 阻塞（Blocked）— 因依赖问题无法推进的任务

**代办与workflow的关系**：工作流定义任务的执行流程，代办看板追踪任务的执行状态。工作流是"静态的路径规划"，代办看板是"动态的执行追踪"。一个工作流的执行过程在代办看板上实时可见。

**代办与production_space的关系**：生产空间中的工作台各自有任务，这些任务的全局视图在代办看板中。代办看板提供了跨工作台的任务统一管理。

**代办与timers的关系**：有些任务有截止时间或定时触发。定时器为代办任务提供时间维度的管理。

**代办与自动化区的关系**：自动化区的合并协议中有触发条件（如"积累5条已验证规则时触发"）。这些触发条件本质上是代办任务 — "当满足条件时执行某个操作"。

**进度的度量**：进度不仅是"完成了多少"，还包括"质量如何"。一个任务完成了但质量不达标，进度状态应该是"待修正"而不是"已完成"。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 代办看板的组织方式 | 按项目分类还是按优先级？ | 两者结合 — 按项目分组，按优先级排序 |
| 任务的生命周期 | 完成的任务是否保留在代办看板上？ | 保留一段时间后归档 |
| 任务依赖的处理 | 阻塞任务如何管理依赖关系？ | 明确标注依赖项和阻塞原因 |
| 代办是否需要提醒 | 任务到期是否需要主动提醒？ | 需要 — 与定时器配合，到期提醒 |

---

## 4. 我的判断

**使用代办与进度的标准动作**：
1. 创建任务 → 写入代办看板，标注状态和优先级
2. 开始执行 → 状态更新为"进行中"
3. 完成 → 状态更新为"已完成"，记录完成质量
4. 遇到阻塞 → 状态更新为"阻塞"，标注原因和依赖
5. 定期回顾 → 检查待办列表，调整优先级

**代办与进度区与其他模块的关系**：
- 代办与进度/ → workflow/ ：工作流执行状态映射到代办看板
- 代办与进度/ → production_space/ ：各工作台的任务在全局看板汇总
- 代办与进度/ → timers/ ：定时提醒与代办任务配合
- 代办与进度/ → automation/ ：自动化触发的任务记录在代办中
- 代办与进度/ → 团队空间/ ：团队任务在全局看板中协调
