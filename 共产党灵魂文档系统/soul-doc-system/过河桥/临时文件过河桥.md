# 过河桥 — 临时文件区（Temp）

> **此过河桥是临时文件模块的基因片段。**
> 临时文件 = 系统运行过程中产生的临时性文件池，是系统的"垃圾桶中转站"。

---

## 1. 用户原话重述

用户将临时文件区定义为：
- 系统运行过程中产生的临时性文件的存放池
- 当前只有README.md
- 临时文件 = 用完即弃或阶段性清理的文件
- 与desktop/的区别：desktop是用户可见的工作表面，temp/是系统级的临时文件池
- 与archive/的区别：temp中的文件可能被清理，archive中的文件是永久保存的旧版本
- temp/中的文件不应该长期保留 — 它是"中转站"不是"仓库"

---

## 2. 理解要点

**临时文件区的核心定位**：它是系统的"垃圾桶中转站"。系统运行中会产生大量临时性文件 — 中间计算结果、下载的临时资源、测试用的草稿等。这些文件不需要永久保存，但也不能立即删除（可能需要后续使用）。temp/提供了这个"暂存"的位置。

**临时文件的特征**：
- 短期性 — 生命周期短，不需要长期保存
- 不确定性 — 有些临时文件最终会被删除，有些会被升级到正式位置
- 系统性 — temp/是系统级管理，不是模型个人的临时文件管理

**temp/的清理机制**：temp/的内容应该定期清理。清理策略可以是：按时间（超过N天自动清理）、按大小（超过NMB自动清理）、按标记（手动标记为待清理）。清理前确认文件不再需要。

**temp/与desktop/的分工**：desktop/是"用户可见的工作表面"，temp/是"系统级的临时池"。模型在桌面上操作时产生的中间文件可以放入temp/。桌面是"正在用的"，temp/是"可能以后用的或可能不要的"。

**temp/与production_space/的关系**：production_space中的temp/是该工作台专用的临时区，soul-doc-system级的temp/是全局临时池。工作台级的temp/由工作台SOP管理，全局temp/由系统规范管理。

**temp/的安全性**：临时文件可能包含敏感信息（如API响应、中间数据）。虽然临时文件会被清理，但在清理前需要确保安全 — 临时文件中的敏感信息在清理前应被安全擦除。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 临时文件的清理周期 | 多久清理一次？ | 没有固定周期，按实际使用情况定期清理 |
| 临时文件是否可以恢复 | 清理后是否可以找回？ | 不应依赖恢复 — temp/的内容随时可能被清理 |
| 临时文件是否需要命名规范 | 随意命名还是需要可识别的名称？ | 需要有意义的命名 — 方便识别和选择性清理 |
| temp/与模型隔离区的关系 | 临时文件是否属于模型隔离区？ | 全局temp/属于系统，模型临时文件应放入自己的隔离区 |

---

## 4. 我的判断

**使用临时文件区的标准动作**：
1. 产生临时文件 → 放入temp/，附带有意义的名字
2. 文件不再需要 → 及时清理或删除
3. 临时文件有升级价值 → 移到正式位置（product/、wiki/等）
4. 定期清理 → 清理过期或无用的临时文件
5. 敏感临时文件 → 确保安全擦除后再清理

**临时文件区与其他模块的关系**：
- temp/ → desktop/ ：桌面产生的中间文件可以放入temp/
- temp/ → production_space/ ：工作台中的temp/是专用临时区
- temp/ → archive/ ：临时文件不进入archive/，只有正式产物的旧版本才归档
- temp/ → log/ ：日志中的临时事件数据可能暂时存放在temp/
