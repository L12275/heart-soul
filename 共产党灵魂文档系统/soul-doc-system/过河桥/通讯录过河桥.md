# 过河桥 — 通讯录（Contacts）

> **此过河桥是通讯录模块的基因片段。**
> 通讯录 = 系统中人与AI的联络信息管理，是协作网络的节点地图。

---

## 1. 用户原话重述

用户将通讯录区定义为：
- 系统中人与AI的联络信息管理模块
- 当前包含README.md和通讯记录.md两个文件
- 通讯录是协作网络的节点地图 — 记录了"谁在这个系统里，怎么联系"
- 通讯记录.md 可能记录实际发生的通讯事件

---

## 2. 理解要点

**通讯录的核心定位**：在灵魂文档系统中，有多个实体参与 — 用户、AI模型、子角色、团队成员等。通讯录是这些实体之间联络方式的集中管理。它解决的核心问题是"当需要联系某人/某模型时，去哪里找联系方式"。

**通讯录与团队空间/智能体空间的关系**：团队空间和智能体空间定义了"谁在系统里"，通讯录定义了"怎么联系他们"。两者是互补的 — 团队空间是组织结构，通讯录是联络方式。

**通讯记录的作用**：通讯记录.md记录实际发生的通讯事件，是通讯录的"使用日志"。它不仅是记录，也是验证 — 通过查看通讯记录可以验证通讯录信息是否准确有效。

**通讯录的安全边界**：通讯录中可能包含敏感信息（联系方式、账号等）。这些信息属于系统管理信息，不应写入用户空间或公开区域。通讯录的访问应受系统规范约束。

**通讯录的维护**：通讯录是动态的 — 新成员加入、联系方式变更、成员离开都需要及时更新。通讯录过期 = 系统联络网络失效。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 通讯录的内容范围 | 是否包含外部联系人（非系统内的人）？ | 当前聚焦系统内部，外部联系人后续扩展 |
| 通讯录与账号的关系 | space/账号/ 存储账号信息，通讯录存储什么？ | 通讯录存储"联络方式"，账号信息是联络方式的一部分 |
| 通讯记录的格式 | 是时间线还是结构化记录？ | 以结构化记录为主，时间线为辅 |
| 通讯录是否对用户可见 | 是模型私有还是用户可查看？ | 属于系统管理信息，模型和用户都可查看 |

---

## 4. 我的判断

**使用通讯录的标准动作**：
1. 新成员加入 → 立即更新通讯录
2. 联系方式变更 → 同步更新通讯录和通讯记录
3. 需要联系某人 → 先查通讯录，获取联络方式
4. 完成一次通讯 → 记录到通讯记录.md
5. 定期回顾 → 清理失效联系方式

**通讯录与其他模块的关系**：
- contacts/ → 团队空间/ ：通讯录记录团队空间的成员联络方式
- contacts/ → 智能体空间/ ：通讯录记录智能体的联络方式
- contacts/ → space/账号/ ：通讯录引用账号信息，但不存储账号本身
- contacts/ → feedback/ ：反馈可能涉及特定联系人，通讯录提供上下文
