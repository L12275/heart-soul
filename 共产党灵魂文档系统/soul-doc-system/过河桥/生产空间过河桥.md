# 过河桥 — 生产空间（Production Space）

> **此过河桥是生产空间模块的基因片段。**
> 生产空间 = 系统各类工作的专用"工作台"集合，每个工作台是一个独立的生产环境。

---

## 1. 用户原话重述

用户将生产空间定义为：
- 系统各类工作的专用"工作台"集合
- 当前包含README.md和多个工作台：
  - 代码工作台（desktop/diary/memory/product/sop/temp/workflow）
  - 文章工作台（同上结构）
  - 研究工作台（同上结构）
  - 科学工作台（同上结构）
  - 视频工作台（同上结构）
  - 音乐工作台（同上结构）
  - 项目工作台（同上结构）
  - 通用默认工作台（同上结构）
  - 新建工作台模版（同上结构）
- 每个工作台内部结构相同：desktop/ + diary/ + memory/ + product/ + sop/ + temp/ + workflow/
- 生产空间 = "建房子"的地方，与desktop/的"摊开图纸"形成工作流闭环

---

## 2. 理解要点

**生产空间的核心定位**：它是系统的"专业工作室集合"。不同类型的生产任务需要不同的工作环境 — 写代码需要代码工作台，写文章需要文章工作台，做研究需要研究工作台。生产空间为每种工作类型提供了独立且完整的工具链。

**工作台的标准结构**：每个工作台内部遵循相同的子模块结构 — desktop（草稿）、diary（过程记录）、memory（工作记忆）、product（产出物）、sop（工作SOP）、temp（临时文件）、workflow（工作流）。这种统一结构降低了模型在不同工作台之间切换的学习成本。

**通用默认工作台的作用**：新建工作台模版为新工作台提供标准结构。任何新型工作都可以基于通用默认工作台快速创建，保持系统结构的一致性。

**生产空间与desktop/的关系**：desktop/是全局的临时工作表面，production_space/中的desktop/是特定工作台的草稿区。全局桌面用于跨项目的临时工作，工作台桌面用于特定项目的深度工作。

**生产空间的工作流**：在某个工作台内，内容从desktop（草稿）→ workflow（流程）→ product（成品），过程中记录在diary/，经验沉淀在memory/和sop/。每个工作台内部形成了一个完整的工作闭环。

**工作台的独立性**：每个工作台是相对独立的 — 有自己独立的记忆、SOP、产出。但工作台之间可以通过memory/context/共享全局上下文，通过经验系统共享跨工作台的知识。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 工作台的数量是否需要限制 | 是否可以无限创建新工作台？ | 不需要硬性限制，但新工作台应有明确的使用场景 |
| 工作台之间的隔离程度 | 工作台内容是否完全隔离？ | 相对隔离，通过全局memory/context共享必要上下文 |
| 通用默认工作台是否可以直接使用 | 还是必须先复制为新工作台？ | 可以直接使用，但建议复制后定制 |
| 工作台SOP与全局SOP的关系 | 工作台内有sop/，全局也有sop/ | 工作台SOP = 该领域专用SOP；全局SOP = 通用SOP |

---

## 4. 我的判断

**使用生产空间的标准动作**：
1. 开始新类型的工作 → 选择对应工作台（或基于模版创建新的）
2. 在工作台内按标准结构组织内容
3. 草稿 → desktop/，流程 → workflow/，成品 → product/
4. 过程记录 → diary/，经验沉淀 → memory/ + sop/
5. 工作完成 → 产物的精华提炼到全局memory/和experience/

**生产空间与其他模块的关系**：
- production_space/ → desktop/ ：工作台内部桌面 vs 全局桌面
- production_space/ → product/ ：生产空间产生产物，产物区存放产物
- production_space/ → sop/ ：工作台内有专用SOP，全局SOP是通用SOP
- production_space/ → memory/ ：工作台记忆 + 全局记忆 = 四驱动
- production_space/ → workflow/ ：每个工作台有自己的工作流
