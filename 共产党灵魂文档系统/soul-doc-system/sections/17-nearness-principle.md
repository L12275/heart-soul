十七、近原则（Nearness Principle）

> 文件存放的黄金法则：内容放在离使用场景最近的位置。
> 与人类记忆的相似度检索原理一致——关系越近，回忆越快越准。

---

## 17.1 什么是近原则

**核心：** 一个文件/文件夹放在哪里，取决于**谁最常使用它、在哪里使用它**。

- 工作台专用的SOP → 放在该工作台内的 sop/，不放在根目录 sop/
- 工作台的通讯录 → 放在该工作台内的 contacts/，不放在根目录 desktop/contacts/
- 通用工具清单 → 放在根目录 sections/11-tools.md，因为所有工作台都可能用到
- 特定工作台的工具补充 → 放在该工作台内，不污染通用清单

---

## 17.2 判断标准：三问法

每次创建/移动文件时，问自己三个问题：

### 问1：谁用？

| 使用者 | 放哪里 |
|--------|--------|
| 所有工作台/所有模型 | 根目录通用文件夹 |
| 特定工作台 | 该工作台内部 |
| 特定生产空间类型 | 该生产空间类型内部 |
| 特定子角色 | 该子角色文件夹内部 |
| 特定模型 | 该模型隔离区内部 |

### 问2：在哪用？

| 使用场景 | 放哪里 |
|---------|--------|
| 在音乐工作台干活时 | 音乐工作台/ 内 |
| 在代码工作台干活时 | 代码工作台/ 内 |
| 在子角色切换时 | 该子角色 vm/ 内 |
| 在主灵魂加载时 | master-soul.md / sections/ 内 |
| 在研究时 | 深度研究/[项目]/ 内 |

### 问3：频率？

| 频率 | 放哪里 |
|------|--------|
| 随时可能用到 | 根目录（最近入口） |
| 特定场景才用 | 该场景的文件夹内 |
| 几乎不用 | 归档或删除 |

## 17.4 近原则的递归应用

近原则在每一层都适用：

```
soul-doc-system/              ← 根目录：通用内容
  sections/11-tools.md        ← 所有模型通用工具清单
  sop/master-sop.md           ← 所有工作台通用SOP

production_space/
  工作室/                      ← 工作室级：跨工作台共用
    音乐工作台/                ← 工作台级：音乐专用
      sop/编曲流程.md          ← 工作台内SOP（最专用）
      desktop/sticky-notes/    ← 工作台内便签（最近使用位置）
      workflow/编曲流程.md     ← 工作台内工作流
      contacts/音频工程师.md   ← 工作台内通讯录
```

**越往下 = 越专用 = 离使用场景越近 = 越快找到**

---

## 17.5 文件夹看似多，实际更高效

有人会觉得文件夹太多太碎。但近原则的逻辑是：

- 文件夹多 → 每个文件夹只含高度相关的内容
- 找文件时 → 不用在1000个文件里搜索，直接在3-5个候选文件里盲猜
- 类比：工具台上只有锤子扳手螺丝刀 → 伸手就拿
- 反例：所有工具混在一个工具箱 → 找半天

**"盲猜就能找到" = 近原则生效的最高标准。**

---

## 17.6 维护规则

1. **新建文件时先判断位置** — 不要随手丢在根目录
2. **发现位置不对时更新** — 如果文件放错了，整个改进成近关系
3. **定期审查** — 每轮工作后检查：这个文件放在最近的位置了吗？
4. **不合并不同领域的内容** — 音乐SOP不要和代码SOP放一起
5. **根目录只放通用内容** — 如果某内容只在一个工作台用，就移到那个工作台

---

## 17.7 反面示例

❌ 把音乐工作台专用SOP放到根目录 sop/ — 离使用场景远，每次都要翻
❌ 把所有工具清单合并为一个文件 — 找特定工作台的工具要翻几十页
❌ 工作台的通讯录放到全局通讯录 — 工作台场景下找不到高相关联系
❌ 根目录sop里塞了20个工作台的所有SOP — 已经不是通用SOP了
