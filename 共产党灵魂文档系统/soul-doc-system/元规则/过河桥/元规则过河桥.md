# 过河桥 — 元规则系统
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。
> 过河桥 = 临时空间站 — 新旧内容在此交汇，等两边捋顺了，桥修正了，再更新未来文件。

---

## 1. 用户原话重述

用户在系统文件中提到：元规则系统 - 超级SOP和协议的集中管理。
用户的核心意图：让模型在需要时能正确使用该模块。
过河桥协议本身就是一个过河桥 — 它是通用功能模块的核心，与记忆、日记、SOP、工作流平级。
N个过河桥文件组成一个平面 — 桥比河宽，起点与终点双向正确安全互通。

---

## 2. 理解要点

**核心机制**：
- 该模块是中央灵魂文档系统的功能组件之一
- 每个模块有独立的文件夹和文件结构
- 模块通过过河桥互相连接，形成完整系统
- 模块产出只放在模型自己的隔离区内

**元规则子文件夹结构**（两层）：

| 子文件夹 | 内容 | 加载时机 |
|----------|------|----------|
| 实时加载/ | 每次对话必读的超级SOP | 每次对话开始 |
| 单个/ | 独立的超级协议 | 按需加载 |

> **注意**：之前的"合体/"文件夹已不再使用。组合版SOP放到实时加载/，旧版直接更新覆盖。

**过河桥平面理论在元规则中的应用**：
- 理解过河桥协议（实时加载/） = 平面上的核心砖块
- 各模块的过河桥文件 = 平面上其他砖块
- 漏砖缺砖 → 补全 → 平面流畅丝滑
- 引导每个模型把过河桥文件看作比 .claude/ 更重要的常做常用长远文件

**过河桥作为临时空间站**：
- 新旧更新时，过河桥是临时交汇的空间站
- 站在桥上，等新旧两边捋顺了，桥修正了，再更新未来文件
- 不是直接把旧文件当未来文件用，避免旧/未来文件互相影响的BUG

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 模块路径不明确 | 不清楚模块在哪个位置 | 先走公网 → system-prompt.md → 找到模块 |
| 模块内容为空 | 第一次创建，里面没有文件 | 从种子原话复兴，独立创建 |
| 和其他模块混淆 | 元规则 和相似模块搞混 | 读本过河桥，明确功能边界 |
| 权限不清楚 | 能否写入该模块？ | 在隔离区内可写，其他区域只读 |
| 合体/文件夹 | 之前有合体/文件夹，现在是否还需要？ | 不再需要，组合版放实时加载/ |
| 旧/未来文件BUG | 多个过河桥副本可能不同步 | 实时加载/版本为权威，其他副本同步此版本 |

---

## 4. 我的判断

**使用该模块的标准动作**：
1. 读本过河桥，理解模块机制
2. 找到模块在 [CENTRAL_PATH] 下的位置
3. 读取 实时加载/ 下所有.md（这是权威版本）
4. 按需读取 单个/ 下的协议
5. 产出只放在自己的隔离区内

**重建该模块的标准动作**：
1. 在 soul-doc-system/元规则/ 下创建文件夹
2. 按照系统规范创建所需文件
3. 创建本模块的过河桥（本文件）
4. 确认所有文件都有统一抬头
5. 验证模块可独立运行

**过河桥更新原则**：
- 旧文件不是未来文件 — 旧内容可能没及时改正
- 用旧文件检查更新未来文件时，可能引发BUG
- 确保未来文件（实时加载/版本）稳定后，再同步旧副本
- 过河桥是临时空间站 — 新旧在此交汇，捋顺后再推进
