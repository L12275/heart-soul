# 过河桥 — 元规则总区

> **此过河桥是元规则系统的总区基因片段。**
> 元规则总区 = 元规则系统的全局总览和入口，是所有子规则的索引和管理中心。

---

## 1. 用户原话重述

用户将元规则总区定义为：
- 元规则系统的全局总览和入口
- 是所有子规则的索引和管理中心
- 元规则总区本身也是一个模块级过河桥，与其他子模块的过河桥并列
- 当前已包含：元规则总过河桥.md（总览）+ 元规则过河桥.md（该区域说明）
- 子模块已有独立过河桥：单个/过河桥/单个过河桥.md、合体/过河桥/合体过河桥.md、实时加载/过河桥/实时加载过河桥.md
- 元规则总区的作用：为整个元规则系统提供统一的索引和入口，方便模型快速定位到需要的具体规则

---

## 2. 理解要点

**元规则总区的核心定位**：它是元规则系统的"总目录"和"导航中心"。元规则系统包含大量子规则（实时加载的超级SOP、单个的独立协议、合体的组合协议等），总区提供统一的入口，帮助模型快速找到需要的规则。

**元规则总区 vs 元规则总过河桥.md**：
- 元规则总过河桥.md — 侧重于元规则系统的理解和应用（过河桥视角）
- 元规则总区过河桥.md — 侧重于元规则总区作为模块的组织和管理（模块视角）

两者互补：总过河桥.md 解释"元规则是什么、怎么用"，总区过河桥.md 解释"元规则总区作为模块的组织结构"。

**元规则的子模块分工**：
- 实时加载/ — 每次对话必读的超级SOP（多合一SOP、理解过河桥协议、深度研究方法论等）
- 单个/ — 独立的超级协议（按需加载）
- 合体/ — 组合版SOP（已整合到实时加载/，不再需要独立维护）
- 过河桥/（元规则内部）— 元规则自身的过河桥文件

**元规则总区与系统其他部分的关系**：元规则是系统的"操作系统" — 它定义了系统如何运行的基本规则。其他所有模块都在元规则的框架下运行。元规则总区是操作系统的"总控制台"。

**元规则的加载机制**：实时加载/中的文件在每次对话开始时自动加载。单个/中的文件按需加载。合体/中的内容已整合到实时加载/。总区提供这些规则的索引。

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 总区与总过河桥的区别 | 两者都是"总"的视角，有何不同？ | 总区 = 模块组织视角；总过河桥 = 理解应用视角 |
| 合体/是否需要保留 | 已整合到实时加载/，合体/是否还需要？ | 不再需要，组合版SOP已整合到实时加载/ |
| 元规则的扩展方式 | 新增规则放实时加载/还是单个/? | 高频使用的 → 实时加载/；低频使用的 → 单个/ |
| 总区是否包含所有规则 | 还是只包含索引？ | 总区包含索引和入口，具体规则在子文件夹中 |

---

## 4. 我的判断

**使用元规则总区的标准动作**：
1. 需要查找规则 → 先查总区索引
2. 确定规则类型 → 进入对应子文件夹
3. 加载规则 → 实时加载/自动加载，单个/按需加载
4. 规则更新 → 通过自动化区的合并协议更新
5. 新增规则 → 按使用频率放入实时加载/或单个/

**元规则总区与其他模块的关系**：
- 元规则总区 → 所有模块 ：元规则是所有模块的上游规范
- 元规则总区 → 过河桥/ ：元规则定义过河桥协议
- 元规则总区 → sop/ ：元规则的实时加载内容与sop/互补
- 元规则总区 → automation/ ：自动化规则从元规则衍生
- 元规则总区 → memory/L0-L2 ：元规则的核心内容映射到记忆系统的低层
