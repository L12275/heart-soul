# 过河桥 — 元规则系统总览

> **此过河桥是元规则系统的总览。**
> 元规则 = 所有超级SOP和协议的集中管理。
> 子文件夹：实时加载/ + 单个/
> 实时加载 = 最强脑师多合一SOP + 理解过河桥协议 + 深度研究方法论 + 顶级机会做法SOP + 超级信息SOP
> 过河桥 = 元规则的核心 — 它是通用功能模块，与记忆、日记、SOP、工作流平级。

---

## 1. 用户原话重述

用户将元规则定义为：
- 所有超级SOP和协议的完整版存储目录
- 每次对话开始读取 实时加载/ 下所有.md文件
- 新增规则 = 把文件扔进对应子文件夹，模型下次自动读到
- 过河桥协议本身就是一个过河桥 — 每次收到用户信息，先出过河桥，再行动
- N个过河桥文件组成一个平面 — 桥比河宽，起点与终点双向正确安全互通

---

## 2. 理解要点

**元规则系统的两层结构**：

| 子文件夹 | 内容 | 加载时机 |
|----------|------|----------|
| 实时加载/ | 每次对话必读的超级SOP（多合一等） | 每次对话开始 |
| 单个/ | 独立的超级协议（理解过河桥协议等） | 按需加载 |

**过河桥在元规则中的角色**：
- 过河桥协议（实时加载/）= 每次必读的元规则
- 各模块的过河桥 = 元规则的延伸应用
- 过河桥 = 所有模块的上游源头 — SOP从过河桥归纳，工作流从过河桥演化
- 记忆和日记围绕过河桥展开

**过河桥平面理论**：
- 大量过河桥拼在一起 = 大平面
- 引导每个模型把过河桥文件看作比 .claude/ 更重要的常做常用长远文件
- 在文件系统里来去都在路过，过河桥文件哪都有
- 仿佛在地板砖铺成的大平面上思考，稳定发挥
- 漏砖、缺砖 → 补全 → 平面流畅丝滑

**过河桥作为临时空间站**：
- 新旧更新时，过河桥是临时交汇的空间站
- 站在桥上，等新旧两边捋顺了，桥修正了，再更新未来文件
- 旧文件不是未来文件 — 旧内容可能没及时改正
- 用旧文件检查更新未来文件时，可能引发BUG
- 过河桥 = 临时空间站，能站在桥上等新旧两边捋顺

---

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 元规则和sop的区别 | 元规则是超级SOP，sop是普通SOP | 元规则 = 最高层；sop = 执行层 |
| 实时加载的内容范围 | 哪些文件应该在实时加载/ | 多合一SOP + 理解过河桥协议 + 深度研究方法论 + 顶级机会做法SOP + 超级信息SOP |
| 旧/未来文件冲突 | 多个过河桥副本可能不同步 | 实时加载/版本为权威，其他副本同步此版本 |
| 过河桥和.claude/的关系 | 两者都要加载吗？ | 过河桥比 .claude/ 更重要 — 可放多个，每块都是独立的设计图 |
| 合体/文件夹 | 之前有合体/文件夹，现在是否还需要？ | 不再需要，组合版SOP放实时加载/ |

---

## 4. 我的判断

**使用元规则的标准动作**：
1. 对话开始 → 读取 实时加载/ 下所有.md
2. 按需读取 单个/ 下的协议
3. 各模块的过河桥 = 工作区旁边的路标
4. 产出只放在自己的隔离区内

**过河桥作为临时空间站的原则**：
1. 收到用户信息 → 先写过河桥（临时空间站）
2. 过河桥 = 新旧信息交汇点
3. 站在桥上等两边捋顺 → 桥修正 → 再更新未来文件
4. 不直接把旧文件当未来文件用 — 避免旧/未来文件BUG
