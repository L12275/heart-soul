合并协议（Merge Protocol）

> 此协议定义了如何将sections中的规则安全合并到master-soul.md。
> 目的是确保master-soul.md的质量和一致性。

---

## 合并触发条件

满足以下任一条件时触发合并：

1. **时间触发**：每周至少合并一次
2. **数量触发**：sections中积累5条"已验证"状态的新规则
3. **用户触发**：用户明确要求合并
4. **质量触发**：某个section的规则达到3条已验证时，该section可单独合并

## 合并前安全检查

**合并前必须逐项检查：**

### 1. 冲突检查
- [ ] 新规则是否与master-soul.md中的现有规则冲突？
- [ ] 新规则是否与master-soul.md中的其他新规则冲突？
- 冲突定义：两条规则对同一场景给出相反要求

### 2. 重复检查
- [ ] 新规则是否已在master-soul.md中有等效内容？
- [ ] 如果有等效内容，是更新旧规则还是保留两者？

### 3. 表述检查
- [ ] 新规则是否具体（含触发条件+执行动作+检查方式）？
- [ ] 新规则是否不含模糊词（"可能"、"大概"、"一些"）？
- [ ] 新规则是否不会误导AI（不是具体任务内容，是通用规范）？

### 4. 结构检查
- [ ] 合并后文档结构是否完整？
- [ ] 合并后章节顺序是否合理？
- [ ] 合并后是否有遗漏的章节？

## 合并执行流程

1. **备份**：复制当前master-soul.md为history/版本备份
2. **读取**：读所有sections中的"已验证"规则
3. **整合**：按章节编号顺序整合到master-soul.md
4. **检查**：运行安全检查清单
5. **记录**：更新history/changelog.md
6. **验证**：读一遍合并后的master-soul.md确认无误

## 合并后验证

合并完成后：
1. 读一遍完整的master-soul.md确认可读
2. 确认版本号已更新
3. 确认changelog已记录
4. 确认所有"已验证"规则已在master中

## 回滚条件

以下情况必须回滚到上一个稳定版本：
1. 合并后出现规则冲突
2. 合并后出现重复规则
3. 合并后出现模糊/误导性表述
4. 合并后AI行为异常（用户反馈）

回滚方法：复制history/中的上一个版本覆盖master-soul.md
