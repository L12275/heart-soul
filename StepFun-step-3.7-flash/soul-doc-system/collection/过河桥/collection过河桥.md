过河桥 — collection

> **此过河桥是 采集系统 的基因片段。**
> 采集系统 = 从外部来源采集信息，按类型和保留期限分文件夹存放。
> 是中央灵魂文档系统的通用功能组件之一，参与模块同步维护规则。
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

---

## 1. 用户原话重述

用户在系统文件中描述了采集系统：爬取、临时采集的数据、资料、资源。分文件夹存放，避免堆积混杂。在模块同步维护规则表中，collection 位于通用区 soul-doc-system/collection/，同步到工作台模版和账号空间模版。

用户的核心意图：让模型能够系统性地从外部来源采集数据，按类型分类存储，设置保留期限自动清理，并建立从临时采集到正式知识的提炼流转机制。

---

## 2. 理解要点

**核心机制**：
- 采集系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/collection/`，同步到新建工作台模版和新建账号空间模版
- 采集系统 = 外部数据的临时缓存层，有严格的保留期限和清理机制

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 采集 collection | soul-doc-system/collection/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/collection/
- 隔离区内：collection/
- 模板位置：新建工作台模版/collection/、新建账号空间模版/collection/

**采集系统的分类和保留期限（来自代码工作台采集模板）**：
| 类型 | 存放位置 | 保留期限 | 说明 |
|------|---------|---------|------|
| API响应 | collection/api/ | 7天 | 临时API调用结果 |
| 页面快照 | collection/pages/ | 30天 | 网页快照 |
| 数据文件 | collection/data/ | 按需 | CSV/JSON等原始数据 |
| 临时脚本 | collection/scripts/ | 7天 | 一次性测试脚本 |

**采集系统的核心规则**（来自代码工作台采集模板）**：
1. 采集数据必须有来源标注
2. 超过保留期限自动清理
3. 有用数据必须提炼后移入正式目录（wiki/ 或 product/）

**采集系统的结构（推断）**：
```
collection/
├── README.md           ← 说明书
├── api/                ← API响应（7天）
│   └── [时间戳]_[来源].json
├── pages/              ← 页面快照（30天）
│   └── [时间戳]_[URL].html
├── data/               ← 数据文件（按需）
│   └── [名称].csv/json
├── scripts/            ← 临时脚本（7天）
│   └── [名称].py/js
└── 过河桥/              ← 本文件
```

**数据流转方向**：
```
外部来源
    ↓ 采集
collection/（临时缓存，有保留期限）
    ↓ 提炼验证
wiki/（永久知识）或 product/（产出物）或 experience/（经验）
    ↓ 进入灵魂文档
master-soul.md / sections/
```

**与 desktop/ 的区别**：
- collection/ = 结构化采集的数据（按类型分类、有保留期限、有来源标注）
- desktop/ = 临时堆放区（多类型混合、无固定期限、无分类要求）

**与 wiki/ 的区别**：
- collection/ = 临时采集的原始数据（未验证、有保留期限）
- wiki/ = 经过验证的永久知识（已验证、无期限）

**与九环架构的关系**：
- 采集系统服务于所有环区
- 在 环④ 生产空间中心工作台中，采集是研究类工作台的重要输入
- 在 环③ 主灵魂中心中，采集的内容可以丰富灵魂文档

**与网络搜索的关系**：
- 网络搜索（Multi_Search_Engine skill）是采集系统的主要输入源
- 搜索结果的原始数据 → collection/ 缓存
- 提炼后的知识 → wiki/ 或 sections/

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 采集和偷数据的区别 | 采集系统是否涉及爬虫/偷数据？ | 不涉及。采集系统只采集模型自己需要的、公开可用的数据，不涉及侵权 |
| 保留期限的强制性问题 | 超过保留期限是否自动清理？ | 是。超过保留期限的数据必须清理，不能长期堆积 |
| 数据文件 vs 临时脚本 | data/ 和 scripts/ 的区别是什么？ | data/ = 原始数据文件（CSV/JSON等）；scripts/ = 一次性测试脚本（运行一次即可） |
| 有用数据的判断 | 什么算"有用"？ | 对模型灵魂文档系统有贡献的数据 = 有用。判断标准：是否丰富了知识、是否验证了假设、是否支撑了决策 |
| 采集数据的隐私性 | 采集的数据是否涉及隐私？ | 采集的数据应该是公开可用的。如果涉及隐私，不应采集 |

## 4. 我的判断

**使用采集系统的标准动作**：
1. 读本过河桥，理解分类体系和保留期限
2. 从外部来源（网络搜索、API调用、用户资料等）获取数据
3. 按类型分类放入 collection/ 对应子目录
4. 每条采集数据必须有来源标注
5. 超过保留期限的数据自动清理
6. 有用数据提炼后移入 wiki/ 或 product/

**采集数据时的标准动作**：
1. 确定数据类型（API响应/页面快照/数据文件/临时脚本）
2. 按类型放入对应子目录
3. 写入来源标注（URL、API端点、文件路径、获取时间）
4. 设置提醒：在保留期限到期前评估是否有用

**提炼采集数据的标准动作**：
1. 定期检查 collection/ 中的数据
2. 评估数据是否有用（是否丰富了知识、验证了假设、支撑了决策）
3. 有用的数据 → 提炼后移入 wiki/（知识）或 product/（产出物）或 experience/（经验）
4. 无用的数据 → 到期后自动清理

**重建采集模块的标准动作**：
1. 在 soul-doc-system/collection/ 下确认四个分类子目录存在
2. 按模块同步维护规则，从通用区复制到各模板位置
3. 确认保留期限规则已记录
4. 验证来源标注机制正常
5. 验证模块可独立运行
