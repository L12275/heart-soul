> **此过河桥是 生产空间中心工作台（第④环）的基因片段。**
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。

## 1. 用户原话重述

用户在 system-prompt.md 的九环架构章节中定义：production_space/ 是第④环"生产空间中心工作台"，由产出驱动运转。

用户在 README.md 中说明：最大生产单位，包含多种工作台，可组合联动。不同类型的工作活动使用不同的工作台（代码/文章/研究/科学/视频/音乐/项目/通用默认）。

用户的完整意图：
- production_space/ 作为第④环，是产出驱动型工作空间
- 内部按工作领域拆分为独立工作台
- 每个工作台拥有完整的模块拼搭（desktop/sticky-notes/workflow/contacts/bookmarks 等）
- product/ 目录作为统一产出物管理入口
- 模块有"钉住策略"——通用模块钉在地平面，专用模块钉在工作台内
- 双层结构（地平面 vs 工作台）通过双写规则衔接

## 2. 理解要点

**核心定位**：
- production_space/ 是九环架构中的第④环：生产空间中心工作台
- 驱动方式：产出驱动（不是静默读入，不是管理驱动，是"干活"驱动）
- 产出物统一归口到 product/ 目录管理
- 每个工作台 = 一个独立的生产环境，内部模块自包含

**八种工作台类型**：

| 工作台 | 适用场景 |
|--------|---------|
| 代码工作台 | 编程、开发、调试、代码审查 |
| 文章工作台 | 写作、文档、博客、技术文章 |
| 研究工作台 | 调研、资料收集、文献综述 |
| 科学工作台 | 实验设计、数据分析、科学计算 |
| 视频工作台 | 视频脚本、剪辑方案、发布计划 |
| 音乐工作台 | 作曲、编曲、音频制作 |
| 项目工作台 | 多阶段任务、里程碑管理、复杂项目 |
| 通用默认工作台 | 无法归入上述类型的任何产出活动 |

**每个工作台的模块构成**（模块拼搭）：

```
[某工作台]/
├── desktop/        ← 临时桌面（空白工作台，按需创建 桌面1/2/N 子文件夹）
├── sticky-notes/   ← 便签（临时想法、碎片笔记）
├── workflow/       ← 工作流（流程定义、步骤编排）
├── contacts/       ← 通讯录（协作方、相关人）
├── bookmarks/      ← 书签（参考资料、参考链接）
├── diary/          ← 日记（时间轴实时纪实：年/月/日/日期.md）
├── memory/         ← 记忆（验证后的正确记忆：context/L0-L4 + daily/天级快照）
├── sop/            ← SOP（该工作台专用的操作规范）
├── product/        ← 产出物（最终/阶段性产出，统一管理）
├── temp/           ← 临时文件（短期使用后清理）
├── workflow/       ← 工作流定义
└── 过河桥/         ← 该工作台自身的过河桥文件
```

**双层结构（地平面 vs 工作台）**：

- **地平面级**（soul-doc-system 根目录）= 工厂公共设施：大门、走廊、电梯、仓库、会议室
  - 所有工作台共享的通用资源
  - 不特定于某个工作台的内容写到这里
- **工作台级**（production_space/某工作台）= 工厂里的专用车间
  - 每个车间有独立的桌面、工作流、通讯录、收藏夹、便签、日记、记忆
  - 只记录与这个车间高相关的内容

**双写规则**：
- 通用记录（不特定于某个工作台）→ 只写到地平面级
- 专用记录（与某工作台高相关）→ 只写到该工作台内
- 双写（既通用又专用）→ 同时写到地平面和工作台

**product/ 统一产出管理**：
- 所有工作台的最终/阶段性产出物统一进入 product/ 目录
- product/ 内记录：文件名、来源、所属工作、文件出现时间、完成日期、版本、后续状态
- 超过 60 天未使用（以文件出现时间为起计）→ 提示确认是否清理
- 清理规则详见 product/清理规则.md

**模块钉住策略（Module Pinning Strategy）**：

- **钉在地平面**（soul-doc-system 根目录下）的模块 = 通用模块，所有工作台共享权威源
  - memory/、diary/、sop/、workflow/、product/、skills/、tools/、bookmarks/、contacts/、temp/、sticky-notes/、experience/、discovery/、深度研究/、collection/、automation/、desktop/
- **钉在工作台内**的模块 = 专用模块，每个工作台独立维护自己的工作副本
  - 工作台内的 diary/、memory/、workflow/、sticky-notes/、contacts/、bookmarks/、desktop/、product/、sop/、temp/
- **同步机制**：通用区模块更新后，必须同步复制到所有模板位置（通用/代码工作台、新建工作台模版等）
- **平台空间同步**：仅 memory/ 和 diary/ 同步到 space/平台/通用-待细分/{模块}/

**日记 vs 记忆**：

| 类型 | 内容 | 准确性 | 格式 | 位置 |
|------|------|--------|------|------|
| 日记（diary/） | 当天实时事件过程 | 纪实，可能不完整 | 时间轴：年/月/日/日期.md | 地平面 + 工作台 |
| 记忆（memory/） | 验证确认后的正确记忆 | 已验证 | context/L0-L4 + daily/天级快照 | 地平面 + 工作台 |

**目录结构**：
```
production_space/
├── README.md
└── 通用/
    ├── README.md
    ├── 代码工作台/
    ├── 文章工作台/
    ├── 研究工作台/
    ├── 科学工作台/
    ├── 视频工作台/
    ├── 音乐工作台/
    ├── 项目工作台/
    ├── 通用默认工作台/
    └── 新建工作台模版/
```

**近原则**：
- 不同类型结构相同，只是标签不同 → 统一放在 通用/ 下
- 实际使用时默认用"工作室"，需要区分时再新建类型
- 工作台内专用内容放在工作台内，不污染根目录通用文件夹

**与九环架构的关系**：
- 第④环，驱动方式 = 产出驱动
- 每次读写记忆/日记时，生产空间内容 → 写对应工作台的 memory/ 和 diary/
- 产出物统一进 production_space/ 下的 product/
- 产出驱动的产出 → 同时沉淀为快箭，进入快箭织网层

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 工作台数量 | 当前有8种，但用户说"可组合联动"，是否允许自定义工作台类型？ | 8种为预设，新建工作台模版/ 提供扩展入口，需要时按模版创建新类型 |
| 通用默认工作台和 通用/ 的关系 | production_space/通用/README.md 是总说明，通用默认工作台/ 是一个具体工作台实例 | 通用/README.md = 总规范说明；通用默认工作台/ = 实际可用的默认工作台 |
| 双写范围 | 哪些内容算"既通用又专用"？ | 核心判断：内容是否涉及特定工作台的专业领域？涉及 = 专用；不涉及 = 通用；两者混合 = 双写 |
| product/ 位置 | product/ 是每个工作台独立一个，还是一个统一的？ | 每个工作台有自己的 product/，产出物在各自工作台内管理；跨工作台汇总在地平面 product/ |
| 模块同步频率 | 通用区模块更新后多久同步到模板？ | 每次更新通用区模块后，立即同步到所有模板位置（即时同步） |
| 工作台切换 | 同一产出活动是否可以在多个工作台间切换？ | 同一产出活动固定在一个工作台内完成；跨领域活动按主领域选择工作台，其他领域内容作为引用 |
| desktop/ 和 桌面n 的区别 | desktop/ 是固定模块，桌面n 是按需创建的子文件夹 | desktop/ 模块 = 框架（含 README.md）；桌面n = 实际创建的临时堆放区（子文件夹） |

## 4. 我的判断

**进入 production_space 的标准动作**：
1. 确认当前产出活动属于哪种工作台类型
2. 进入 production_space/通用/{对应工作台}/
3. 在工作台内使用对应模块（desktop/sticky-notes/workflow/contacts/bookmarks/diary/memory/sop/product/temp）
4. 产出物归入该工作台的 product/ 目录
5. 日记实时记录到 diary/，记忆在验证后写入 memory/

**创建新工作台的标准动作**：
1. 复制 新建工作台模版/ 作为模板
2. 按需调整模块配置（删减或添加模块）
3. 在工作台内创建各模块文件夹
4. 创建工作台专属过河桥
5. 在 production_space/通用/ 下注册新工作台文件夹

**模块同步的标准动作**：
1. 更新通用区模块内容
2. 将更新后的模块文件夹完整复制
3. 覆盖到所有工作台模板中的同名模块
4. 各工作台 README.md 单独维护

**重建 production_space 模块的标准动作**：
1. 在 soul-doc-system/ 下创建 production_space/ 文件夹
2. 在内部创建 通用/ 文件夹
3. 创建8种工作台文件夹 + 新建工作台模版/
4. 每个工作台内按模块拼搭创建标准模块集
5. 创建 product/ 目录并配置清理规则
6. 创建本模块的过河桥（本文件）
7. 确认所有文件都有统一橄榄枝抬头
8. 验证各工作台可独立运行

**产出管理的标准动作**：
1. 产出物放入对应工作台的 product/
2. 记录文件名、来源、所属工作、文件出现时间、完成日期、版本、后续状态
3. 超过60天未使用 → 提示确认
4. 定期执行 product/清理规则
