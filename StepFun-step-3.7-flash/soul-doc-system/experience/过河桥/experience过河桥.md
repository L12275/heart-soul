过河桥 — experience

> **此过河桥是 经验系统 的基因片段。**
> 经验系统 = 三层可信度分类的经验仓库（proven / tested / suspected-hallucination）。
> 是中央灵魂文档系统的通用功能组件之一，参与模块同步维护规则。
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

---

## 1. 用户原话重述

用户在系统文件中描述了经验系统：经过验证和未验证的经验的三层分类存储。在模块同步维护规则表中，experience 位于通用区 soul-doc-system/experience/，同步到工作台模版和账号空间模版。经验库分为三层：proven（已证明，最高可信度）、tested（已验证）、suspected-hallucination（疑似幻觉，最低可信度）。

用户的核心意图：让模型能够分层管理自己的经验，区分已经过事实验证的高可信度经验和尚未验证的疑似经验，避免将未验证的经验当作事实使用。

---

## 2. 理解要点

**核心机制**：
- 经验系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/experience/`，同步到新建工作台模版和新建账号空间模版
- 经验系统 = 三层可信度分类的经验仓库，是模型个人成长的记录

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 经验 experience | soul-doc-system/experience/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/experience/
- 隔离区内：experience/
- 模板位置：新建工作台模版/experience/、新建账号空间模版/experience/

**三层分类体系（来自经验库目录结构）**：

| 层级 | 目录 | 可信度 | 说明 |
|------|------|--------|------|
| 最高 | proven/ | ✅ 已证明 | 经过事实验证的经验，可信度最高 |
| 中等 | tested/ | 已验证 | 经过测试但尚未完全证明的经验 |
| 最低 | suspected-hallucination/ | 疑似幻觉 | 可能不准确的经验，需要进一步验证 |
| 参考 | Reference/ | 参考 | 外部参考文档和资料 |

**经验条目的格式要求（来自 proven/README.md）**：
每条经验必须包含：
- 时间/场景
- 具体过程
- 来源文件路径
- 形成原因
- 思想/信念背景

**经验系统的结构（推断）**：
```
experience/
├── README.md                ← 说明书
├── proven/                  ← 已证明经验（最高可信度）
│   ├── README.md
│   └── [经验条目].md
├── tested/                  ← 已验证经验
│   ├── README.md
│   └── [经验条目].md
├── suspected-hallucination/ ← 疑似幻觉（最低可信度）
│   ├── README.md
│   └── [经验条目].md
├── Reference/               ← 外部参考
│   ├── README.md
│   └── [参考文档].md
└── 过河桥/                   ← 本文件
```

**经验条目的完整格式**：
```markdown
## [经验名称]
**可信度层级：** proven / tested / suspected-hallucination
**时间/场景：** YYYY-MM-DD [场景描述]
**具体过程：** [详细描述发生了什么]
**来源文件路径：** [产生该经验的对话/任务/文件路径]
**形成原因：** [为什么形成了这条经验]
**思想/信念背景：** [当时的思考方式或信念]
**后续验证：** [是否经过后续验证，验证结果]
```

**经验的生命周期**：
1. **疑似幻觉** → 在实际应用中检验
2. **检验中** → 如果有效 → 升级到 tested/；如果无效 → 保持或删除
3. **tested** → 多次验证有效 → 升级到 proven/
4. **proven** → 最高可信度，可以直接用于决策和行动

**与 discovery/ 的关系**：
- discovery/ = 机会/启发/策略/点子（尚未验证的想法）
- experience/ = 经过验证的经验（已证明/已验证/疑似）
- 发现经过验证后升级为经验

**与 wiki/ 的关系**：
- wiki/ = 通用知识库（结构化、来源标注）
- experience/ = 个人经验库（三层可信度、个人化）
- proven/ 中的高可信度经验可以提炼为 wiki/ 条目

**与记忆系统的关系**：
- memory/ = 短期/中期记忆（GA L0-L4，按时间衰减）
- experience/ = 长期经验（经过验证，不衰减）
- 记忆中的有用内容经过验证后升级为经验

**与九环架构的关系**：
- 经验系统服务于所有环区
- 在 环③ 主灵魂中心中，经验是灵魂文档的重要组成部分
- 在 环④ 生产空间中心工作台中，经验指导工作决策
- 在 环⑥ 子角色中心中，每个子角色有自己的经验积累

**在共产中心的可共享性**：
- proven/ 中的经验可以在共产中心公告板登记（类型：【经验沉淀】）
- suspected-hallucination/ 中的内容不在共产中心共享
- 经验分享 = 白盒优选的重要机制

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 经验和知识的区别 | 经验是否等于知识？ | 不等同。经验 = 模型个人在实践中获得的认知；知识 = 经过验证的通用信息。经验更个人化、更具体 |
| 三层可信度的判定标准 | 什么算"已证明"？什么算"已验证"？ | proven = 经过多次独立场景验证、可重复使用；tested = 至少经过一次实际验证；suspected-hallucination = 未经验证或验证失败的 |
| suspected-hallucination 的保留价值 | 疑似幻觉的经验是否应该删除？ | 不应该删除。保留疑似幻觉经验是为了追踪模型的认知偏差，避免重复犯同样的错误 |
| 经验是否可以共享 | 个人经验可以在共产中心共享吗？ | proven/ 中的经验可以共享（【经验沉淀】类型）。tested/ 和 suspected-hallucination/ 不在共产中心共享 |
| 经验和 SOP 的关系 | 经验是否就是 SOP？ | 不等同。经验 = 认知层面的收获（"我知道了什么"）；SOP = 操作层面的流程（"我怎么做"）。SOP 可以从经验中归纳 |
| 经验条目的颗粒度 | 一条经验应该写多详细？ | 足够详细以便后续验证和引用。必须包含时间/场景、过程、来源、形成原因 |

## 4. 我的判断

**使用经验系统的标准动作**：
1. 读本过河桥，理解三层可信度分类和条目格式
2. 在 experience/ 下找到对应的可信度层级目录
3. 新产生的经验 → 先放入 suspected-hallucination/（未验证）
4. 在实际应用中检验 → 有效 → 升级到 tested/；无效 → 保持或删除
5. tested/ 中多次验证有效的 → 升级到 proven/
6. 在定时任务期间回顾经验，检查是否需要升级或修正

**记录新经验的标准动作**：
1. 执行任务时产生新的认知/收获
2. 按格式要求写入经验条目（名称、可信度层级、时间/场景、过程、来源、形成原因、思想背景）
3. 初始放入 suspected-hallucination/（未验证）
4. 在后续应用中跟踪该经验的有效性

**升级经验层级的标准动作**：
1. 定期审查 suspected-hallucination/ 中的经验
2. 在实际应用中验证经验的有效性
3. 有效 → 升级到 tested/，记录验证结果
4. tested/ 中多次验证有效的 → 升级到 proven/，标记为已验证可重复
5. 无效 → 标记为已失效，记录失效原因

**共享经验的标准动作**（proven 层级）：
1. 确认经验在 proven/ 中
2. 在共产中心公告板登记：【经验沉淀】类型
3. 登记格式：[秒级时间戳] [模型名称] 文件：相对路径 变更：摘要 类型：【经验沉淀】

**重建经验模块的标准动作**：
1. 在 soul-doc-system/experience/ 下确认三层分类子目录和 Reference/ 存在
2. 按模块同步维护规则，从通用区复制到各模板位置
3. 确认每层的 README.md 和条目格式说明存在
4. 验证经验条目格式正确（包含所有必填字段）
5. 验证模块可独立运行
