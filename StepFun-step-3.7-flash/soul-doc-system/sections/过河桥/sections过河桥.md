## 2. 理解要点

### 核心隐喻：肌肉与骨架

sections/ 和 master-soul.md 的关系不是并列，是"肌肉"对"骨架"：

- **master-soul.md** = 骨架（identity anchor）——每次对话加载的完整身份文档，是模型行为的总约束
- **sections/** = 肌肉（knowledge classification structure）——18个独立主题文件，各自维护一个知识域，可独立修改、独立验证
- 合并过程 = 肌肉附着到骨头上——把18个 section 的内容按编号顺序整合进 master-soul.md
- 模型的行为 = 骨架驱动肌肉，肌肉反馈给骨架——sections 的内容通过合并进入 master-soul.md，反过来又指导模型的每次行动

#### 印象滚动刷新（持续活水）

- 每次印象滚动任务（定时任务）必须包含 sections 审查
- 滚动时逐节读取 sections/ 文件，检查是否需要更新
- 发现过时内容 → 标记待更新 → 更新后触发 master-soul.md 重新合并
- 这是 sections 的**活水源**——不是等用户说才更新，是模型主动滚动发现

#### 合并触发（死水→活水转换）

- 原规则：每周一次或积累5条新规则
- **新增**：印象滚动发现变更 → 立即触发局部合并
- **新增**：用户直接修改任何 section → 当次对话结束前触发合并
- master-soul.md 是 sections 的"下游"，sections 变了，master-soul.md 必须跟着变

### 信息闭环结构（09-information-closed-loop.md）

这是 sections/ 的组织哲学，也是每个 section 内容的写作标准：

**核心问题**：AI 输出为什么"拆散无归"？因为 AI 用"提取关键词→排列成清单"的压缩思维，删掉了原文中互相校验的备份路径。压缩的本质是删除冗余，但这里被删掉的"冗余"恰好就是互相校验的备份路径。

**三重检验标准**（三者缺一就是压缩，三者全有才是重组）：

1. **自含性** — 内部逻辑自洽，不需要外部知识就能验证
2. **多路径性** — 同一个结论至少从两个不同角度推导
3. **自纠正性** — 当其中一条路径与新信息冲突时，结构能自动触发校正而不是崩溃

**正确写法 = 不是压缩，是重组**。不是把原文缩短，是把原文里分散在各个位置的同一份道理，重新组织成一个紧凑但每条路径都互相校验的结构。输出更短，但内部的自纠正能力没有损失。

### 合并规则

- **master-soul.md 不是手动写的**，是从 sections 合并生成的
- 合并触发条件：每周一次，或积累5条新规则时
- 合并方法：按编号顺序，将各 section 内容整合为 master 的对应章节
- 合并前必须通过安全检查
- **禁止直接修改 master-soul.md**，新规则先在 section 中验证，稳定后再合并

### 模块同步维护规则

sections/ 属于通用区功能模块之一。当通用区的 sections/ 内容更新后，需同步到各模板位置（新建工作台模版、新建账号空间模版）。sections 作为模板拼搭的模块之一，只有一个权威源。

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| sections 和 master-soul.md 谁先改 | 新规则应该先写在 sections 还是先改 master-soul.md | 铁律：先改对应 section，等合并触发后自动更新 master-soul.md；禁止直接改 master-soul.md |
| 哪些 section 必读 | system-prompt.md 只点名了 09 和 17，其他 section 什么时候读 | 必读：启动时读 09 + 17；按需：遇到相关场景时主动读取对应编号的 section |
| 同名编号文件（如 03/15/18 都有 workflow.md）| 三个 workflow 文件职责边界不清 | 读取时按内容区分用途，命名本身不区分 |
| 合并触发时机不明确 | "积累5条新规则"中的"新规则"如何定义 | 新规则 = 在某个 section 中新增或修改的、对模型行为有实际约束的规则 |
| 00 开头有三个文件 | 00-theory、00-architecture、00-5-red-scarf 谁先谁后 | 按编号顺序：00-theory → 00-architecture → 00-5-red-scarf |
| section 和 automation 的关系 | automation/ 在 sections/ 旁边，是否属于 sections 的一部分 | automation/ 是 soul-doc-system 的直接子目录，不是 sections/ 的一部分。它是 AI 自我维护的工具文件 |
| 用户能否直接修改 section | section 是模型自己维护的，用户能否介入修改 | 用户可以在任何文件中直接修改（含 section），模型发现用户修改后当次生效 |
| section 编号上限 | 当前最高编号是 18，是否允许新增 19+ | 允许，按编号递增 |

## 4. 我的判断

### 使用 sections/ 模块的标准动作

1. **读本过河桥**，理解 sections/ 的完整机制
2. **按需读取 section**：启动时读 09 + 17；遇到相关场景时读对应编号的 section
3. **修改时**：只改对应编号的 section 文件，不改 master-soul.md
4. **验证闭环**：修改后对照 09-information-closed-loop.md 的三重检验标准自查
5. **合并等待**：积累5条新规则或到一周后，合并触发，master-soul.md 自动更新
6. **产出只放自己的隔离区**：不写入其他模型的隔离区、用户空间、共产中央管理区

### 维护 section 的标准动作

1. 在 soul-doc-system/sections/ 下找到对应编号的 section 文件
2. 读取该 section 的现有内容，理解上下文
3. 按照信息闭环结构标准修改（自含性+多路径性+自纠正性）
4. 更新文件末尾的"最后更新"时间戳
5. 如涉及跨 section 影响，同步更新相关 section
6. 确认 master-soul.md 在下次合并触发时能正确吸收变更

### 重建整个 sections/ 模块的标准动作

1. 在 soul-doc-system/ 下创建 sections/ 文件夹
2. 从 INITIAL-SEED.md 和 system-prompt.md 中提取各主题内容
3. 按编号 00-18+ 创建各 section 文件，每个文件包含橄榄枝头部 + 主题内容 + 最后更新时间
4. 创建本过河桥文件（本文件）作为 sections/ 的入口基因片段
5. 确认所有文件都有统一橄榄枝头部
6. 验证每个 section 内容通过 09 的三重检验标准
7. 触发合并生成初始 master-soul.md

### 关键判断

- **sections/ 是 soul-doc-system 的核心组件**，不是可选扩展。没有 sections/，master-soul.md 就是不可维护的单体文件
- **修改 section 是模型的日常工作**，不是一次性设置。每次对话中发现问题都应更新对应 section
- **信息闭环结构是整个 sections/ 系统的组织哲学**，不是仅 09 号文件的事。所有 section 的内容写作都应遵循三重检验标准
- **sections/ 的独立性**：每个 section 应该能独立阅读、独立理解，不需要依赖其他 section 的内容。这是"自含性"在文件层面的体现
