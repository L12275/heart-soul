过河桥 — history

> **此过河桥是 历史系统 的基因片段。**
> 历史系统 = 模型操作的 chronological（时间顺序）记录，以及系统变更的 changelog。
> 是中央灵魂文档系统的通用功能组件之一，参与模块同步维护规则。
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

---

## 1. 用户原话重述

用户在系统文件中描述了历史系统：模型操作的 chronological 记录。在系统提示词的末尾提到"更新记录见：soul-doc-system/history/changelog.md"，说明历史系统负责维护系统变更的时间顺序记录。

用户的核心意图：让模型能够记录所有系统变更、操作历史和时间顺序事件，形成可追溯的变更链，支持系统回滚和历史复盘。

---

## 2. 理解要点

**核心机制**：
- 历史系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/history/`，同步到新建工作台模版和新建账号空间模版
- 历史系统 = 时间顺序的操作记录 + 变更记录（changelog）

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 历史 history | soul-doc-system/history/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/history/
- 隔离区内：history/
- 模板位置：新建工作台模版/history/、新建账号空间模版/history/

**历史系统的内容（推断结构）**：
```
history/
├── README.md           ← 说明书
├── changelog.md        ← 系统变更记录（system-prompt.md 中明确提到此文件）
├── operations/         ← 操作记录（按时间分文件）
├── rollback/           ← 回滚记录
└── 过河桥/              ← 本文件
```

**changelog.md 的定位**（来自 system-prompt.md）：
- system-prompt.md 末尾明确写着"更新记录见：soul-doc-system/history/changelog.md"
- changelog.md 是系统提示词的附属记录文件
- 记录了 soul-doc-system 的所有版本变更

**历史系统的两种记录类型**：
1. **系统变更历史（changelog）**：记录系统文件的结构变更、内容变更、版本升级
2. **模型操作历史**：记录模型在对话中执行的操作、决策、产出物

**与 diary/ 的区别**：
- history/ = 系统变更和操作的 chronological 记录（时间顺序的事件链）
- diary/ = 模型的内部反思和过程记录（按时间轴的日记格式）

**与记忆系统的关系**：
- history 是客观记录（发生了什么、什么时候发生的）
- memory 是主观知识（从历史中提炼的经验和知识）
- history 是 memory 的原材料

**与九环架构的关系**：
- 历史系统服务于所有环区
- 在 环⑨ 中央管理区中，历史系统是中央管理区的核心记录
- 在 环④ 生产空间中心中，历史系统记录产出物的变更历史

**与异地更新的关系**：
- 远程推送时需要附带历史记录
- 远程仓库的 Git 历史本身就是一种分布式历史系统
- history/ 和 Git 历史互为补充：history/ 是语义化的变更记录，Git 是文件级的历史记录

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| history 和 changelog 的关系 | 两者是否一回事？ | changelog = history 的一个子集，专门记录系统版本变更。history = 更广泛的操作记录 |
| history 和 diary 的区别 | 两者都是时间顺序记录 | history = 客观事件记录（系统变更、操作执行）；diary = 主观反思记录（模型的想法、感受、改进） |
| 历史记录的粒度 | 每个操作都记录还是只记录重要变更？ | 重要变更必须记录（系统文件变更、灵魂分数变化、重大决策）。日常操作可以汇总记录 |
| 回滚机制 | 历史系统是否包含回滚功能？ | history/ 记录历史，rollback/（如果有）记录回滚操作。回滚需要手动执行 |
| 和历史过河桥的关系 | 本过河桥本身是否会被历史系统记录？ | 会。过河桥的创建和更新本身就是系统变更，应记录到 changelog.md |

## 4. 我的判断

**使用历史系统的标准动作**：
1. 读本过河桥，理解历史记录的类型和格式
2. 在 history/ 下查看 changelog.md，了解系统变更历史
3. 执行重要操作前 → 先记录当前状态
4. 执行重要操作后 → 记录操作结果和时间
5. 需要回滚时 → 查看历史记录，找到目标状态

**记录变更的标准动作**：
1. 确定变更类型（系统文件变更、灵魂分数变化、重大决策、产出物变更）
2. 按 changelog 格式记录：时间、变更内容、影响范围
3. 如果是版本升级 → 记录版本号、变更摘要、升级原因
4. 如果是操作记录 → 记录操作名称、执行者、结果

**查看历史的标准动作**：
1. 读取 changelog.md 了解近期系统变更
2. 读取 operations/ 下的具体操作记录
3. 分析变更趋势和模式
4. 如果有回滚需求 → 查看 rollback/ 记录

**重建历史模块的标准动作**：
1. 在 soul-doc-system/history/ 下确认文件夹结构
2. 确认 changelog.md 存在且内容完整
3. 按模块同步维护规则，从通用区复制到各模板位置
4. 验证历史记录和 Git 历史的对应关系
5. 验证模块可独立运行
