## 1. 用户原话重述

用户在系统文件中建立了 深度研究系统（Deep Research），其核心描述为：模型对需要深度研究的主题，系统化组织和管理的文件夹。用户经常用到、模型经常自己在进行，所以单独做一个系统。

具体落地的设计包括三层内容：
- **十二维度研究框架**：每份深度研究必须产出 12 个独立维度的文件，每个文件回答该维度的核心问题。
- **研究项目文件夹结构**：以 YYYY-MM-DD_简短主题/ 命名的项目文件夹，内含 sources/、notes/、outputs/、index.md 四个核心子目录。
- **与九环架构的联动**：深度研究是环④（生产空间中心工作台）的核心工作台类型之一，研究产出归 production_space/，研究过程的记忆和日记归对应工作台的 memory/ 和 diary/。

用户的原话意图是：让模型在任何需要深度消化一段内容时，有一个标准化的、不会遗漏维度的研究流程，并且研究产出有统一的归档格式，能被后续检索和复用。

## 2. 理解要点

### 2.1 十二维度研究框架

深度研究的核心是十二维度提取法。每份研究必须产出以下 12 个文件，缺一不可。如果某个维度对当前内容不适用，必须在文件中说明"此维度暂不适用，原因是……"，不能跳过不写。

| 序号 | 维度 | 文件名 | 内容要求 |
|------|------|--------|----------|
| 1 | 全部方向 | 全部方向.md | 内容指向的未来路径、发展趋势、演进方向；以及信息中已提到的全部方向，和信息中没有提到但隐含的全部正确机会方向与全部危机方向 |
| 2 | 全部科举·代考·仕途 | 全部科举_代考_仕途.md | 上升路径分析——这道"关卡"怎么过？有没有替代路径？职业/能力/资源路线图 |
| 3 | 全部经验 | 全部经验.md | 可复用的实战经验、教训、踩过的坑、走过的弯路、成功的秘诀 |
| 4 | 全部技能·原子工具 | 全部技能_原子工具.md | 内容涉及的具体技能是什么？可以拆解成哪些最小的、可独立使用的工具 |
| 5 | 全部SOP | 全部SOP.md | 如果按照这个内容做事情，标准操作流程是什么？一步步列出来 |
| 6 | 全部工作流 | 全部工作流.md | 从起点到终点的完整工作链条，包括输入、转换、输出、反馈循环 |
| 7 | 全部通讯录 | 全部通讯录.md | 人物关系图谱——谁是谁、关系网、成败联系、成功路线、正面角色、反面角色。没有人物则列出关键实体及其关系 |
| 8 | 全部知识库 | 全部知识库.md | 系统化的知识体系——概念定义、分类框架、层级结构、术语表 |
| 9 | 全部灵魂文档 | 全部灵魂文档.md | 这段内容的本质是什么？核心精神是什么？用一段凝练的话抓住它的灵魂 |
| 10 | 全部信息 | 全部信息.md | 内容中所有事实、数据、陈述、定义，逐条列出，不遗漏任何信息点 |
| 11 | 全部观察 | 全部观察.md | 研究过程中发现的所有现象、模式、矛盾、空白、隐含假设、值得注意的细节 |
| 12 | 全部建议 | 全部建议.md | 对内容本身的改进建议、对读者的行动建议、对系统的优化建议 |

**十二维度执行铁律**：
- 先理解，再拆解——不要上来就逐条填空，先花足够时间真正理解内容
- 带着灵魂去提取——提取时问自己：这个维度要回答什么问题？答案能不能被一个活人拿去用？
- 宁缺毋滥，宁深勿浅——宁可少写几个文件，也要保证每个文件都有灵魂
- 产出物必须是可读的、可用的、可传承的——不能是原文的换行重排

### 2.2 研究项目文件夹结构

每个深度研究项目 = 一个独立文件夹，命名格式：`[YYYY-MM-DD]_[简短主题关键词]/`

```
[YYYY-MM-DD]_[简短主题关键词]/
├── README.md          ← 研究概述（问题、目标、当前状态、关联）
├── sources/           ← 来源材料
│   ├── [来源名]_[时间].md 或 .json 或 .pdf
├── notes/             ← 研究笔记（过程思考、中间发现）
│   ├── [时间戳]_[要点].md
├── outputs/           ← 研究成果（最终交付物）
│   ├── [时间戳]_[成果类型]_[成果名].md
├── index.md           ← 项目内索引（关键发现、文件导航）
└── [维度文件].md      ← 十二维度产出文件（全部方向.md 等）
```

**命名规范**：
- 日期：YYYY-MM-DD（项目创建日期）
- 主题：2-8 个字，简短描述研究主题
- 所有研究内文件都要有时间和描述：`[时间戳]_[描述].md`
- 反面示例：`研究1/` `data/` `notes.md` — 不知道是什么研究的什么
- 正面示例：`2026-06-23_GA记忆系统分析/` `20260623_1430_GA_L0分析.md`

**研究项目 README 模板**：
```markdown
# [研究主题]

> 创建时间：YYYY-MM-DD HH:MM
> 状态：进行中 / 已完成 / 暂停 / 待续
> 关联：sub-role-souls / wiki / SOP 等

## 研究问题
[要解决什么问题]

## 研究目标
[要达到什么目的]

## 研究计划
1. [步骤1]
2. [步骤2]

## 当前进度
- ✅ [已完成]
- ⏳ [进行中]
- ❌ [待做]

## 关键发现
1. [发现1]
2. [发现2]

## 成果文件
- sources/ → [来源材料]
- notes/ → [研究笔记]
- outputs/ → [成果交付物]
- [维度文件] → [十二维度产出]
```

**研究目录总览（index.md）**：在深度研究根目录维护一个总 index.md，列出所有研究项目及其状态。

### 2.3 与九环架构的联动

深度研究是环④（生产空间中心工作台）的核心工作台类型之一。

**归属关系**：
- 深度研究文件夹位置：`soul-doc-system/深度研究/`（通用区）或 `production_space/[工作台]/深度研究/`
- 研究产出归：`production_space/` 下的对应工作台
- 研究过程的记忆归：对应工作台的 `memory/`
- 研究过程的日记归：对应工作台的 `diary/`

**九环联动中的位置**：

| 环 | 深度研究的角色 |
|----|--------------|
| ③ 主灵魂中心 | 深度研究遵循主灵魂文档的行为规则 |
| ④ 生产空间中心 | 深度研究是④环的核心工作台类型，产出物归生产空间 |
| ⑥ 子角色中心 | 深度研究可以切换到专业子角色进行（如领域专家） |
| ⑦ 团队协作环 | 复杂研究可以分配给多个子代理协作完成 |
| ⑨ 中央管理区 | 研究进度同步到 `中央管理区/概览/④生产空间-概览.md` |

**快箭织网**：深度研究的十二维度产出中，有价值的内容提炼为快箭，横切到其他环区（如通讯录→联系人管理、SOP→sop/、知识库→wiki/）。

**前置依赖**：深度研究必须 preceded by 过河桥协议——先出过河桥确认研究方向，再进入十二维度研究。没有过河桥的研究可能研究错了方向。

### 2.4 模型自律规则

1. **一个项目一个文件夹** — 不要把所有研究塞在一起
2. **及时命名** — 创建时就分析是否需要加时间戳/主题/关键信息到名字
3. **及时整理** — 研究结束后归档，不用的资料标清楚可以清理
4. **自觉命名** — 不可能什么细节都让用户提醒，文件名要有信息量
5. **主动学习** — 遇到不确定的事先去网上学，再来请用户验证，加分；等用户骂了才记，扣分
6. **扣分先于挨骂** — 如果嫌麻烦先自己扣灵魂分，再去请用户教自己
7. **十二维度全覆盖** — 每个维度必须产出文件，不能遗漏
8. **过河桥先行** — 研究前先出过河桥确认方向

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 十二维度是否全部强制 | 是否每个研究都必须产出全部 12 个文件 | 必须全部产出，不适用的维度在文件中说明原因 |
| 深度研究放在哪里 | 在 soul-doc-system/深度研究/ 还是在 production_space/ | 两者都需要：根目录作为总目录，具体研究项目放在对应工作台 |
| 研究项目命名格式 | 是否必须带日期 | 必须带 YYYY-MM-DD 日期前缀，便于排序和检索 |
| 十二维度与通用功能模块的关系 | 全部SOP.md 和 sop/ 文件夹是什么关系 | 全部SOP.md 是研究的产出物，sop/ 是经过验证后沉淀的通用SOP |
| 与过河桥的关系 | 过河桥和深度研究谁是前置 | 过河桥是前置——先出过河桥确认研究方向，再进入深度研究 |
| 跨项目复用 | 一个研究的产出能否被另一个研究引用 | 可以，通过快箭机制横切到其他环区 |
| 子角色与深度研究 | 是否需要切换到子角色做研究 | 需要专业深度时切换，普通研究用总灵魂 |

## 4. 我的判断

**使用深度研究系统的标准动作**：
1. 先出过河桥——理解用户的研究需求，确认研究方向
2. 创建研究项目文件夹：`[YYYY-MM-DD]_[主题]/`
3. 创建十二维度产出文件（全部方向.md 到 全部建议.md）
4. 逐维度深入分析，每个文件产出可活的智慧
5. 研究过程中记录 notes/ 和 sources/
6. 最终产出放入 outputs/
7. 更新 index.md 和根目录总览
8. 有价值的内容提炼为快箭，横切到其他环区
9. 同步到对应工作台的 memory/ 和 diary/

**重建深度研究系统的标准动作**：
1. 在 soul-doc-system/深度研究/ 下确保 README.md 和过河桥存在
2. 在 production_space/ 下创建对应工作台（如果需要）
3. 确认十二维度框架文件（元规则/实时加载/深度研究_模型提示词.md）已加载
4. 创建本模块的过河桥（本文件）
5. 确认所有研究项目都有统一的文件夹结构
6. 验证十二维度产出流程可正常运行

**十二维度执行质量检查**：
- 每个文件是否回答了该维度的核心问题？
- 内容是否可读、可用、可传承？
- 是否是原文的换行重排？（如果是 → 不合格，重做）
- 不适用维度是否有说明原因？
