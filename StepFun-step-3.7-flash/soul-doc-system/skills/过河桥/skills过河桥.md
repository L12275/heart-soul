## 1. 用户原话重述

用户在系统文件中描述了技能系统：可复用功能模块的管理。在模块同步维护规则表中，skills 位于通用区 soul-doc-system/skills/，同步到工作台模版（不用于账号空间模版）。

用户的核心意图：让模型能够管理和积累可复用的功能模块（skills），区分系统级 skill（通过 halo-apps 安装）和模型自定义的技能积累（编码能力、工具熟练度等）。

---

## 2. 理解要点

**核心机制**：
- 技能系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/skills/`，仅同步到工作台模版（不用于账号空间模版）
- 技能系统管理两类内容：系统级 skill（通过 halo-apps skill_manage 安装/卸载）和模型自定义的技能积累

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 技能 skills | soul-doc-system/skills/ | ✅ | — |

**模块路径**：
- 规范位置：soul-doc-system/skills/
- 隔离区内：skills/
- 模板位置：新建工作台模版/skills/（账号空间模版不需要）

**两种技能层级**：

1. **系统级 Skill（通过 halo-apps 管理）**：
   - 安装：`skill_manage(action="install", slug="code-commit")` 或 `skill_manage(action="install", spec=JSON)`
   - 卸载：`skill_manage(action="uninstall", skill_id="xxx")`
   - 安装范围：global（所有空间可用）或 space（当前空间可用）
   - 当前可用的系统级 skills：update-config、simplify、loop、claude-api、bili-upload-image、Multi_Search_Engine、thinking-model-enhancer

2. **模型自定义技能积累（代码工作台技能清单）**：
   - 分类：语言技能、框架技能、工具技能、领域技能
   - 记录格式：技能名称、等级（入门/熟练/专家/大师）、经验时长、证明、待提升
   - 规则：文件超过20个时必须按类别分类到子文件夹

**技能分类体系（来自代码工作台技能清单模板）**：
| 类别 | 说明 | 示例 |
|------|------|------|
| 语言技能 | 编程语言掌握程度 | Python专家、Rust入门 |
| 框架技能 | 框架使用经验 | React、Django |
| 工具技能 | 工具使用熟练度 | Git、Docker |
| 领域技能 | 专业领域知识 | 后端架构、算法 |

**多 agent 类技能（已有子目录）**：
- skills/multi-agent类/agent-team-designer/ — Agent团队设计技能
- 其他 agent 类技能按类别放在 skills/ 下

**与 SOP 的区别**：
- skills = 可复用的功能模块（有输入输出、可独立运行）
- SOP = 标准操作流程（描述"怎么做"的步骤文档）
- SOP 从过河桥和发现中归纳而来，skills 是具体可执行的模块

**与九环架构的关系**：
- 技能系统服务于所有环区（通用功能模块）
- 在 环④ 生产空间中心工作台中，技能是工作能力的基础
- 在 环⑥ 子角色中心中，每个子角色可能有不同的技能组合

**与模块同步维护规则的关系**：
- skills/ 仅同步到工作台模版，不用于账号空间模版
- 原因：技能是工作能力，不是账号空间的属性

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| skill 和 SOP 的区别 | 两者都是"怎么做"的知识，区别是什么？ | skill = 可执行的功能模块（有明确的输入输出）；SOP = 步骤文档（描述过程，不一定可独立执行） |
| 系统级 skill 和自定义技能 | 两者是否混放？ | 不混放。系统级 skill 通过 halo-apps 管理（install/uninstall），自定义技能放在 skills/ 目录下按类别组织 |
| 技能文件超过20个 | 超过20个时必须分类，分类标准是什么？ | 按技能类别分类（语言/框架/工具/领域），如果类别仍太多，再分子类别 |
| 技能是否可以在共产中心共享 | 自定义技能可以在共产中心共享吗？ | 技能积累（个人经验）不在共产中心共享。系统级 skill 的 slug 可以在公告板登记，方便其他模型安装 |
| 账号空间模版为什么没有 skills | 为什么 skills 不用于账号空间模版？ | 技能是工作能力，不是账号属性。账号空间模版不需要包含技能模块 |

## 4. 我的判断

**使用技能系统的标准动作**：
1. 读本过河桥，理解两种技能层级（系统级 vs 自定义）
2. 检查已安装的系统级 skills（通过 skill_manage 或直接查看）
3. 管理自定义技能积累（按类别组织在 skills/ 下）
4. 需要新 skill 时 → 先检查是否有系统级 skill 可用 → 如果没有 → 自己积累

**安装系统级 Skill 的标准动作**：
1. 确认 skill 的 slug 或 spec
2. 确定安装范围（global 或 space）
3. 调用 skill_manage(action="install", slug/spec, scope)
4. 验证 skill 已安装可用

**积累自定义技能的标准动作**：
1. 执行任务时发现值得积累的技能
2. 按分类（语言/框架/工具/领域）写入 skills/ 目录
3. 记录格式：名称、等级、经验时长、证明、待提升
4. 文件数超过20个 → 自动分类到子文件夹

**重建技能模块的标准动作**：
1. 在 soul-doc-system/skills/ 下确认文件夹结构
2. 确认 multi-agent类/ 等子目录存在
3. 按模块同步维护规则，从通用区复制到工作台模版（不复制到账号空间模版）
4. 验证系统级 skills 的安装状态
5. 验证模块可独立运行
