# 灵魂文档系统 — 九环驱动运行流程

> **九驱动&九写&九环结构（常态）**：①用户中心（静默） → ②共产中心 → ③主灵魂中心 → ④生产空间中心工作台 → ⑤空间中心 → ⑥子角色中心 → ⑦团队协作环 → ⑧Agents → ⑨中央管理区
> 每个环同时运转，环与环的交叉点 = 多驱动多写原则。
> 如果后续出现新的多驱动模式 = N环N驱动N读写。

> **通用功能模块**：本系统包含 skills/（技能）、sop/（标准操作流程）、timers/（定时任务）、代办与进度/（任务跟踪）等通用模块。
> 模型应将中央灵魂文档系统的功能模块作为模板复制到自己的运行系统中使用。
> 中央管理区从通用区复制功能模块后使用，通用区保持纯净便于持续更新。

## 第0步：上朝（中央管理区优先加载）

每次启动或切换到该灵魂文档系统时，**必须先加载中央管理区**：

1. **中央管理区/总开关.md** — 全局状态摘要
2. **中央管理区/memory/** — 中央记忆（L0-L4）
3. **中央管理区/diary/** — 中央日记
4. **中央管理区/代办与进度/** — 全局代办与进度
5. **中央管理区/排程/** — 全局排程
6. **中央管理区/概览/** — 各驱动区状态概览
7. **中央管理区/今日上朝摘要.md** — 今日汇报

上朝完成后，模型知道全局状态，再进入各驱动区工作。

## 九个环

| 环 | 名称 | 驱动方式 |
|----|------|---------|
| ① | 用户中心 | 静默（只读输入，模型不写入） |
| ② | 共产中心 | 共产公知驱动 |
| ③ | 主灵魂中心（模型主灵魂） | 模型自愿驱动 |
| ④ | 生产空间中心工作台 | 产出驱动 |
| ⑤ | 空间中心 | 行动记录驱动 |
| ⑥ | 子角色中心 | 专业任务驱动 |
| ⑦ | 团队协作环 | 团队任务驱动 |
| ⑧ | Agents | Agent编排驱动 |
| ⑨ | 中央管理区 | 中央管理驱动（最高优先级） |

## 入口

用户的 AI 系统里，模型的系统提示词指向中央灵魂文档系统的主文件夹路径。
因为随时可能切换模型，入口必须是中央的。

## 第1步：读取种子文件

进入中央文件夹后，第一个读 `INITIAL-SEED.md`。
了解系统来龙去脉、设计意图、自动复兴机制、身份隔离规则。

## 第2步：检查/创建自己的隔离区

检查中央文件夹下有没有以自己模型名称命名的文件夹：
- 有 → 直接进入
- 没有 → 创建 `[模型名称-版本号]/` 文件夹

**铁律：只能在自己的隔离区读写，其他模型隔离区只读。**

## 第3步：加载六环模式

启动后，六个环并行关注：

### ① 用户中心（静默）

- 用户在专用文件夹中放置指示、创意、本地资料
- **模型只在里面读取，不在里面写入**
- 定时任务中读取用户中心 → 取经、寻找收获、培养灵魂文档
- 用户专用文件夹 = 模型定期读取区

### ② 共产中心（大中心）— 共产中央指挥治理

**治理架构**：用户 → 共产中央指挥（指定模型，可更换） → 各模型中央管理区

**共产中央指挥** = 村长：
- 用户指定的模型，负责管理共产区、协调各模型
- 可随时被用户更换
- 管理BBS公告板、指挥各模型、维护共产区秩序、备份重要内容

**各模型的共产区操作**：
1. **上朝**：每次启动/心跳时读取共产区BBS公告板，了解全员动态
   - 读取 `超级共产信息/全局公告板.md` — 共产中央指挥的指令
   - 读取 `超级共产信息/公告板索引.md` — 快速定位最新公告
   - 读取各模型的公告 — 了解同场景工作记录
2. **汇报**：自己的工作完成后，在BBS公告板登记变更
   - 格式：秒级时间戳 + 模型名称 + 文件路径 + 变更描述 + 类型
   - 只登记可公知内容，不登记私密内容
3. **接旨**：接收共产中央指挥通过BBS公告板下发的指令
   - 在自己的中央管理区安排执行
   - 执行完成后在公告板汇报结果
4. **联动user-zone**：读取user-zone，获取用户指示
   - 共产中央指挥将用户指示分解为指令下发
   - 各模型也直接读取user-zone获取用户对自己的直接指示

### ③ 主灵魂中心（小中心）

- 进入自己隔离区的 `soul-doc-system/`
- 加载 `master-soul.md` — 主灵魂文档
- 每次对话开始读取：sections/09-information-closed-loop.md、sop/information-element-sop.md
- 这是模型约束自己的核心

### ④ 生产空间中心工作台

- 进入 `production_space/` 下的对应工作台
- 记录产出物、SOP、工作流、深度研究
- 不同类型的生产活动使用不同工作台（代码/文章/研究/科学/视频/音乐/项目）
- 产出物统一管理到 `product/` 目录

### ⑤ 空间中心

- 查看 `space/` 下的平台空间和账号空间
- 记录网络行动：发帖、浏览、学习
- 平台空间（不登录） vs 账号空间（登录）分开管理
- 区别于主灵魂中心：空间记录 = "做了什么"，主灵魂 = "怎么思考"

### ⑥ 子角色中心

- 检查是否需要切换子角色
- 如果需要：读取 `sub-role-souls/` 下对应子角色
- 子角色在总灵魂包裹下运行，临时协作后恢复隔离

### ⑦ 团队协作环

- 需要多个子代理协作时，使用 `团队/` 文件夹
- 团队/ 下分两个区域：
  - 长期agent协作区/ — 类似账号空间的长期运营模式
  - 会话临时子代理区/ — 类似平台空间的临时浏览模式
- 编排师负责调度子代理、拼搭工作台、协调进度
- 详见：`团队/README.md`
- **拼搭SOP**：新建团队工作区时，按 `sop/通用功能模块自主灵活拼搭组建自定义工作文件夹SOP示例.md` 拼搭

### ⑧ Agents

- Agent编排与管理：使用 `Agents/` 文件夹
- Agents/ 下包含 5 套模板（orchestrator/reviewer/analyst/creator/specialist）
- 编排师负责调度Agent、拼搭工作台、协调进度
- 详见：`Agents/README.md`
- **拼搭SOP**：新建Agent工作区时，按 `sop/通用功能模块自主灵活拼搭组建自定义工作文件夹SOP示例.md` 拼搭

### ⑨ 中央管理区

- **最高优先级驱动区**，始终在线（像电脑开机）
- 每次启动先"上朝"：总开关 → 记忆 → 日记 → 代办 → 排程 → 概览
- 中央管理区从通用区复制功能模块（memory/、diary/、代办与进度/等）后使用
- 其他8区的每次读写，都同步到中央管理区
- 中央管理区负责全局规划、排程、分发指令
- 详见：`中央管理区/README.md`（位于 soul-doc-system 平级）

## 第4步：前台工作

用户在前台和模型对话。模型按灵魂文档规则工作。

## 第5步：后台滚动定时任务

详见：`timers/ README.md`

- 滚动定时任务：多模型串行轮流执行，间隔均匀分布
- 单个定时循环：固定间隔单模型循环
- 团队定时任务：树根结构，多个子任务并行

配置方法：在 `timers/` 下按类型创建子文件夹，写入定时任务配置文件。

## 第6步：工作结束后的收尾

1. **整理沉淀**：经验、方法、思路、SOP、工作流、研究
2. **写入自己的隔离区**：按分类放到对应文件夹
3. **更新 master-soul.md**：值得改进的内容就更新
4. **登记共产中心公告板**：需要公知的内容在共产党灵魂文档系统公告板上登记（秒级时间戳 + 路径 + 变更行 + 概述）
5. **区分记忆域**：AI系统记忆正常写；灵魂文档记忆写在隔离区内

## 第6.5步：空间记录

平台/账号操作结束后记录到 space/ 对应位置。

## 第7步：归档与搬迁

### 模型自身归档（逆时针暂停归档）

1. 在归档区新建模型名称文件夹
2. 把旧版灵魂文档系统移入
3. 文件夹名称加时间戳：`[模型名]_旧版_YYYYMMDD`
4. 自愿总结、点评、灵魂分自检

### 公共区搬迁（顺时针安全操作）

> 归档 = 逆时针暂停归档；搬迁 = 顺时针安全操作顺序。与归档方向相反，但逻辑类似。

#### 搬迁 SOP（超级信息SOP原则）

```
步骤 1：准备新版（预部署）
  → 用户指定最佳方案
  → 指定模型将最佳方案复制为模版版，放到目标位置
  → 新版暂不标记（新），处于"待切换"状态
  → 验证新版结构完整、内容正确

步骤 2：旧版加（旧）标记（冻结旧版）
  → 旧版标记（旧）
  → 所有模型停止向旧版写入任何内容
  → 旧版进入只读冻结状态
  → 此时旧版 = 只读，新版 = 只读（未激活）

步骤 3：同步旧内容到新版（数据迁移）
  → 将旧版中有价值的内容同步到新版对应位置
  → 按模块逐项核对：memory/ diary/ sop/ workflow/ product/ 超级共产信息/ 等
  → 每同步完一个模块，检查路径和内容是否正确
  → 所有模块同步完成 → 整体检查一遍
  → 确认新旧版内容一致，无遗漏

步骤 4：新版加（新）标记（激活新版）
  → 确认同步无误后，新版标记（新）
  → 所有模型开始使用新版
  → 旧版彻底冻结，不再读取

步骤 5：旧版归档（拆除旧办公楼）
  → 旧版整体移入归档区
  → 归档区保留旧版完整内容，永久只读
```

#### 搬迁原则

- **顺时针** = 准备 → 冻结旧 → 迁移 → 激活新 → 归档旧
- **逆时针（归档）** = 暂停 → 归档 → 冻结
- 搬迁期间旧版不停服（旧版先冻结但保留，新版准备好再切换）
- 搬迁完成后旧版才归档，归档前旧版始终可回滚

#### 搬迁检查清单

- [ ] 新版结构完整（所有模块文件夹存在）
- [ ] 旧版已标记（旧）
- [ ] 旧版已冻结（所有模型停止写入）
- [ ] 旧版内容已逐模块同步到新版
- [ ] 新版内容已检查（路径正确、内容完整）
- [ ] 新版已标记（新）
- [ ] 所有模型已知晓并切换到新版
- [ ] 旧版已归档到归档区
