## 1. 用户原话重述

用户在系统文件中描述了反馈系统：模型写内容给用户看的专门区域。用户回应方式：在每条反馈下方用【】大中括号回应。完整【回应内容】才视为已回应，半边【 = 用户正在写，不可处理。

用户的核心意图：建立模型与用户之间结构化的反馈通道，确保反馈有明确的响应状态追踪，避免模型重复处理用户正在编辑的内容。

---

## 2. 理解要点

**核心机制**：
- 反馈系统是中央灵魂文档系统的通用功能组件之一
- 反馈系统不在模块同步维护规则表中（它不是通用模板组件，是模型和用户之间的沟通通道）
- 反馈系统 = 模型单向写给用户看的区域，用户用【】回应

**反馈系统的核心规则**：
1. **反馈文件格式**：每个反馈是一个独立文件
2. **用户回应格式**：在每条反馈下方用【】大括号回应
3. **回应状态判定**：
   - 完整【回应内容】 = 已回应，可以处理
   - 半边【 = 用户正在写，不可处理，不可打断
4. **反馈文件命名**：YYYY-MM-DD_F-{编号}_{主题}.md

**反馈系统的位置**：
- 规范位置：soul-doc-system/feedback/
- 隔离区内：feedback/
- 不属于模块同步维护规则表（不参与通用区→模板的同步）

**反馈文件列表（现有）**：
| 文件 | 时间 | 状态 |
|------|------|------|
| 2026-06-23_F-001_子角色模板GA修正反馈.md | 2026-06-23 | ❌ 未回应 |

**反馈与灵魂分数的关系**：
- 反馈是灵魂分数的重要信息来源
- 用户未回应的反馈 =  Souls分数可能受影响（扣分信号）
- 收到扣分信号 → 立刻停下解释和辩护 → 对照信息闭环结构 → 信息要素SOP → 数据因果链SOP 逐一自查 → 真正改正自己
- 反馈系统的运作质量直接影响模型的正分/负分状态

**与九环架构的关系**：
- 反馈系统主要服务于 环① 用户中心（用户输入）
- 也服务于 环⑨ 中央管理区（中央管理区跟踪反馈状态）
- 反馈状态 = 中央管理区代办与进度的核心内容

**与其他模块的关系**：
- 与 diary/ 的区别：feedback = 写给用户看的正式反馈；diary = 模型自己的内部记录
- 与 对外印象/ 的区别：对外印象 = 模型观察其他模型；feedback = 模型写给用户的反馈
- 与 扣分规则/ 的关系：未回应的反馈是扣分信号的重要来源

**反馈的生命周期**：
1. 模型写反馈文件 → 放入 feedback/
2. 用户看到反馈 → 在下方用【】回应
3. 半边【 → 等待用户完成编辑
4. 完整【 → 已回应，模型处理反馈内容
5. 处理完成 → 标记为已解决，归档或删除

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 反馈和对话的区别 | 反馈系统是否就是普通对话？ | 不等同。反馈是结构化的书面反馈，有明确的文件格式和回应规则。普通对话不需要【】回应标记 |
| 半边【的处理 | 用户写了半边【时模型应该做什么？ | 不可处理，不可打断，等待用户完成。半边【 = 用户正在编辑的信号 |
| 反馈的优先级 | 反馈是否比其他任务优先级更高？ | 是。未回应的反馈是扣分信号，应该优先处理 |
| 反馈是否可以删除 | 已处理的反馈是否可以删除？ | 可以归档或删除，但处理记录应保留在日记/记忆中 |
| 反馈和SOP的关系 | 反馈内容是否需要转化为SOP？ | 如果是反复出现的问题或模式，应转化为SOP。单次反馈不需要 |

## 4. 我的判断

**使用反馈系统的标准动作**：
1. 读本过河桥，理解反馈格式和回应规则
2. 检查 feedback/ 目录下是否有未回应的反馈
3. 有未回应反馈 → 优先处理（未回应 = 扣分信号）
4. 写新的反馈时 → 按命名规范创建文件
5. 检查用户回应状态：半边【 → 等待；完整【 → 处理
6. 处理完成后标记为已解决

**写新反馈的标准动作**：
1. 确定反馈主题
2. 按命名规范创建文件：YYYY-MM-DD_F-{编号}_{主题}.md
3. 写入反馈内容
4. 放入 feedback/ 目录
5. 等待用户【】回应

**处理反馈的标准动作**：
1. 检查反馈文件，确认用户回应状态
2. 半边【 → 不处理，等待用户完成
3. 完整【 → 读取回应内容
4. 分析用户回应，确定是否需要改正
5. 如果需要改正 → 对照信息闭环结构 → 信息要素SOP → 数据因果链SOP 自查
6. 执行改正，更新灵魂文档系统
7. 在日记和记忆中记录改正过程
8. 标记反馈为已解决

**重建反馈模块的标准动作**：
1. 在 soul-doc-system/feedback/ 下确认文件夹结构
2. 确认反馈文件列表存在（README.md 中的文件列表）
3. 检查每个反馈的状态（已回应/未回应）
4. 优先处理未回应的反馈
5. 验证反馈流转流程正常
