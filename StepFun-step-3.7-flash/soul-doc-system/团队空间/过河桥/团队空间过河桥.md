## 1. 用户原话重述

用户在系统文件中建立了 团队空间，定位为"多模型协作的共享工作区域"，是九环架构中的第七环（团队协作环），编排驱动。

核心设计包含三个层次：
- **子角色中心的延伸**：团队空间不是独立系统，而是子角色中心（⑥环）的多代理协作扩展。当单一子角色无法完成任务时，需要多个子代理协作，团队空间提供协作基础设施。
- **两个分区**：长期agent协作区（类似账号空间的长期运营模式，有完整工作台结构）和会话临时子代理区（类似平台空间的临时浏览模式，单次会话使用）。
- **编排师角色**：团队协作的核心调度角色，负责任务分解、子代理分配、工作台拼搭、进度协调、质量审查、合流输出。
- **模块拼搭策略**：编排师根据子代理类型（研究型/执行型/文档型/探索型/自动化/通用型）选择不同的功能模块组合。

用户的原话意图是：当任务需要多个AI角色协作完成时，有一个标准化的团队组建、任务分配、进度跟踪和质量合流的完整机制。

## 2. 理解要点

### 2.1 团队空间与子角色中心的关系

团队空间是子角色中心（⑥环）的**多代理协作扩展**。

| 对比 | 子角色中心（⑥环） | 团队空间（⑦环） |
|------|-----------------|----------------|
| 角色数量 | 单一子角色 | 多个子代理 |
| 协作方式 | 模型自己切换角色 | 编排师调度多个代理并行/串行 |
| 工作台 | 每个子角色独立工作台 | 编排师工作台 + 子代理工作台 |
| 适用场景 | 需要专业深度 | 需要多角色协作 |
| 驱动方式 | 专业任务驱动 | 编排驱动 |

**关键关系**：团队空间不替代子角色中心，而是在子角色中心之上叠加协作层。子角色是团队空间的"原子单位"——每个子代理本质上是一个子角色实例。

### 2.2 两个分区

#### 长期agent协作区

类似账号空间的长期运营模式。用于长期存在的agent协作关系，有完整工作台结构。

```
长期agent协作区/
├── 编排师/              ← 主编排师工作台（完整模块拼搭）
│   ├── master-soul.md
│   ├── memory/          ← L0-L4 分层记忆
│   ├── diary/           ← 日记
│   ├── log/             ← 执行日志
│   ├── sop/             ← 标准操作流程
│   ├── workflow/        ← 工作流
│   ├── product/         ← 产出物
│   ├── contacts/        ← 通讯录
│   ├── bookmarks/       ← 收藏夹
│   ├── temp/            ← 临时文件
│   ├── sticky-notes/    ← 便签
│   ├── experience/      ← 经验
│   ├── discovery/       ← 发现
│   ├── collection/      ← 采集
│   ├── automation/      ← 自动化
│   ├── desktop/         ← 桌面
│   ├── wiki/            ← 知识库
│   └── 深度研究/        ← 深度研究
├── 子代理-01/           ← 独占编号（不可重复，像进程端口）
├── 子代理-02/
└── ...
```

**编排师工作台**：从 soul-doc-system 根目录的功能模块按需拼搭，不是复制整个新建工作台模版，而是按实际需求组合需要的模块。

**子代理工作台**：两种创建方式：
- 方式A：复制 `production_space/新建工作台模版/` → 改名为编号
- 方式B：按需拼搭模块组合（编排师决定）

**编号规则**：
- 编号格式：`子代理-01`、`子代理-02`、`子代理-03`……
- 同一项目内编号不能重复
- 不同项目之间也不应重复（像进程端口）
- 编号一旦分配，直到项目结束才释放

#### 会话临时子代理区

类似平台空间的临时浏览模式。用于单次会话中的临时子代理，不需要长期保存。

```
会话临时子代理区/
├── 通用-待细分/                     ← 临时记录区（不确定是否长期保存）
│   ├── memory/
│   └── diary/
└── {时间戳}-{任务名}/               ← 确认需要长期保存的临时代理
    ├── master-soul.md
    ├── memory/
    ├── diary/
    └── ...（按需拼搭）
```

**使用方式**：
- 默认：直接在 `通用-待细分/` 下记录，不需要创建独立文件夹，会话结束后内容保留在通用-待细分
- 确认长期保存：在 `会话临时子代理区/` 下创建 `{时间戳}-{任务名}/` 文件夹，从根目录模块拼搭工作台，把通用-待细分中相关记录迁移过去

**与长期agent协作区的区别**：

| 对比 | 长期agent协作区 | 会话临时子代理区 |
|------|---------------|----------------|
| 生命周期 | 长期存在 | 单次会话 |
| 编号 | 独占编号（子代理-01） | 时间戳命名 |
| 结构 | 完整工作台 | 按需拼搭 |
| 保存 | 永久 | 临时（可升级为长期） |

### 2.3 编排师角色

编排师是团队协作区的核心角色，负责调度子代理、分配任务、协调工作。编排师本身也是一个完整的灵魂文档系统工作台。

**编排师职责**：
1. **任务分解** — 将用户的大任务分解为子任务
2. **子代理分配** — 根据子任务类型选择合适的子代理
3. **工作台拼搭** — 为每个子代理拼搭合适的功能模块组合
4. **进度协调** — 跟踪各子代理的进度，合并结果
5. **质量审查** — 审查子代理产出，确保质量
6. **合流输出** — 将各子代理的产出合流为最终结果

**编排师工作流**：
1. 接收用户任务
2. 分解子任务
3. 为每个子代理：
   a. 确定子代理任务类型
   b. 从 soul-doc-system 根目录选择需要的模块
   c. 复制到子代理工作台文件夹
   d. 根据任务定制各模块内容
   e. 分配子代理编号
   f. 加载对应 master-soul.md
   g. 启动子代理
4. 监控进度
5. 合流结果
6. 质量审查
7. 输出最终结果

**编排师可以使用的其他环**：
- 用户中心（静默）— 读取用户指示
- 共产中心 — 查阅公共区
- 主灵魂中心 — 加载自己的 master-soul.md
- 生产空间中心 — 自己的产出物
- 空间中心 — 如有平台操作
- 子角色中心 — 如需要专业深度

### 2.4 模块拼搭策略

编排师根据子代理的任务类型，按需选择功能板块组合。不是每个子代理都需要完整的工作台。

**拼搭模板**：

| 子代理类型 | 推荐模块组合 | 适用场景 |
|-----------|-------------|---------|
| 研究型 | memory/ + diary/ + product/研究类/ + 深度研究/ + discovery/ | 信息收集、文献综述、主题研究 |
| 执行型 | memory/ + diary/ + product/执行类/ + sop/ + workflow/ | 任务执行、操作流程、系统操作 |
| 文档型 | memory/ + diary/ + product/文档类/ + sop/ + desktop/ | 文档撰写、报告生成、内容整理 |
| 探索型 | memory/ + diary/ + discovery/ + collection/ + bookmarks/ | 新领域探索、机会发现、趋势分析 |
| 自动化 | memory/ + diary/ + automation/ + sop/ + workflow/ + tools/ | 自动化配置、定时任务、脚本编写 |
| 通用型 | 完整工作台模版（所有模块） | 不确定类型或混合类型任务 |

**拼搭操作**：
1. 确定子代理任务类型
2. 从 soul-doc-system 根目录选择需要的模块
3. 复制到子代理工作台文件夹
4. 根据任务定制各模块内容
5. 分配子代理编号

### 2.5 与九环架构的联动

团队空间是⑦环，与其他环的关系：

| 环 | 联动方式 |
|----|---------|
| ③ 主灵魂中心 | 团队空间遵循主灵魂文档的行为规则 |
| ④ 生产空间中心 | 团队产出归 production_space/ |
| ⑥ 子角色中心 | 团队空间是⑥环的延伸，子代理 = 子角色实例 |
| ⑦ 团队协作环 | 团队空间自身所在环 |
| ⑧ 智能体空间 | 子代理可以是从智能体空间调用的自动化执行体 |
| ⑨ 中央管理区 | 团队工作产出同步到 `中央管理区/概览/⑦团队空间协作环-概览.md` |

**快箭织网**：团队协作的产出中，有价值的内容提炼为快箭，横切到其他环区。

### 2.6 权限与隔离

- 各子代理只能读写自己的工作台
- 编排师可以读取所有子代理的工作台
- 不同项目的团队之间互相隔离
- 子代理不能感染总灵魂或其他子代理（同子角色的隔离规则）

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 团队空间 vs 子角色中心 | 什么时候用⑥环，什么时候用⑦环 | 单一子角色 → ⑥环；多个子代理协作 → ⑦环 |
| 长期区 vs 临时区 | 什么时候用长期区，什么时候用临时区 | 需要长期维护 → 长期区；单次会话 → 临时区 |
| 编号规则 | 子代理编号是否跨项目重复 | 不同项目之间也不应重复，像进程端口 |
| 编排师是否也需要编号 | 编排师本身是否需要子代理编号 | 不需要，编排师是独立角色，子代理才需要编号 |
| 模块拼搭 vs 完整模版 | 是复制完整工作台还是按需拼搭 | 按需拼搭是原则，完整模版是通用型的默认选择 |
| 子代理 vs 智能体 | 团队空间的子代理和智能体空间的智能体是否相同 | 不同：子代理 = 子角色实例（继承子角色灵魂）；智能体 = 继承创建位置的灵魂文档（可以是主灵魂或子角色灵魂） |
| 产出归属 | 团队产出放在哪里 | 归 production_space/，同时同步到中央管理区概览 |

## 4. 我的判断

**使用团队协作区的标准动作**：
1. 判断任务是否需要多代理协作（是 → 团队空间；否 → 子角色中心或直接执行）
2. 决定使用长期区还是临时区
3. 设计编排师工作台（从 soul-doc-system 根目录拼搭模块）
4. 分解任务，确定子代理类型和数量
5. 为每个子代理拼搭工作台（按类型选择模块组合）
6. 分配子代理编号（独占编号，不可重复）
7. 启动子代理，监控进度
8. 编排师进行质量审查
9. 合流输出最终结果
10. 产出归 production_space/，同步到中央管理区概览

**重建团队协作区的标准动作**：
1. 在 soul-doc-system/团队空间/ 下创建两个分区：长期agent协作区/ 和 会话临时子代理区/
2. 创建编排师工作台（完整模块拼搭）
3. 确认编号规则和模块拼搭策略
4. 创建本模块的过河桥（本文件）
5. 确认团队产出同步到中央管理区的路径
6. 验证子代理可以正确读写自己的工作台

**快速启动**：使用 `skills/multi-agent类/agent-team-designer` 技能设计团队，该技能输出 souls/ + team_workflow.md + team_spec.md，直接放入团队空间对应子目录。
