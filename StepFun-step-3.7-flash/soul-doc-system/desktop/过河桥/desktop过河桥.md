过河桥 — desktop

> **此过河桥是 桌面系统 的基因片段。**
> 桌面系统 = 空白工作台、临时堆放区、平面白板。
> 是中央灵魂文档系统的通用功能组件之一，参与模块同步维护规则。
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

## 1. 用户原话重述

用户在系统文件中描述了桌面系统：像空白桌面和地面，平面白板，临时堆放区。已有功能模块的内容分放到对应文件夹（bookmarks/、contacts/、sticky-notes/、workflow/、sop/等），不放到桌面。只有临时内容、材料收集场景、复杂临时文件管理才放到桌面。

用户的核心意图：为模型提供一个灵活的临时工作空间，用于汇集多种不同类型的材料，同时通过明确的规则防止临时内容被误当成稳定内容。

---

## 2. 理解要点

**核心机制**：
- 桌面系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/desktop/`，同步到新建工作台模版和新建账号空间模版
- 桌面 = "空白平面白板"，是临时堆放区，不是正式存储区

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 桌面 desktop | soul-doc-system/desktop/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/desktop/
- 隔离区内：desktop/
- 模板位置：新建工作台模版/desktop/、新建账号空间模版/desktop/

**桌面系统的结构**：
```
desktop/
├── README.md          ← 说明书（不可删除）
├── 桌面1/             ← 按需创建，空文件夹
│   ├── README.md      ← 说明用途
│   └── ...
├── 桌面2/             ← 按需创建
└── 桌面n/             ← 按需创建
```

**使用规则（来自 desktop README.md）**：
1. 说明书文件夹不能删，在说明书文件夹里新建 桌面n 使用
2. 不适合秒删的文件 → 复制到桌面，不要移动（桌面内容容易随桌面n一起秒删）
3. 桌面n 也是创建为空文件夹，按需建立
4. 创建桌面n 时同步写 README.md 说明用途
5. 删除桌面n 之前：必须检查里面所有文件，该转移的转移、该删除的删除，确认安全后才能删除桌面n

**什么放桌面 vs 不放桌面**：
| 已有模块 | 用途 | 桌面放不放 |
|----------|------|-----------|
| bookmarks/ | 收藏夹 | 不放 |
| contacts/ | 通讯录 | 不放 |
| sticky-notes/ | 便签 | 不放 |
| workflow/ | 工作流 | 不放 |
| sop/ | SOP | 不放 |
| memory/ | 记忆 | 不放 |
| diary/ | 日记 | 不放 |
| product/ | 产出物 | 不放 |
| desktop/ | 临时堆放 | **放这里** |

**桌面的三种使用场景**：
1. **临时内容**：暂时放到桌面，等整理后再分放
2. **材料收集场景**：把一堆各种文件当材料准备放到一个文件夹时，先放到桌面文件夹。桌面是空的平面白板，适合临时汇集多种不同类型的材料
3. **复杂临时文件管理**：按需创建 桌面1/、桌面2/、桌面n/ 子文件夹

**内容流转规则**：
- 稳定长期内容 → 升级到更高层级的用途存放区（product/、memory/、diary/等）
- 临时内容 → 留在桌面，任务完成后清理
- 不适合秒删的文件 → 复制到桌面，不要移动

**与九环架构的关系**：
- 桌面是通用功能模块，服务于所有环区
- 在 环④ 生产空间中心工作台中，桌面是各工作台的临时缓冲区
- 在 环⑨ 中央管理区中，桌面是待整理文件的临时区

**与过河桥协议的关系**：
- 过河桥是用户意图的持续校准记录，是记忆和日记的上游
- 桌面不是停止记忆和日记——桌面是临时堆放区，过河桥是源头记录
- 两者配合：过河桥记录意图 → 桌面临时堆放材料 → 整理后进入正式模块

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 桌面和临时文件夹 | 桌面和 collection/、temp/ 有什么区别？ | 桌面 = 多类型混合的临时堆放区；collection = 结构化采集的数据；temp = 纯临时文件 |
| 桌面内容能否长期放 | 桌面内容可以放多久？ | 原则上短期。稳定内容必须升级到正式模块 |
| 多个桌面n的管理 | 桌面1、桌面2...如何区分？ | 每个桌面n = 一个独立的任务/场景，README.md 说明用途 |
| 桌面的删除安全 | 删除桌面n 时如何确认安全？ | 必须检查所有文件，该转移的转移、该删除的删除，确认无重要内容后才能删除 |
| 和其他模块混淆 | desktop 和 production_space 有什么区别？ | production_space = 正式产出空间（有工作台、SOP、产出物）；desktop = 临时堆放区（无正式结构） |

## 4. 我的判断

**使用桌面系统的标准动作**：
1. 读本过河桥，理解桌面是临时堆放区，不是正式存储
2. 判断当前内容是否适合放到桌面（临时/收集/混合类型 = 适合；稳定/正式 = 不适合）
3. 如需创建桌面n，同步写 README.md 说明用途
4. 不适合秒删的文件 → 复制到桌面，不要移动
5. 任务完成后：将桌面中的内容整理到正式模块，清理桌面

**创建桌面n的标准动作**：
1. 在 desktop/ 下创建 桌面{n}/ 空文件夹
2. 同步写入 README.md 说明该桌面的用途
3. 将临时材料放入该桌面
4. 任务完成后按规则清理

**删除桌面n的标准动作**：
1. 检查 桌面n/ 中的所有文件
2. 有价值的文件 → 转移到对应正式模块（product/、memory/、wiki/等）
3. 无价值的文件 → 删除
4. 确认安全后删除 桌面n/ 文件夹
5. 删除前必须有完整的文件检查记录

**重建桌面模块的标准动作**：
1. 在 soul-doc-system/desktop/ 下确认 README.md 说明书存在
2. 按模块同步维护规则，从通用区复制到各模板位置
3. 验证桌面n的创建和删除流程正常
4. 验证模块可独立运行
