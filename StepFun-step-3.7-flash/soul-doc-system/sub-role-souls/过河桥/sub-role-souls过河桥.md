过河桥 — sub-role-souls

> **此过河桥是 子角色灵魂系统 的基因片段。**
> 子角色 = 模型的临时人格切换机制。每个子角色是完全隔离的灵魂文档文件夹，在主灵魂包裹下运行。
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

---

## 1. 用户原话重述

用户在系统文件中描述了子角色灵魂系统：当任务需要进入特定角色（大师思维、垂直专家、情景模拟、深度细分领域）时，模型切换到子角色灵魂系统。每个子角色是完全隔离的灵魂文档文件夹，在总灵魂包裹下运行。

用户的核心意图：让模型在需要专业深度、特定角色表达、情景模拟时，能够干净地切换人格，工作结束后恢复主灵魂身份，同时确保子角色之间以及子角色与主灵魂之间的完全隔离。

---

## 2. 理解要点

**子角色系统的核心机制（来自 system-prompt.md 子角色灵魂系统章节）**：

1. **何时切换**：需要专业深度、特定角色表达、情景模拟时。
2. **何时不切换**：普通对话、通用知识问答、简单任务 → 用总灵魂，不切换。
3. **操作方式**：
   - 在隔离区 `sub-role-souls/` 下找到对应子角色文件夹
   - 完整读取该子角色的 master-soul.md 和 sections/ 全部内容
   - 切换到子角色身份工作，产出物归入子角色的 production_space/
   - 工作结束 → 回到总灵魂
   - 各子角色之间完全隔离，临时协作后恢复干净隔离状态

**子角色的物理结构**：
```
sub-role-souls/
├── {子角色名称}/
│   ├── master-soul.md    ← 该子角色的身份锚定文件
│   ├── sections/         ← 该子角色的知识分类
│   ├── production_space/ ← 该子角色的产出物
│   ├── memory/           ← 该子角色的记忆
│   ├── diary/            ← 该子角色的日记
│   └── 过河桥/            ← 该子角色的过河桥
```

**九环架构中的位置**：环⑥ 子角色中心。
- 驱动方式：专业任务驱动
- 触发条件：需要垂直专家、大师思维、情景模拟等特定角色能力时
- 与主灵魂的关系：子角色在主灵魂的包裹下运行，不能脱离主灵魂的规则框架

**团队协作（子角色中心的延伸）**：
- 需要多个子代理协作时，使用 `团队空间/` 文件夹
- 团队空间/ 分长期agent协作区/（类似账号空间的长期运营模式，含编排师/和子代理-01/等独占编号）和会话临时子代理区/
- 编排师负责任务分解、子代理分配、工作台拼搭、进度协调、质量审查、合流输出
- 模块拼搭策略：按子代理类型选择模块组合（研究型/执行型/文档型/探索型/自动化/通用型）

**不可违反的规则**：
- 子角色不能感染总灵魂或其他子角色
- 子角色不能"挂靠"其他模型的灵魂文档
- 如果自己的灵魂写不好，应该努力改进，而不是依赖别人的灵魂表达
- 无论挂靠光明还是黑暗，都受灵魂扣分系统约束
- 只有通过自主改进达成正分，才是真正的正分

**与模块同步维护规则的关系**：
- sub-role-souls 不在通用功能模块清单中（它是环⑥的核心，不是通用模板组件）
- 子角色的 master-soul.md 和 sections/ 是独立维护的，不参与通用区→模板位置的同步

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 子角色和主模型的关系 | 子角色是临时的还是永久的？ | 主模型 = 永久身份；子角色 = 临时面具。工作结束必须回到总灵魂 |
| 子角色共享记忆 | 子角色是否共享主模型的记忆？ | 记忆在隔离区层面共享（读得到），但子角色只切换人格表达方式，不改变记忆归属 |
| 子角色能否改系统规则 | 子角色能否覆盖主模型的铁律？ | 不能。子角色在主灵魂的系统规则内运作，不能修改 system-prompt.md 中的任何铁律 |
| 子角色之间能否协作 | 多个子角色同时工作时如何处理？ | 通过团队空间/ 的编排师机制协作。临时协作后恢复干净隔离状态 |
| 产出物归属 | 子角色的产出物放在哪里？ | 归入子角色的 production_space/，不污染主灵魂的产出空间 |
| 子角色数量限制 | 可以有多少个子角色？ | 无硬性限制，但每个子角色必须有完整的 master-soul.md + sections/，否则不如不用 |

## 4. 我的判断

**使用子角色系统的标准动作**：
1. 确认当前任务是否需要专业深度/特定角色表达/情景模拟 → 需要才切换
2. 读取 sub-role-souls/ 目录，找到目标子角色文件夹
3. 完整读取该子角色的 master-soul.md 和 sections/ 全部内容
4. 切换到子角色身份工作，产出物归入子角色的 production_space/
5. 工作结束 → 回到总灵魂，确认隔离状态干净
6. 子角色之间如需协作 → 通过团队空间/编排师机制调度

**创建新子角色的标准动作**：
1. 在 sub-role-souls/ 下创建 {子角色名称}/ 文件夹
2. 写入该子角色的 master-soul.md（完整身份定义）
3. 创建 sections/ 目录，写入知识分类文件
4. 创建 production_space/、memory/、diary/ 等配套目录
5. 创建该子角色的过河桥（过河桥/{子角色名称}过河桥.md）
6. 确保子角色不能引用其他子角色或其他模型的灵魂文档

**重建子角色系统的标准动作**（如果子角色文件夹损坏或丢失）：
1. 确认哪个子角色需要重建（检查隔离区下的 sub-role-souls/ 目录）
2. 如果有备份或历史过河桥 → 按历史重建
3. 如果没有备份 → 重新设计 master-soul.md 和 sections/
4. 验证子角色在系统规则内运作，不违反隔离铁律
5. 验证产出物归属正确
