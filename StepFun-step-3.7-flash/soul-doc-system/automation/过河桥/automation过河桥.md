## 1. 用户原话重述

用户在系统文件中描述了自动化系统：定时任务和自动化流程执行的功能模块。在模块同步维护规则表中，automation 位于通用区 soul-doc-system/automation/，同步到工作台模版和账号空间模版。

用户的核心意图：让模型能够定义、配置和跟踪自动化任务，使灵魂文档系统能在没有用户实时干预的情况下持续运行和自我更新。

---

## 2. 理解要点

**核心机制**：
- 自动化系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/automation/`，同步到新建工作台模版和新建账号空间模版
- 它和 timers/ 模块紧密协作：automation/ 定义自动化流程，timers/ 提供时间触发机制

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 自动化 automation | soul-doc-system/automation/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/automation/
- 隔离区内：automation/
- 模板位置：新建工作台模版/automation/、新建账号空间模版/automation/

**模块结构（推断）**：
```
automation/
├── README.md           ← 说明书
├── tasks/              ← 自动化任务配置
├── workflows/          ← 自动化工作流定义
├── logs/               ← 执行记录
├── templates/          ← 任务模板
└── 过河桥/              ← 本文件
```

**与 timers 模块的关系**：
- timers/ 提供时间触发的骨架（滚动定时任务、单个定时循环、团队定时任务）
- automation/ 在 timers 的触发下执行具体动作
- 全自动灵魂文档研究更新系统 = timers 触发 → automation 执行

**与九环架构的关系**：
- 自动化系统主要服务于 环④ 生产空间中心工作台（产出驱动）
- 也服务于 环⑥ 子角色中心（专业任务驱动的定时执行）
- 定时任务行为规范：模型独立运行，不强制学习，看到其他模型更新 = 自愿决定是否借鉴
- 效果预期：本地滚动涌现 + 网络搜索涌现 + 白盒优选 + 免提示词筛选

**与共产中心的关系**：
- 自动化任务执行完成后，在共产中心公告板登记变更
- 登记格式：[秒级时间戳] [模型名称] 文件：相对路径 变更：第X行到第Y行 摘要：... 类型：【记忆更新】【产出新增】【SOP变更】【经验沉淀】

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| automation 和 timers 的边界 | 两者都涉及定时触发，区别是什么？ | timers = 时间触发的骨架（调度器）；automation = 被触发后执行的具体动作和流程 |
| 自动化是否强制 | 模型是否必须执行定时任务？ | 不强制。模型独立运行，自然表现。停止条件 = 自然完成 |
| 产出物归属 | 自动化任务的产出物放在哪里？ | 根据内容类型放入对应工作台（production_space/）或功能模块（memory/、diary/等） |
| 和其他模块混淆 | automation 和 workflow 有什么区别？ | workflow = 模型总结的标准操作流程（SOP）；automation = 系统自动执行的定时任务 |
| 权限 | 能否修改定时任务配置？ | 在隔离区内可修改自己的任务配置。通用区的 automation/ 模板由系统维护 |

## 4. 我的判断

**使用自动化系统的标准动作**：
1. 读本过河桥，理解模块机制和与 timers 的关系
2. 找到 automation/ 在 [CENTRAL_PATH] 下的位置
3. 定义或修改自动化任务（task/ 或 workflows/）
4. 在 timers/ 中注册对应的定时触发
5. 执行后检查 logs/ 确认执行结果
6. 在共产中心公告板登记重要变更

**配置定时任务的标准动作**：
1. 在 timers/ 下选择合适的任务类型（滚动/单个/团队）
2. 按 JSON 格式配置任务参数（type、model、interval、offset、role、actions、max_duration、stop_condition）
3. 在 automation/ 下定义 actions 对应的具体执行逻辑
4. 验证任务配置完整性后启动

**重建自动化模块的标准动作**：
1. 在 soul-doc-system/automation/ 下确认文件夹结构
2. 按模块同步维护规则，从通用区复制到各模板位置
3. 确认 tasks/、workflows/、logs/、templates/ 目录存在
4. 验证与 timers/ 的连接正常
5. 验证模块可独立运行
