过河桥 — 智能体空间

> **此过河桥是 智能体空间 — 自动化执行体的管理区域 的基因片段。**
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

## 1. 用户原话重述

用户在系统文件中建立了 智能体空间，定位为"自动化执行体的管理区域"。智能体空间 = 衣柜（存放配置）+ 大厅（运行区）。

核心设计包含四个层次：
- **衣柜+大厅架构**：衣柜存放经过验证的智能体配置模板（智能体的"衣服"），大厅是智能体"穿好马甲"运行的区域。衣柜不直接运行——复制到大厅再运行。
- **灵魂继承规则**：通用区衣柜的智能体继承主灵魂文档；子角色区衣柜的智能体继承子角色灵魂文档。智能体不创造新灵魂——它继承创建它所在位置的灵魂文档。
- **自动化应用（Automation Apps）的创建与管理**：智能体空间通过 automation app（数字人）机制实现——使用 `mcp__halo-apps__create_automation_app` 工具创建和安装自动化应用，使用 `mcp__halo-apps__get_automation_status`、`mcp__halo-apps__update_automation_app`、`mcp__halo-apps__pause_automation_app`、`mcp__halo-apps__resume_automation_app`、`mcp__halo-apps__trigger_automation_app`、`mcp__halo-apps__delete_automation_app`、`mcp__halo-apps__list_automation_apps` 管理。
- **三模式执行**：默认执行（直接使用）/ 边做边解释（使用+说明）/ 只思考不执行（分析配置但不启动）。前置思考系统（世界最强脑师-前置思考系统）是使用智能体前必须走的第一步。
- **与通用功能模块拼搭的关系**：智能体空间本身只提供"智能体管理"能力，不包含记忆/日记/产物等基础模块。拼搭时必须把配套模块（memory/、diary/、product/、sop/、workflow/）一起复制过去。

用户的原话意图是：让模型有一个标准化的"智能体 wardrobe"——把验证过的智能体配置存起来（衣柜），需要时穿上运行（大厅），运行数据归档，有价值的修改回传衣柜形成模板迭代闭环。

## 2. 理解要点

### 2.1 衣柜+大厅架构

#### 衣柜（模板仓库）

**作用**：存放经过验证的智能体配置模板（智能体的"衣服"）。

```
衣柜/
├── README.md
├── 通用智能体/                    ← 通用模板（一件通用马甲）
│   ├── README.md
│   └── 通用智能体配置.json        ← 智能体配置模板
└── [专用名称]/                    ← 专用模板（特定角色衣服）
    ├── README.md
    └── [专用配置].json
```

**规则**：
- 模板必须是**已经验证过**的配置，不是实验性的
- 每个模板有 README.md 说明：用途、参数、适用场景
- 配置文件（.json）包含完整的智能体定义
- 新增模板前必须经过验证 → 模板化 → 存入衣柜
- 衣柜的内容**不直接运行**——复制到大厅再运行

**通用智能体配置模板结构**（参考 `通用智能体配置.json`）：
```json
{
  "name": "通用智能体",
  "description": "通用任务执行智能体",
  "version": "1.0",
  "permissions": ["read", "write", "execute"],
  "sub_agents": [],
  "tools": [],
  "memory": {
    "type": "L0-L4",
    "context_path": "memory/context/",
    "daily_path": "memory/daily/"
  },
  "diary": {
    "path": "diary/"
  },
  "sop": {
    "path": "sop/",
    "required": ["世界最强脑师-前置思考系统", "顶级机会做法SOP", "超级信息SOP"]
  }
}
```

**添加新模板流程**：
1. 在大厅验证配置
2. 验证通过 → 提炼为模板
3. 存入衣柜，带 README.md 说明

#### 大厅（运行区）

**作用**：智能体"穿好马甲"运行的区域。

```
大厅/
├── README.md
├── 活跃/                          ← 正在运行的智能体
│   ├── README.md
│   └── [智能体名称]/              ← 从衣柜复制配置 + 运行数据
│       ├── README.md
│       ├── 配置.json              ← 从衣柜复制
│       └── 运行数据/              ← 运行时产生的数据
└── 历史/                          ← 已停用或归档的智能体
    ├── README.md
    └── [归档名称]/
        ├── README.md
        └── 配置.json
```

**使用流程**：
1. **挑衣服**：从 `衣柜/` 复制需要的模板到 `大厅/活跃/[智能体名称]/`
2. **微调**：在副本中根据需要微调参数
3. **穿上运行**：启动智能体
4. **记录**：运行数据存入 `运行数据/`
5. **脱衣归档**：停用后移动到 `大厅/历史/`

**关键规则**：
- 大厅的内容是衣柜模板的**副本**，不是模板本身
- 衣柜的内容**不直接运行**
- 大厅副本的修改不影响衣柜
- 如果大厅的修改有价值 → 提炼为新模板 → 回传衣柜
- 大厅历史区的内容可以随时清理

**通用-待细分**：`大厅/活跃/通用-待细分/` 是占位文件夹，从衣柜复制模板过来后将此文件夹改名为具体智能体名称。

### 2.2 灵魂继承规则

这是智能体空间最关键的设计决策——智能体不创造新灵魂，它继承创建它所在位置的灵魂文档。

| 场景 | 继承来源 | 说明 |
|------|---------|------|
| 通用区智能体（主灵魂区） | 继承主灵魂文档 | 在主灵魂文档所在区域创建的智能体 |
| 子角色区智能体 | 继承子角色灵魂文档 | 在子角色区域创建的智能体 |

**与子角色灵魂的区别**：
- 子角色灵魂有自己的灵魂文档，是完全隔离的灵魂文档文件夹
- 智能体不创造新灵魂——它继承创建它所在位置的灵魂文档
- 举例：主灵魂区创建智能体 → 智能体继承主灵魂文档；子角色A区创建智能体 → 智能体继承子角色A的灵魂文档

### 2.3 前置思考系统

**铁律**：使用智能体前必须先走 `世界最强脑师-前置思考系统`，确认这个智能体配置是最佳机会方案。

前置思考系统要求：
1. 确认这个智能体配置是最佳机会方案（有没有更好的替代？）
2. 确认是否需要智能体（还是直接用总灵魂就够了？）
3. 确认执行模式（默认执行 / 边做边解释 / 只思考不执行）

**三模式执行**：
- **默认执行**：直接使用智能体，按配置运行
- **边做边解释**：使用智能体的同时向用户说明每一步在做什么
- **只思考不执行**：分析配置但不启动智能体，给出建议

### 2.4 自动化应用（Automation Apps / 数字人）

智能体空间的另一个核心能力是通过 automation app 机制创建和管理自动化执行体（数字人）。

**创建自动化应用**：
- 使用 `mcp__halo-apps__create_automation_app` 工具
- 需要提供完整的 App Spec（JSON 格式），包含：
  - `name`：智能体名称
  - `description`：功能描述
  - `system_prompt`：系统提示词（驱动智能体的核心指令）
  - `subscriptions`：订阅触发条件（定时任务触发）
  - `config_schema`：配置参数定义
  - `permissions`：权限列表
  - `output`：输出配置

**管理自动化应用**：
- `mcp__halo-apps__list_automation_apps` — 列出所有已安装的自动化应用
- `mcp__halo-apps__get_automation_status` — 查看单个应用的完整状态
- `mcp__halo-apps__update_automation_app` — 更新应用配置（JSON Merge Patch）
- `mcp__halo-apps__trigger_automation_app` — 手动触发应用运行
- `mcp__halo-apps__pause_automation_app` — 暂停应用
- `mcp__halo-apps__resume_automation_app` — 恢复应用
- `mcp__halo-apps__delete_automation_app` — 删除应用

**与衣柜+大厅的关系**：
- 自动化应用创建成功后，可以将其配置存入衣柜作为模板
- 从衣柜复制模板到大厅 = 启动该自动化应用的实例
- 运行数据（触发记录、产出物）存入大厅的运行数据/

### 2.5 智能体空间与通用功能模块拼搭

智能体空间本身只提供"智能体管理"能力，不包含记忆/日记/产物等基础模块。拼搭时必须把配套模块一起复制过去。

**配套模块**：

| 配套模块 | 来源 | 说明 |
|---------|------|------|
| `memory/` | 从通用区复制 | 智能体的记忆（L0-L4分层） |
| `diary/` | 从通用区复制 | 智能体的日记 |
| `product/` | 从通用区复制 | 智能体的产出物存放 |
| `sop/` | 从通用区复制 | 智能体的SOP（必须包含最强脑师三合一SOP） |
| `workflow/` | 从通用区复制 | 智能体自己的工作流 |

**在其他位置创建智能体空间**：
1. 在目标位置创建 `智能体空间/` 文件夹
2. 从源位置的 `衣柜/` 复制需要的模板到新位置的 `智能体空间/衣柜/`
3. 配备配套模块（memory/、diary/、product/、sop/、workflow/）
4. 拼搭完成后，新位置的智能体空间 = 完整功能模块，可以独立运行

**灵魂继承**：新位置的智能体继承**创建它所在位置的灵魂文档**（通用区→主灵魂，子角色区→子角色灵魂）。

### 2.6 与九环架构的联动

智能体不是独立环，是⑥环（子角色中心）下的执行单元，也是⑦环（团队协作）的基础组件。

| 环 | 智能体的角色 |
|----|------------|
| ③ 主灵魂中心 | 智能体的配置/继承来自主灵魂文档 |
| ⑥ 子角色中心 | 智能体 = ⑥环下的子角色执行单元 |
| ⑦ 团队协作环 | 多个智能体 = 团队协作 |
| ④ 生产空间中心 | 智能体的产出物归生产空间 |

**快箭织网**：智能体的运行产出中，有价值的内容提炼为快箭，横切到其他环区。

### 2.7 铁律

1. **衣柜不可直接运行** — 衣柜只存模板，大厅复制使用
2. **大厅副本可修改** — 在大厅的副本中微调，不影响衣柜
3. **有价值的修改回传衣柜** — 大厅的改进经验提炼为新模板
4. **灵魂继承正确** — 智能体继承创建它所在位置的灵魂文档
5. **前置思考先行** — 使用智能体前必须先走最强脑师前置思考
6. **三模式执行** — 默认执行 / 边做边解释 / 只思考不执行
7. **配套模块齐全** — 智能体空间必须配备 memory/、diary/、product/、sop/、workflow/

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 智能体 vs 子角色 | 智能体和子角色有什么区别 | 子角色 = 灵魂文档切换；智能体 = 继承创建位置的灵魂文档的执行单元 |
| 衣柜 vs 大厅 | 什么时候用衣柜，什么时候用大厅 | 衣柜 = 存模板；大厅 = 运行实例。永远先选衣柜模板，再复制到大厅运行 |
| 自动化应用 vs 智能体 | 两者是同一概念吗 | 不同但相关：自动化应用 = halo-apps 平台的数字人；智能体 = 灵魂文档系统内的执行单元。自动化应用可以存入衣柜作为智能体模板 |
| 灵魂继承 | 智能体到底继承谁的灵魂 | 继承创建它所在位置的灵魂文档：通用区→主灵魂，子角色区→子角色灵魂 |
| 智能体是否可以跨区运行 | 在子角色区创建的智能体能否在主灵魂区运行 | 可以，但运行时会继承创建时的灵魂文档（子角色灵魂），不会切换 |
| 配套模块是否必须 | 智能体空间是否必须配备所有5个配套模块 | 必须齐全才能独立运行，缺一不可 |
| 三模式执行的选择 | 如何决定用哪种执行模式 | 默认执行 = 常规任务；边做边解释 = 用户需要了解过程；只思考不执行 = 需要分析建议但不确定是否执行 |

## 4. 我的判断

**使用智能体空间的标准动作**：
1. 走前置思考系统——确认是否需要智能体、选哪个模板、用哪种执行模式
2. 从 `衣柜/` 选择需要的模板
3. 复制到 `大厅/活跃/[智能体名称]/`（不是直接使用衣柜模板）
4. 在副本中微调参数
5. 确认配套模块齐全（memory/、diary/、product/、sop/、workflow/）
6. 确认灵魂继承正确
7. 启动智能体运行
8. 记录运行数据到 `运行数据/`
9. 有价值的修改提炼为新模板回传衣柜
10. 停用后脱衣归档到 `大厅/历史/`

**创建自动化应用（数字人）的标准动作**：
1. 准备 App Spec JSON（name、description、system_prompt、subscriptions、config_schema、permissions、output）
2. 使用 `mcp__halo-apps__create_automation_app` 创建
3. 创建成功后，将配置存入 `衣柜/` 作为模板
4. 从衣柜复制到大厅 = 启动实例
5. 使用管理工具监控和管理应用状态

**重建智能体空间的标准动作**：
1. 在 soul-doc-system/智能体空间/ 下创建 衣柜/ 和 大厅/ 两个核心目录
2. 创建通用智能体模板（通用智能体配置.json）
3. 创建大厅的活跃/ 和 历史/ 两个子目录
4. 确认灵魂继承规则文档
5. 创建本模块的过河桥（本文件）
6. 验证衣柜→大厅→运行→归档的完整流程
