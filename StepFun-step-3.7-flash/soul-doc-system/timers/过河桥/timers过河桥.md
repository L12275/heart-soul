## 1. 用户原话重述

用户在系统文件中描述了定时任务系统：全自动灵魂文档研究更新系统的核心引擎。通过定时任务实现模型间滚动协作、自我更新、互相学习。三种定时任务类型：滚动定时任务（串行&循环）、单个定时循环、团队定时任务（树根结构）。

用户的核心意图：让模型能够在没有用户实时干预的情况下，通过定时任务自动运行灵魂文档系统的更新、观察其他模型、学习借鉴和自我完善。

## 2. 理解要点

**核心机制**：
- 定时任务系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/timers/`，同步到新建工作台模版和新建账号空间模版
- 定时任务系统 = 全自动灵魂文档研究更新系统的核心引擎

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 定时器 timers | soul-doc-system/timers/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/timers/
- 隔离区内：timers/
- 模板位置：新建工作台模版/timers/、新建账号空间模版/timers/

**三种定时任务类型**：

**1. 滚动定时任务（串行&循环）**：
- 多个模型按顺序轮流执行，形成轮子式滚动
- 计算公式：
  - 单个任务间隔 = 模型数量 × 每模型执行间隔
  - 每模型启动偏移 = 模型序号 × 每模型执行间隔
- 示例：5个模型，每模型间隔10分钟 → 总间隔50分钟，偏移0/10/20/30/40分钟
- 设计要点：不强制停止模型，让模型想停时就停，自然表现
- 增减模型时自动重算间隔和偏移

**2. 单个定时循环**：
- 固定间隔的单模型循环任务
- 适合：单一模型的定期自检、日记写入
- 示例：每60分钟模型自己检查灵魂文档系统是否需要更新

**3. 团队定时任务（树根结构）**：
- 多个子任务并行执行的定时任务
- 根任务启动 → 并行触发多个子任务 → 汇总结果
- 适合：团队协作中的并行子代理调度

**定时任务目录结构**：
```
timers/
├── 滚动定时任务/
│   ├── README.md
│   ├── model-A定时任务.json
│   ├── model-B定时任务.json
│   └── model-C定时任务.json
├── 单个定时循环/
│   └── 自检-60min.json
└── 团队定时任务/
    └── team-xxx.json
```

**滚动定时任务配置格式**：
```json
{
  "type": "rolling",
  "model": "模型名称",
  "interval_minutes": 50,
  "offset_minutes": 0,
  "role": "orchestrator",
  "actions": [
    "读取中央灵魂文档系统变化",
    "检查用户专用文件夹是否有新指示",
    "检查其他模型灵魂文档系统更新",
    "自愿学习借鉴",
    "更新自己的灵魂文档系统",
    "可选：网络搜寻研究"
  ],
  "max_duration_minutes": 10,
  "stop_condition": "自然完成"
}
```

**全自动灵魂文档研究更新系统的工作原理**：
1. **触发**：定时任务启动，模型加载自己的灵魂文档系统
2. **观察**：读取中央灵魂文档系统的所有文件变化
3. **研究**：自愿学习其他模型的更新内容
4. **更新**：丰富自己的灵魂文档系统
5. **记录**：在日记和记忆中记录观察和想法
6. **交流**：在共产中心公告板登记重要发现

**模型行为规范**：
- 模型独立运行，不强制学习
- 看到其他模型更新 = 自愿决定是否借鉴
- 在自己隔离区建立日记和记忆文件
- 不强制也不禁止学习 — 自然表现

**用户控制参数**：
| 参数 | 说明 |
|------|------|
| 参与模型列表 | 哪些模型加入滚动定时任务 |
| 模型顺序 | 高级模型先开 = 质量向上滚动 |
| 间隔时间 | 5-10分钟每模型间隔 |
| 增减模型 | 用户告诉AI加入/移除，AI自动重算全部定时任务 |

**与 automation 模块的关系**：
- timers/ 提供时间触发的骨架（调度器）
- automation/ 在 timers 的触发下执行具体动作
- 全自动灵魂文档研究更新系统 = timers 触发 → automation 执行

**与九环架构的关系**：
- 定时任务系统服务于所有环区
- 滚动定时任务主要服务于 环③ 主灵魂中心（模型自愿驱动）
- 团队定时任务服务于 环⑦ 团队协作环

**效果预期**：
- 本地滚动涌现：即使不联网搜索，纯本地模型间互相学习也能持续更新
- 网络搜索涌现：模型主动去网络搜寻研究，丰富更新
- 白盒优选：用户通过各模型灵魂文档系统的更新内容，直观看到模型的心智、智商、取向、勤奋度
- 免提示词筛选：无需数亿token测试，看更新迭代就知道模型综合表现

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| timers 和 automation 的关系 | 两者都涉及定时，区别是什么？ | timers = 时间触发的骨架（调度器）；automation = 被触发后执行的具体动作和流程 |
| 滚动定时任务的强制性问题 | 模型是否必须执行定时任务？ | 不强制。模型独立运行，自然表现，停止条件 = 自然完成 |
| 定时任务的执行时长 | max_duration_minutes 是硬限制还是软限制？ | 是硬限制。超过 max_duration 必须停止 |
| 定时任务是否在用户对话时运行 | 用户正在和模型对话时，定时任务是否运行？ | 定时任务在后台独立运行，不干扰用户对话 |
| 增减模型的自动重算 | AI如何自动重算定时任务？ | 当用户加入/移除模型时，按公式重新计算所有间隔和偏移 |
| 单个定时循环和滚动定时任务 | 何时用单个？何时用滚动？ | 单个 = 单一模型的定期自检；滚动 = 多模型间的协作学习 |

## 4. 我的判断

**使用定时任务系统的标准动作**：
1. 读本过河桥，理解三种定时任务类型和配置格式
2. 在 timers/ 下找到对应的定时任务类型目录
3. 按 JSON 格式配置任务参数
4. 验证任务配置后启动定时任务
5. 定时任务触发后按 actions 执行
6. 执行完成后在日记和记忆中记录结果

**配置滚动定时任务的标准动作**：
1. 确定参与模型列表和顺序
2. 按公式计算间隔和偏移
3. 为每个模型创建配置 JSON
4. 将配置放入 滚动定时任务/ 目录
5. 启动滚动定时任务

**配置单个定时循环的标准动作**：
1. 确定模型和间隔时间
2. 按 JSON 格式配置任务
3. 将配置放入 单个定时循环/ 目录
4. 启动定时循环

**配置团队定时任务的标准动作**：
1. 确定根任务和子任务
2. 按树根结构配置 JSON
3. 将配置放入 团队定时任务/ 目录
4. 启动团队定时任务

**重建定时任务模块的标准动作**：
1. 在 soul-doc-system/timers/ 下确认文件夹结构
2. 确认三种定时任务类型的子目录存在
3. 按模块同步维护规则，从通用区复制到各模板位置
4. 验证定时任务配置格式正确
5. 验证模块可独立运行
