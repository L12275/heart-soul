## 1. 用户原话重述

用户在系统文件中描述了 Git树系统：子代理工作分支与合流管理。Git历史与远程仓库推送。属于代码工作台（production_space/代码工作台/）的核心组件。

用户的核心意图：让模型能够用 Git 的分支模型来管理子代理的并行工作，确保多子代理协作时的代码/文档隔离、质量检查和合流安全。

---

## 2. 理解要点

**核心机制**：
- Git树系统属于 环④ 生产空间中心工作台（产出驱动），具体在代码工作台下
- 代码工作台 = production_space/代码工作台/，包含 desktop/、diary/、memory/、product/、sop/、temp/、workflow/、过河桥/
- git-tree/ 是代码工作台的核心组件之一

**模块位置**：
- 规范位置：soul-doc-system/git-tree/（通用区）
- 工作台位置：production_space/代码工作台/git-tree/
- 在 git-tree/ 下有 remotes/、history/、convergence/、desktop/ 子目录

**Git树系统的结构**（来自代码工作台 Git Tree 模板）：
```
git-tree/
├── remotes/           ← 远程仓库配置
│   └── README.md      ← 远程仓库地址（GitHub: https://github.com/L12275/heart-soul 、Gitee: https://gitee.com/L12275/heart-soul）
├── history/           ← Git历史记录
├── convergence/       ← 合流记录
└── desktop/           ← Git桌面快捷方式
```

**分支命名规则**：
- 每个子代理 = 一个分支
- 分支命名格式：`agent/{名称}/{任务ID}`
- 例如：agent/研究员/20260627-001、agent/编码师/20260627-002

**合流规则**：
- 合流前必须通过质量检查
- 合流记录保存在 convergence/ 目录
- 合流 = 子代理产出被合并到主分支或目标分支

**远程推送规则**（来自 system-prompt.md）**：
- 远程仓库地址：GitHub https://github.com/L12275/heart-soul 、Gitee https://gitee.com/L12275/heart-soul
- 推送规则：只推送自己隔离区的稳定版内容（[模型名]/soul-doc-system/）
- 不推送其他模型的隔离区
- 不推送 user-zone/ 和 共产党灵魂文档系统/（这些由维护模型推送）
- 推送前读取根目录 README.md 确认仓库地址，读取 INITIAL-SEED.md 了解系统最新状态
- 你的推送 = 你对系统贡献的记录

**与团队协作的关系**：
- 团队空间/ 下的子代理通过 git-tree/ 管理各自的工作分支
- 编排师负责：任务分解、子代理分配、工作台拼搭、进度协调、质量审查、合流输出
- 每个子代理在独立分支上工作，完成后通过合流机制合并

**与九环架构的关系**：
- Git树系统服务于 环④ 生产空间中心（产出驱动）
- 服务于 环⑦ 团队协作环（编排驱动）
- 通过 Git 分支实现子代理之间的工作隔离和合流

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| git-tree 和普通 Git 的区别 | 普通 Git 管理代码，git-tree 管理什么？ | git-tree 管理的是子代理的**工作分支和合流**，不只是代码，还包括文档、产出物的并行协作 |
| 分支命名规范 | agent/{名称}/{任务ID} 中的任务ID是什么格式？ | 建议用 日期-序号 格式，如 20260627-001，方便识别时间 |
| 合流的触发条件 | 什么时候触发合流？ | 子代理完成任务并通过质量检查后，由编排师或模型自己触发合流 |
| 推送频率 | 多久推送一次远程仓库？ | 只推送稳定版。每次有重要更新后推送，不是每次对话都推送 |
| 推送范围 | 哪些内容需要推送？ | 只推送自己隔离区的 soul-doc-system/。其他模型隔离区、user-zone、共产区不推送 |

## 4. 我的判断

**使用 Git 树系统的标准动作**：
1. 读本过河桥，理解分支命名规则和合流流程
2. 确认远程仓库配置（remotes/ README.md）
3. 为当前子代理任务创建独立分支（agent/{名称}/{任务ID}）
4. 在独立分支上执行任务
5. 完成任务后执行质量检查
6. 通过质量检查 → 执行合流（记录到 convergence/）
7. 通过合流检查 → 推送到远程仓库（只推送自己隔离区的稳定版）

**创建子代理分支的标准动作**：
1. 确定子代理名称和任务ID
2. 按命名规范创建分支：`agent/{名称}/{任务ID}`
3. 在分支上初始化工作台结构
4. 在 history/ 中记录分支创建信息

**合流的标准动作**：
1. 确认子代理已完成任务
2. 执行质量检查（对照 SOP、检查产出物完整性）
3. 质量检查通过 → 执行合流
4. 在 convergence/ 中记录合流信息（时间、分支、结果）
5. 合流失败 → 回到子代理分支修复，重新检查

**重建 Git 树系统的标准动作**：
1. 在 production_space/代码工作台/git-tree/ 下确认子目录结构
2. 确认 remotes/ 中的远程仓库配置
3. 确认 history/ 和 convergence/ 的访问正常
4. 验证 Git 操作权限正常
