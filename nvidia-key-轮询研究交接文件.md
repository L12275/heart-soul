## 2.6 卷尺模式（35次强制休息）设计与实现

### 2.6.1 设计目标

- 每个密钥调用 35 次后强制休息（进入 COOLDOWN），不再接受新请求
- 自动切换到下一个密钥，直到所有密钥都 COOLDOWN
- 最早恢复的密钥重新激活，继续服务

### 2.6.2 核心参数

| 参数    | 值          | 说明         |
| ----- | ---------- | ---------- |
| 单密钥上限 | 35 次/分钟    | 留 5 次余量    |
| 冷却窗口  | 1 分钟       | 窗口结束后计数器归零 |
| 切换延迟  | 0ms        | 用完即切，不等待   |
| 回退检测  | 连续 30 秒无请求 | 流量下降时回到主密钥 |
| 主密钥数  | 2          | 默认只用 K1-K2 |

### 2.6.3 状态机

```
密钥状态：
- ACTIVE：可用，调用次数 < 35
- COOLDOWN：已满 35 次，等待窗口重置
- DEPLETED：所有密钥都 COOLDOWN

流程：
1. 优先取 K1、K2
2. K1 满 35 → COOLDOWN → 切 K2
3. K2 满 35 → COOLDOWN → 切 K3...
4. 依次扩展到 K12
5. K1 窗口恢复 → 检查流量：仍高则继续用当前密钥组，低则回退 K1
```

## 5. 弹性轮询方案（卷尺式密钥取用）

### 5.1 设计理念

- **核心原则**：一个密钥每分钟最多安全调用 35 次（留 5 次余量），必须强制休息
- **弹性扩展**：正常 2 个密钥交替轮询，流量大时动态启用第 3、4...N 个密钥
- **智能回退**：流量下降时自动回到 2 密钥模式，减少密钥消耗
- **绝对禁止**：任何理由调用已进入冷却（满 35 次）的密钥，否则会被检测为攻击式请求

### 5.2 轮询状态机

```
密钥库：[K1, K2, K3, ..., K12]（按加入顺序）

状态定义：
- ACTIVE：当前可用，调用次数 < 35
- COOLDOWN：已调用 35 次，进入强制休息（等待下一分钟窗口重置）
- DEPLETED：所有密钥都 COOLDOWN，需等待最早恢复的密钥

轮询流程：
1. 优先取用 K1、K2（固定双密钥）
2. 如果 K1/K2 都 COOLDOWN，取用 K3
3. 如果 K1-K3 都 COOLDOWN，取用 K4
4. 以此类推，直到密钥库耗尽
5. 当流量下降，检测到 K1/K2 已恢复，立即回到双密钥模式
6. 流量再次上升时，再次弹性扩展
```

### 5.3 关键参数

| 参数         | 安全值             | 说明                      |
| ---------- | --------------- | ----------------------- |
| 单密钥每分钟上限   | 35 次            | 留 5 次余量，避免触发 429        |
| 单密钥单次测试上限  | 50 次            | 仅用于验证，生产环境不超过 35        |
| 纵向间隔（请求间隔） | 待定              | 取决于实际吞吐需求，建议 ≥ 800ms    |
| 横向冷却       | 1 分钟窗口          | 记录每个密钥的首次调用时间，计算滑动窗口    |
| 弹性阈值       | 连续 2 次 COOLDOWN | 连续 2 个密钥都满 35 次，启用第 3 个 |
| 回退阈值       | 连续 30 秒无请求      | 流量下降，回到双密钥模式            |

### 5.4 安全时间阈值测试（待验证）

**目的**：找出一个密钥在 1 分钟内消耗完 35 次的最短安全时间，避免因请求过快被英伟达判定为攻击。

**测试方法**：

1. 单密钥，固定间隔从 500ms 开始，逐步加速（400ms、300ms、200ms...）
2. 每次测试 35 次，记录：
   - 总耗时（35 次调用从开始到结束的时间）
   - 是否触发 429
   - 首次 429 出现的位置（第几次调用）
3. 找到**最大成功速率**（35 次调用中 0 次 429 的最短总耗时）

**预期结果示例**：
| 纵向间隔 | 35次总耗时 | 429次数 | 结论 |
|---------|-----------|---------|------|
| 500ms | 17.5秒 | 0 | ✅ 安全 |
| 400ms | 14秒 | 0 | ✅ 安全 |
| 300ms | 10.5秒 | 0 | ✅ 安全 |
| 200ms | 7秒 | 5 | ❌ 触发限速 |

**关键指标**：

- **最小安全总耗时**：35 次调用无 429 的最短时间（如 10.5 秒）
- **最小安全单次间隔**：最小安全总耗时 ÷ 35（如 300ms）
- **最大安全 RPM**：60秒 ÷ 最小安全单次间隔（如 200 RPM，但受 35 次/分钟上限限制）

**智能脚本应用**：

- 如果检测到请求队列等待时间 > 3 秒，说明当前密钥无法及时处理
- 立即分流到下一个可用密钥（即使前一个密钥还没满 35 次）
- 分流后仍准确记录每个密钥的调用次数，确保 35 次上限不被突破

### 5.5 智能防呆设计

1. **强制冷却锁**：密钥调用 35 次后，立即标记为 COOLDOWN，后续请求自动跳过
2. **时间戳校验**：每个密钥记录 `first_call_time`，计算滑动窗口内调用次数
3. **纵向计数（无论是否被取用）**：每个密钥独立记录 1 分钟内的调用次数，即使请求被分流到其他密钥，该密钥的计数器仍正常计时（防止因未取用而漏计数）
4. **防重复触发**：同一密钥不会被连续调用 2 次（除非所有其他密钥都 COOLDOWN）
5. **流量检测**：实时统计每秒请求数，动态决定是否扩展密钥池
6. **等待时间检测**：如果请求在队列中等待超过 3 秒，触发分流
7. **日志审计**：记录每个密钥的调用时间、次数、状态、等待时长，便于排查问题

### 5.6 分流泄压机制

**触发条件**：

- 当前密钥队列等待时间 > 3 秒
- 或当前密钥响应时间 > 2 秒（服务慢）

**分流逻辑**：

1. 检查下一个可用密钥（按轮询顺序）
2. 如果下一个密钥是 ACTIVE 状态，立即分流部分请求过去
3. 分流后仍记录原密钥的调用次数（即使请求实际由其他密钥处理）
4. 原密钥达到 35 次后强制 COOLDOWN，不再接受新请求

**示例**：

- K1 当前有 30 次调用，队列等待 4 秒
- 检测到等待时间 > 3 秒，触发分流
- 新请求直接发给 K2（K2 当前 10 次，ACTIVE）
- K1 继续服务已有请求，不再接受新请求
- K1 达到 35 次后 COOLDOWN，K2 继续承担

### 5.7 使用场景示例

**场景 1：正常对话（低吞吐）**

- 用户 A 发起对话，K1 承担请求
- 1 分钟内 K1 调用 20 次，未达 35 次上限
- 继续使用 K1，无需切换

**场景 2：高频对话（中等吞吐）**

- 用户 A + B 同时对话，K1 快速达到 35 次
- 自动切换到 K2，K2 承担后续请求
- K1 进入 COOLDOWN，等待下一分钟窗口
- 如果 K2 也达到 35 次，启用 K3

**场景 3：批量任务（高吞吐 + 快速响应）**

- 系统同时处理 10 个对话，请求量大且模型响应快（< 1 秒）
- K1 在 15 秒内达到 35 次（35 × 400ms = 14 秒）
- K1 COOLDOWN，立即切换到 K2
- K2 同样快速达到 35 次，启用 K3
- 依次扩展至 K6-K12
- 当 K1 恢复（下一分钟窗口），检测到流量仍高，不立即切回 K1
- 继续使用 K6-K12，直到流量下降
- 流量下降后，逐步回到 K1-K2 双密钥模式

**场景 4：突发流量（智能分流）**

- 突然涌入 50 个请求，K1 当前 30 次，队列等待 4 秒

- 触发分流，K2（10 次）和 K3（5 次）立即加入服务

- 50 个请求在 3 个密钥间分担，每个密钥增加约 15 次

- K1 达到 35 次后 COOLDOWN，K2 达到 35 次后 COOLDOWN

- K3 继续服务剩余请求，达到 35 次后 COOLDOWN

- 启用 K4-K12 继续服务
  
  ### 5.8 异常反馈与告警机制

**反馈插入点**：

- 当检测到 429 时，在模型调用的响应信息**开头**插入异常标记
- 格式必须包含**智能脚本的文件名和路径**，方便 AI 定位检修

**标准反馈格式**：

```
[KEY-POOL-ALERT] 脚本: <脚本文件名> | 路径: <脚本绝对路径> | 密钥 K3 在第 18 次调用时触发 429，已切换至 K4。异常类型：过早限速。
```

**示例**：

```
[KEY-POOL-ALERT] 脚本: key-pool-proxy.js | 路径: A:\AI-space\key-pool\key-pool-proxy.js | 密钥 K3 在第 18 次调用时触发 429，已切换至 K4。异常类型：过早限速。
```

**告警分级**：

| 级别       | 触发条件                     | 反馈内容                                       |
| -------- | ------------------------ | ------------------------------------------ |
| INFO     | 单次过早 429（15-20 次）        | `[KEY-POOL-INFO] 脚本: key-pool-proxy.js     |
| WARNING  | 连续 2 次过早 429             | `[KEY-POOL-WARNING] 脚本: key-pool-proxy.js  |
| CRITICAL | 第 1 次调用就 429，或连续 3 次 429 | `[KEY-POOL-CRITICAL] 脚本: key-pool-proxy.js |

**AI 自主检修流程**：

1. 收到 CRITICAL 告警后，AI 从反馈中提取**脚本路径**
2. 读取脚本文件，检查该密钥的历史调用记录
3. 分析 429 发生的时间点、次数、间隔
4. 判断是"正常限速"还是"攻击式封禁"
5. 提供检修建议：
   - 正常限速：建议等待 5-10 分钟后重试
   - 攻击式封禁：建议更换密钥，或等待次日恢复
   - 密钥本身问题：建议从密钥库中永久移除

**用户通知**：

- 如果多个密钥同时出现 CRITICAL 告警，AI 主动告知用户
- 通知内容包括：异常密钥列表、可能原因、建议操作
- 通知中必须包含**脚本路径**，方便用户自行检查

### 5.8 风险与注意事项

| 风险     | 说明                           | 缓解措施                     |
| ------ | ---------------------------- | ------------------------ |
| 密钥集中调用 | 如果脚本逻辑错误，导致同一密钥被连续调用         | 强制冷却锁 + 调用次数校验 + 纵向计数    |
| IP 被封禁 | 同一 IP 多账号并发可能触发封禁            | 降低并发，增加纵向间隔，避免攻击式测试      |
| 密钥耗尽   | 所有密钥都 COOLDOWN，无可用密钥         | 等待最早恢复的密钥，或返回 503        |
| 时钟漂移   | 服务器时间不准导致滑动窗口计算错误            | 使用 NTP 同步时间，或固定窗口（整点刷新）  |
| 分流风暴   | 大量请求同时触发分流，导致多个密钥同时 COOLDOWN | 设置分流冷却时间（如 5 秒内最多分流 1 次） |
| 计数不同步  | 分流后原密钥计数与实际调用不一致             | 无论是否分流，每个密钥独立计数，不依赖实际处理者 |

## 6. 轮询策略库（通用，不限于 NVIDIA）

### 6.1 固定顺序轮询（Round Robin）

- **原理**：按固定顺序依次调用密钥，循环往复
- **适用**：密钥能力相同，请求均匀分布
- **参数**：纵向间隔固定，横向间隔 = 密钥数 × 纵向间隔
- **示例**：12 密钥，150ms 间隔，每个密钥 1.8 秒一次

### 6.2 独立冷却轮询

- **原理**：每个密钥独立计时，调用后冷却 X 秒，冷却结束可再次使用
- **适用**：密钥响应时间差异大，需要灵活调度
- **参数**：冷却时间根据实际限速调整（如 800ms = 75RPM）

### 6.3 用完即切（ exhaustion-based）

- **原理**：一个密钥连续调用直到 429 或达到安全上限，立即切换下一个
- **适用**：密钥限速不明确，需要试探性使用
- **风险**：429 切换可能已经太晚，导致请求失败

### 6.4 弹性卷尺式轮询（本文件核心方案）

- **原理**：固定双密钥为主，流量大时弹性扩展，流量小时回退
- **适用**：请求量波动大，需要平衡密钥消耗和吞吐
- **优势**：减少密钥消耗，提高利用率，避免集中调用

### 6.5 权重轮询

- **原理**：按权重比例分配请求，权重高的密钥承担更多流量
- **适用**：密钥能力不同（如付费 vs 免费，高速 vs 低速）
- **示例**：K1 权重 3，K2 权重 1，则 K1 承担 75% 请求

### 6.6 健康检查优先轮询

- **原理**：优先选择响应快、成功率高的密钥，跳过故障密钥
- **适用**：密钥质量参差不齐，需要自动容错
- **风险**：快密钥可能被过度使用，提前耗尽

### 6.7 随机轮询

- **原理**：每次请求随机选择密钥
- **适用**：密钥能力相同，需要简单实现
- **风险**：可能导致部分密钥过热，部分闲置

### 6.8 一致性哈希轮询

- **原理**：根据用户 ID 或会话 ID 哈希到固定密钥，同一用户始终用同一密钥
- **适用**：需要会话亲和性（如保持上下文）
- **优势**：减少密钥切换，提高缓存命中率

### 6.9 分层轮询（主备模式）

- **原理**：主密钥组承担正常流量，备用密钥组在主组故障时接管
- **适用**：高可用场景，需要故障转移
- **示例**：K1-K10 为主，K11-K12 为备，主组全部 429 才启用备用

### 6.10 时间片轮询

- **原理**：不同时间段使用不同密钥组
- **适用**：密钥有使用时间限制（如某些密钥仅工作日可用）
- **示例**：K1-K6 用于白天，K7-K12 用于晚上

## 7. 可行方案（按优先级）

### 7.1 方案 A：单 IP 单账号（最稳定，已验证）

- **操作**：只使用 1 个密钥，800ms 间隔串行调用
- **预期**：30-40 次/分钟，稳定无 429
- **适用**：低吞吐需求，或作为备用方案
- **风险**：低，不会触发封禁
- **验证状态**：✅ 已验证（单密钥直连 30-40 次/分钟）

### 7.2 方案 B：多 IP 分流（理论上唯一高吞吐方案，未验证）

- **操作**：
  1. 准备 12 个不同出口 IP（不同服务器/VPN 节点）
  2. 每个密钥绑定一个独立 IP
  3. 通过代理规则（Clash Verge 等）实现分流
- **预期**：12 × 40 = 480 次/分钟
- **风险**：配置复杂，需要 12 个独立 IP 资源
- **验证状态**：❌ 未验证（需要多 IP 环境）

### 7.3 方案 C：中转站 new-api（低吞吐稳定，部分验证）

- **操作**：继续使用中转站，800ms 间隔
- **预期**：37 次/分钟（基于 50 次请求测试）
- **风险**：中转站本身可能有单渠道 RPM 限制
- **适用**：接受低吞吐，不想自己维护代理
- **验证状态**：⚠️ 部分验证（50 次请求测试，但未做长时间稳定性测试）

### 7.4 方案 D：弹性卷尺式轮询（理论最优，待实现）

- **操作**：实现智能脚本，按 5.1-5.6 设计
- **预期**：2 密钥交替时 60-70 RPM，弹性扩展时可达 300+ RPM
- **风险**：逻辑复杂，需要严格防呆设计
- **验证状态**：❌ 未验证（需要 IP 封禁解除后安全测试）

## 8. 安全测试规范（必须遵守）

### 8.1 单密钥安全阈值

| 指标           | 安全值              | 危险值        |
| ------------ | ---------------- | ---------- |
| 单密钥每分钟请求数    | ≤ 35 次           | > 40 次     |
| 单密钥单次测试最大请求数 | ≤ 40 次           | > 40 次     |
| 请求间隔         | ≥ 500ms          | < 500ms    |
| 12 密钥总计每分钟   | ≤ 密钥数×40 次（单 IP） | > 密钥数×40 次 |
| 密钥切换间隔       | ≥ 200ms          | < 200ms    |

### 8.2 测试纪律

1. **禁止攻击性测试**：单次测试总请求数不得超过 40 次
2. **禁止高并发**：同一时间只允许 1 个请求在飞行
3. **禁止连续测试**：测试后等待 5-10 分钟，观察密钥状态
4. **禁止未经授权的批量操作**：任何批量测试必须提前告知并获同意
5. **IP 封禁应急**：如发现全部 429，立即停止所有测试，等待 30 分钟以上

### 8.3 密钥管理

- 12 个密钥应视为**备用资源**，同一时间只使用 1-2 个
- 定期轮换使用（如密钥 A 用 1 周，换密钥 B）
- 记录每个密钥的使用情况，避免过度使用

## 13. 轮询策略库（通用经验）

> **提示**：以下策略不限于 NVIDIA 场景，适用于任何多账号/多密钥轮询场景。无论当前验证结果如何，策略本身都有价值，应长期积累。

### 13.1 固定顺序轮询（Round Robin）

- **原理**：按固定顺序依次调用密钥，循环往复
- **适用**：密钥能力相同，请求均匀分布
- **参数**：纵向间隔固定，横向间隔 = 密钥数 × 纵向间隔
- **示例**：12 密钥，150ms 间隔，每个密钥 1.8 秒一次
- **风险**：如果某个密钥失效，会阻塞整个轮询

### 13.2 独立冷却轮询

- **原理**：每个密钥独立计时，调用后冷却 X 秒，冷却结束可再次使用
- **适用**：密钥响应时间差异大，需要灵活调度
- **参数**：冷却时间根据实际限速调整（如 800ms = 75RPM）
- **风险**：冷却时间设置不当可能导致限速或资源浪费

### 13.3 用完即切（Exhaustion-based）

- **原理**：一个密钥连续调用直到 429 或达到安全上限，立即切换下一个
- **适用**：密钥限速不明确，需要试探性使用
- **风险**：429 切换可能已经太晚，导致请求失败

### 13.4 弹性卷尺式轮询（本文件核心方案）

- **原理**：固定双密钥为主，流量大时弹性扩展，流量小时回退
- **适用**：请求量波动大，需要平衡密钥消耗和吞吐
- **优势**：减少密钥消耗，提高利用率，避免集中调用
- **风险**：逻辑复杂，需要严格防呆设计

### 13.5 权重轮询

- **原理**：按权重比例分配请求，权重高的密钥承担更多流量
- **适用**：密钥能力不同（如付费 vs 免费，高速 vs 低速）
- **示例**：K1 权重 3，K2 权重 1，则 K1 承担 75% 请求

### 13.6 健康检查优先轮询

- **原理**：优先选择响应快、成功率高的密钥，跳过故障密钥
- **适用**：密钥质量参差不齐，需要自动容错
- **风险**：快密钥可能被过度使用，提前耗尽

### 13.7 随机轮询

- **原理**：每次请求随机选择密钥
- **适用**：密钥能力相同，需要简单实现
- **风险**：可能导致部分密钥过热，部分闲置

### 13.8 一致性哈希轮询

- **原理**：根据用户 ID 或会话 ID 哈希到固定密钥，同一用户始终用同一密钥
- **适用**：需要会话亲和性（如保持上下文）
- **优势**：减少密钥切换，提高缓存命中率

### 13.9 分层轮询（主备模式）

- **原理**：主密钥组承担正常流量，备用密钥组在主组故障时接管
- **适用**：高可用场景，需要故障转移
- **示例**：K1-K10 为主，K11-K12 为备，主组全部 429 才启用备用

### 13.10 时间片轮询

- **原理**：不同时间段使用不同密钥组
- **适用**：密钥有使用时间限制（如某些密钥仅工作日可用）
- **示例**：K1-K6 用于白天，K7-K12 用于晚上

### 13.11 最小负载优先轮询

- **原理**：选择当前调用次数最少的密钥
- **适用**：需要均匀分布负载
- **风险**：需要实时统计，增加复杂度

### 13.12 响应时间优先轮询

- **原理**：选择响应时间最短的密钥
- **适用**：对延迟敏感的场景
- **风险**：快密钥可能被过度使用

### 13.13 失败隔离轮询

- **原理**：密钥连续失败 N 次后暂时隔离，一段时间后恢复
- **适用**：密钥质量不稳定，需要自动隔离故障
- **风险**：隔离阈值设置不当可能导致误杀

### 13.14 滑动窗口限速轮询

- **原理**：记录每个密钥最近 N 秒的请求时间戳，超过阈值则跳过
- **适用**：需要精确控制 RPM
- **风险**：需要存储时间戳，内存消耗随请求数增加

### 13.15 固定窗口限速轮询

- **原理**：按固定时间窗口（如每分钟整点）统计请求数，超过则跳过
- **适用**：简单实现，不需要记录时间戳
- **风险**：窗口边界可能出现突发请求

### 16.4 代码缺陷记录

**缺陷1：setInterval中cooldownUntil清除逻辑存在但可能有竞态条件**

setInterval(2000ms)中同时做两件事：

```
if (now - keyStates[i].minuteWindowStart >= KEY_WINDOW_MS) {
  keyStates[i].minuteCalls = 0;
  keyStates[i].minuteWindowStart = now;
  keyStates[i].active429 = false;
}
if (keyStates[i].cooldownUntil > 0 && now >= keyStates[i].cooldownUntil) {
  keyStates[i].cooldownUntil = 0;
}
```

window重置和cooldownUntil归零在同一个循环内、同一毫秒级时间点执行。
但stats接口可能在此setInterval两次执行之间被调用，此时：

- minuteCalls已归零（上轮setInterval已处理）
- cooldownUntil已归零（上轮setInterval已处理）
- 但active429可能还未被清除（如果window到期检查已过但cooldownUntil归零还没执行）

实际上两条if是顺序执行的，不是原子操作。但因为在同一个2000ms间隔内，竞态窗口很小。

**缺陷2：K5的active429=true + minuteCalls=0 的矛盾**

stats快照中K5的minuteCalls=0但active429=true，这在当前代码逻辑下**不应发生**。
可能原因：

- 数据在setInterval两次循环之间被捕获，处于中间状态
- 或K5的window在捕获瞬间刚好到期，minuteCalls被重置但active429还没被重置

**结论：** 这两个缺陷不影响主要判断。核心问题仍然是IP封禁。
