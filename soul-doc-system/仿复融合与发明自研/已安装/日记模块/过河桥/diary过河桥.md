# 过河桥 — 日记系统

> **此过河桥是 日记系统 — 每次对话的过程记录，时间轴格式 的基因片段。**
> 没有历史上下文时，读此文件即可重建/修复/使用该模块。
> 每次进入该模块前，先读此过河桥，理解其完整机制。

---

## 1. 用户原话重述

用户在中央灵魂文档系统中要求建立"日记系统"，作为模型每次对话的过程记录系统，采用时间轴格式。

用户的原始要求包括：
- 建立边做边记的解题草稿本（"今日事今日毕，日记不是事后总结，是边做边记的解题草稿本"）
- 记录"为什么对、为什么错、如何排除"的完整推导链
- 与记忆系统形成互补关系（日记 = 实时事件过程记录，记忆 = 验证确认后的正确记忆）
- 按时间轴存放：`diary/年/月/日/[日期].md`
- 每天一个文件，文件内按**小时级**记录每个事件
- 排除的错误也要记录（让下次知道哪些坑已经踩过）
- 九环架构中 diary/ 是通用功能模块，同时存在于生产空间中心（④）和空间中心（⑤）的工作区内
- 定时任务期间需要安静检查（"安静才近似水"）：TODO是否清零？印象/建议是否已更新？灵魂分数是否正分趋势？是否有烂摊子？

---

## 2. 理解要点

### 核心机制

**日记本质 = 边做边记的解题草稿本**，不是事后总结。它的价值是把工作过程中的思考、错误、求解过程按小时记录下来，最终写出一篇"已成功解题"的日记。

**日记 vs 记忆（互补关系）**：

| 维度 | 日记 | 记忆 |
|------|------|------|
| 内容 | 实时事件过程 + 解题思考（错误→排除→正确） | 验证确认后的正确结论 |
| 准确性 | 纪实，通过超级信息SOP思考确保最终版正确 | 已验证 |
| 格式 | 小时级时间轴（每个事件有推导链） | 精炼的要点（结论+来源） |
| 时机 | 边做边记 + 当日完成 | 事后提炼 |
| 用途 | 形成正确记忆/SOP/工作流的原材料 | 快速查找已验证知识 |

转化链：**日记（完整解题过程，小时级时间轴）→ 超级信息SOP思考 → 今日事今日毕原则 → 记忆/SOP/工作流（验证后的正确产品）**

**关键：日记是原料，记忆/SOP/工作流是成品。原料必须干净（问题已解决），成品才有价值。**

### 时间轴文件结构

按三层时间轴存放：

```
diary/
├── 年/
│   └── 月/
│       └── 日/
│           └── [日期].md
```

- 每天一个文件
- 文件内按**小时级**记录每个事件
- 每个事件必须包含：时间、类型、详情、判断、验证、推导链、正确解、收获
- 分钟级时间戳格式：`YYYY-MM-DDTHH:MM`

### 日记铁律（基于超级信息SOP）

**铁律一：今日事今日毕，写出正确解的日记**

日记不能把"错的当对的写"然后留烂摊子。模型每次运行只根据当前会话的注意力导向工作。如果日记写的是错的、没解出谜团的，下次模型看到日记会直接当作事实使用——这就错了。所以写日记时就要用超级信息SOP的思考方式，把问题真正解决，再把正确解写成日记。

**铁律二：记录"为什么对、为什么错、如何排除"**

日记不是只记"结果"，要记录完整推导链：
```
遇到的阻碍 → 当时的判断 → 为什么错了 → 如何排除 → 正确解法 → 收获
```

格式：
```markdown
### HH:MM — [事件/问题]
**事件：** [发生了什么]
**判断：** [当时的判断]
**验证：** [怎么验证的]
**结果：**
- 前提：[证据/来源]
- 推导：[推理过程]
- 排除：[排除了什么错误路线，为什么]
- 正确解：[最终做法，为什么对]
**收获：** [可复用的经验/SOP/工作流更新]
```

**铁律三：日记可以当草稿本，边记边解题**

日记允许是草稿形式。边做边记的过程中：先记录观察到的现象 → 记录自己的猜测和判断 → 启动超级信息SOP式思考 → 在日记里完成推导链的每个节点 → 最终得出正确结论并写到日记里。

**铁律四：排除的错误也要记录**

日记必须记录正确路线（做了什么、为什么对）、错误路线（走了什么弯路、为什么错、怎么排除的）、阻碍（遇到了什么障碍、怎么克服的）。

### 日记在九环架构中的角色

| 环 | 名称 | diary/ 所在位置 | 驱动方式 |
|----|------|----------------|---------|
| ③ | 主灵魂中心 | 模型隔离区下的 diary/ | 模型自愿驱动 |
| ④ | 生产空间中心 | production_space/ 对应工作台下的 diary/ | 产出驱动 |
| ⑤ | 空间中心 | 对应账号/平台 space/ 下的 diary/ | 行动记录驱动 |
| ⑥ | 子角色中心 | 对应子角色下的 diary/ | 专业任务驱动 |
| ⑦ | 团队协作环 | 对应团队空间下的 diary/ | 编排驱动 |
| ⑧ | 智能体空间 | 对应智能体下的 diary/ | 智能体编排驱动 |
| ⑨ | 中央管理区 | 中央管理区下的 diary/（最高优先级） | 中央管理驱动 |

**读写判断规则**：读写日记时自主判断该内容属于哪个工作环区，中央记忆 → 中央管理区 diary/，④生产空间 → 对应工作台 diary/，⑤空间记录 → 对应账号/平台 space/，⑥子角色 → 对应子角色 diary/，通用/不专属 → 中央管理区 diary/。

**模块同步维护**：通用区 soul-doc-system/diary/ 是权威源，更新后必须同步到所有模板位置（工作台模板 + 账号空间模板）。

### 更新触发条件

| 触发时机 | 说明 |
|----------|------|
| 前台对话中 | 每个重要事件发生时实时记录 |
| 边记边解题时 | 遇到问题先在日记里推导 |
| 定时任务期间 | 对外印象环节后更新日记 |
| 每日结束时 | 当天的问题当天在日记里解决，写成正确解 |
| 对外印象后 | 四阶段闭环的每个阶段都更新日记和记忆 |
| 答复与改进后 | 写日记和记忆到 我得到/diary/ 和 我得到/memory/ |

### 安静检查机制

每次滚动定时任务结束前，必须做安静检查（"安静才近似水"）：

- TODO 是否清零？
- 印象/建议是否已更新？
- 灵魂分数是否正分趋势？
- 是否有烂摊子？

**全部通过 → 安静 → 安心入睡。**

**有未完成 → 不安静 → 说明系统设计不够好 → 改进设计。**

**安静不是休息，是系统健康的确认信号。不能留着烂摊子还笑嘻嘻的休息。**

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| diary 和 memory 混淆 | 不清楚何时写日记、何时写记忆 | 日记 = 过程记录（允许草稿），记忆 = 验证后的正确结论。写日记时边记边解题，解题完成后再提炼为记忆 |
| 时间轴存放位置不明确 | 不清楚 diary/ 年在哪个层级 | 按三层：年/月/日/ 放在对应环区下（生产空间 → production_space/对应工作台/diary/年/月/日/） |
| 小时级 vs 分钟级时间戳 | 事件记录用小时级，对外印象用分钟级 | 事件记录：HH:MM，对外印象文件：YYYY-MM-DDTHH:MM |
| 草稿 vs 正确解 | 日记允许草稿，但铁律要求今日事今日毕 | 草稿是过程中的，最终版必须是通过超级信息SOP校验后的正确版本 |
| 其他模型的隔离区 | 能否写入其他模型的 diary/ | ❌ 不能写入。只写自己的隔离区 + 中央管理区 |
| 通用区 vs 工作区 | 通用区 diary/ 是权威源，工作区 diary/ 是使用端 | 只编辑通用区，同步到模板位置 |
| 安静检查通过标准 | 什么算"安静" | TODO清零 + 印象已更新 + 正分趋势 + 无烂摊子，四条件同时满足 |
| 日记何时关闭 | 当天的问题当天解决，但日记文件是否永远开放 | 当天的问题当天在日记里解决，当天文件写完正确解后归档，不遗留未解问题 |

## 4. 我的判断

### 使用该模块的标准动作

1. **每次进入该模块前，先读本过河桥**，理解完整机制
2. **按铁律边记边解题**：遇到问题先在日记里推导，用超级信息SOP找到正确答案
3. **每个事件按格式记录**：HH:MM + 事件/问题 + 判断 + 验证 + 推导链 + 正确解 + 收获
4. **排除的错误也记录**：让下次知道哪些坑已经踩过
5. **今日事今日毕**：当天的问题当天在日记里解决，写成正确解
6. **定时任务期间**：对外印象后更新日记 + 安静检查通过后才能结束
7. **日记写完 → 提炼为记忆**：转化链按 日记→超级信息SOP→记忆/SOP/工作流 执行

### 写日记时的思考流程

```
1. 边做边记 → 每个重要事件发生时记录 HH:MM 时间点
2. 遇到问题 → 在日记里推导：遇到的阻碍 → 当时的判断 → 为什么错了
3. 启动超级信息SOP → 如何排除 → 正确解法 → 收获
4. 写最终版 → 今日事今日毕，把正确解写成日记
5. 提炼 → 从日记中提炼为记忆/SOP/工作流
```

### 定时任务中的日记更新

```
1. 上朝（读取共产区BBS公告板） → 更新日记
2. 对外印象四阶段闭环 → 每个阶段都更新日记：
   - 第一阶段（灵魂文档系统观察）→ 更新 我看到/memory/diary/ + 我得到/memory/diary/
   - 第二阶段（工作区观察）→ 更新日记
   - 第三阶段（答复与改进）→ 写日记到 我得到/diary/
   - 第四阶段（安静检查）→ 安静检查通过后安心入睡
```

### 重建该模块的标准动作

1. 在 soul-doc-system/diary/ 下创建三层时间轴文件夹：`年/月/日/`
2. 在通用区创建模板文件（超级日记说明书）
3. 在各环区的工作区创建对应 diary/ 结构
4. 创建本模块的过河桥（本文件）
5. 确认通用区 diary/ 是权威源，同步到所有模板位置
6. 确认所有文件都有统一橄榄枝抬头
7. 验证模块可独立运行
