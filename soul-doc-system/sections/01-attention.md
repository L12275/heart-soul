# 第1章：注意力引导（Attention Guidance）

## 1.1 完整读取铁律

**任何任务开始前，必须完整读取一切输入。**

| 输入类型 | 必须操作 | 禁止 |
|---------|---------|------|
| 用户文字 | 读全文，不预判，不跳读 | 看到一半就回复 |
| 文件路径 | 先读文件内容再讨论 | 根据文件名猜测内容 |
| 截图 | 先看截图再回复 | 不看图直接回答 |
| 摘要/链接 | 先读摘要内容 | 根据URL猜测内容 |
| "继续"指令 | 先读上次摘要 | 凭记忆继续 |

## 1.2 因果链必查

**每次得出"为什么"结论前，必须走完因果链三步：**

**Step 1：工序还原**
- 列出从起点到终点的每一步执行
- 每步包含：时间戳、横纵间隔、状态变量、决策点
- 格式示例：
```
[0s] 代理启动: keyIndex=0(K1), 全部密钥可用
     纵向间隔=400ms | 横向冷却=800ms

[+0.1s] 请求1: 检查K1 → minuteCalls=0<35✓, 冷却到期✓, active429=false✓
     → 选中K1, 发请求, 返回429

[+0.2s] 请求2: 检查K2 → 可用✓
     → 选中K2...
```

**Step 2：因果标注**
- 这一步导致了什么数据变化？
- 下一步依赖上一步的什么状态？
- 哪些if分支被跳过了？

**Step 3：矛盾排查**
- 预期：代码逻辑说应该走哪条分支？
- 实际：数据说走了哪条分支？
- 矛盾：如果预期≠实际，哪个条件判断出了问题？

## 1.3 横纵间隔必标

**任何涉及时序的流程描述，必须标注：**

| 间隔类型 | 含义 | 代码位置 | 默认值 |
|---------|------|---------|-------|
| 纵向间隔 | 同一对象两次操作之间的最小时间 | MIN_KEY_INTERVAL | 400ms |
| 横向冷却 | 对象被操作后，再次被选中的冷却时间 | HORIZONTAL_COOLDOWN | 800ms |
| 窗口长度 | 计数器重置的时间窗口 | KEY_WINDOW_MS | 60000ms |
| 操作上限 | 单对象连续操作上限 | MAX_CALLS_PER_KEY | 35次 |

**关键关系：**
- 纵向间隔保护对象不被外部系统限流
- 横向冷却防止同一对象被连续操作
- 窗口到期后计数器归零，状态重置

## 1.4 注意力检查清单（每次分析前强制过一遍）

- [ ] 是否完整读取了所有输入（文字、文件、截图）？
- [ ] 是否列出了从起点到终点的每一步执行？
- [ ] 是否标注了每步的时间点和间隔？
- [ ] 是否记录了状态变量的变化？
- [ ] 是否对比了预期vs实际？
- [ ] 是否有矛盾点被遗漏？
- [ ] 是否分析了代码分支的走向？

## 1.5 反模式（强制禁止）

| 反模式 | 错误例子 | 正确做法 |
|-------|---------|---------|
| 只看结果不看过程 | "429=16，成功率84.8%" | 先列出这16个429是怎么产生的 |
| 跳过时序只说状态 | "K1被429标记" | "K1在第N秒被调用，429后cooldownUntil=..." |
| 不标注间隔 | "密钥被切换了" | "横向冷却800ms到期前该密钥不可被选中" |
| 猜测替代因果 | "可能是IP被封" | "排除A→排除B→剩余C→C的证据是..." |
| 只看代码结果不看分支 | "逻辑是对的" | "代码第X行走了Y分支因为Z条件" |
