第3章：工作流SOP（Workflow Standard Procedures）

## 3.1 数据因果链SOP（Data-Causality SOP）

**触发条件：**
- 测试数据出现异常（429增多、成功率下降、分布异常）
- 数据与预期不符
- 任何涉及"为什么"的追问
- 任何涉及时序、间隔、状态流转的分析

**执行步骤：**

### Step 1：完整工序还原
列出从起点到终点的每一步执行，包含：
- 时间戳（相对时间或绝对时间）
- 横纵间隔（数值和代码位置）
- 状态变量（每个对象的完整状态）
- 决策点（为什么选择这个分支）

### Step 2：因果标注
对每一步回答：
- 这一步导致了什么数据变化？
- 下一步依赖上一步的什么状态？
- 哪些代码分支被跳过了？

### Step 3：矛盾排查
对比实际vs预期：
- 预期：代码逻辑说应该走哪条分支？
- 实际：stats/日志显示走了哪条？
- 矛盾：哪个条件判断出了问题？

## 3.2 测试执行SOP（Test Execution SOP）

**测试前：**
1. 确认代理进程运行中（netstat检查端口）
2. 确认密钥可用（curl单请求测试）
3. 确认测试脚本逻辑正确（读一遍脚本）
4. 确认结果文件路径

**测试中：**
1. 客户端发送请求（按设定间隔）
2. 代理接收→排队→分流→调用上游→返回结果
3. 记录每个密钥的调用次数、429次数、响应时间
4. 每N个请求打印进度

**测试后：**
1. 保存完整结果到JSON文件
2. 打印每个密钥的累计调用分布
3. 打印状态码分布（200/429/其他）
4. 打印响应时间统计（avg/min/max）
5. 分析卷尺切换是否正常（哪个密钥用了多少次，何时切的）

## 3.3 代码修改SOP（Code Modification SOP）

**修改前：**
1. 读完整文件内容
2. 理解当前逻辑（每行每分支）
3. 明确修改目标和范围
4. 备份原文件或确认可回滚

**修改中：**
1. 用Edit工具精确替换，不手动改
2. 修改后读一遍确认正确
3. 测试验证（运行、检查输出）

**修改后：**
1. 重启进程加载新代码
2. 发送测试请求验证
3. 检查stats确认状态正确

## 3.4 数据记录SOP（Data Recording SOP）

**必须记录的内容：**
- 总请求数、成功数、429数、错误数
- 每个密钥的：累计调用、成功、429、minuteCalls、active429状态
- 响应时间：平均、最小、最大
- 发送速率和间隔
- 代理配置参数

**存储位置：**
- 测试结果 → `test-*-result.json`
- 研究结论 → `docs/nvidia-key-轮询研究交接文件.md`
- 临时数据 → 对话中展示

**标注要求：**
- 所有测试数据标注时间
- 所有"不可信"数据明确标注原因
- 所有"待验证"数据明确标注验证条件

## 3.5 IP封禁处理SOP

**发现全429后的处理流程：**

1. 检查stats：是否所有密钥active429=true？
   - 如果是：等待5分钟让window重置
   - 如果否：检查K6-K12是否0调用但仍429

2. 如果K6-K12（从未调用过的密钥）也429：
   → 判定为IP级限流
   → 解决方案：改proxy上游走中转站，或切换Clash Verge节点

3. 如果只有部分密钥429：
   → 密钥级限流
   → 等待window重置或手动清除active429

4. 恢复验证：
   → curl单请求测试 → 200才继续
   → 不200不继续任何测试
