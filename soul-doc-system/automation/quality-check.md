# 质量检查清单（Quality Check）

> 用于AI在写入section和合并到master前的自我检查。
> 每次写入/合并前必须逐项过一遍。

---

## 写入section前的检查

### 规则质量
- [ ] 规则是否具体（有明确的触发条件和执行动作）？
- [ ] 规则是否不含模糊词（"可能"、"大概"、"一些"）？
- [ ] 规则是否是通用规范（不包含具体任务内容）？
- [ ] 规则是否不与已有规则冲突？
- [ ] 规则是否不会误导AI（不会让AI把规则当作任务执行）？

### 格式质量
- [ ] 是否使用了标准格式（触发条件+执行动作+检查方式）？
- [ ] 是否标注了状态（待验证/已验证）？
- [ ] 是否标注了发现时间和场景？
- [ ] 是否使用了正确的标题层级？

### 安全性
- [ ] 此规则是否应该直接写入section（还是需要先验证）？
- [ ] 此规则是否可能被未来AI误解？
- [ ] 此规则是否改变了核心约束？

---

## 合并到master前的检查

### 冲突检查
- [ ] 新规则是否与现有规则冲突？
- [ ] 新规则之间是否互相冲突？
- [ ] 新规则是否与master-soul.md的核心理念冲突？

### 重复检查
- [ ] 新规则是否已在master中有等效内容？
- [ ] 如果等效，是更新还是补充？

### 结构检查
- [ ] 合并后章节顺序是否正确？
- [ ] 合并后章节是否完整（无遗漏）？
- [ ] 合并后是否有空章节或标记为TODO的章节？

### 质量检查
- [ ] 所有新规则是否已通过section阶段的验证？
- [ ] 所有新规则是否表述精确？
- [ ] 所有新规则是否无模糊词？

### 最终验证
- [ ] 读一遍合并后的master-soul.md，确认可读
- [ ] 确认版本号已更新
- [ ] 确认changelog已记录

---

## 常见问题检查

| 问题 | 检查方法 |
|------|---------|
| 规则太模糊 | 问：AI看了这条规则后，知道具体要做什么吗？ |
| 规则与任务混淆 | 问：这条规则离开当前任务后还适用吗？ |
| 规则相互冲突 | 问：如果同时触发两条规则，AI该听谁的？ |
| 规则过于具体 | 问：这个规则能在其他场景下复用吗？ |
| 规则过度概括 | 问：AI看了这条规则后，知道在什么情况下触发吗？ |
