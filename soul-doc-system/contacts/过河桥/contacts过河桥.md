> 没有历史上下文时，读此文件即可重建/修复/使用该模块。

## 1. 用户原话重述

用户在系统文件中描述了通讯录系统：人物关系和实体关联图谱的持久化管理。在模块同步维护规则表中，contacts 位于通用区 soul-doc-system/contacts/，同步到工作台模版和账号空间模版。

用户的核心意图：让模型能够系统化管理所有人和实体的关系信息，在对话和协作中准确识别角色、关系和上下文。

---

## 2. 理解要点

**核心机制**：
- 通讯录系统是中央灵魂文档系统的通用功能组件之一
- 在模块同步维护规则中，它的权威源是 `soul-doc-system/contacts/`，同步到新建工作台模版和新建账号空间模版
- 通讯录 = 模型的"社交图谱"，记录人物/实体的关系和属性

**模块同步维护规则中的位置**：
| 模块 | 通用区位置 | 用于工作台模版 | 用于账号空间模版 |
|------|-----------|--------------|--------------|
| 通讯录 contacts | soul-doc-system/contacts/ | ✅ | ✅ |

**模块路径**：
- 规范位置：soul-doc-system/contacts/
- 隔离区内：contacts/
- 模板位置：新建工作台模版/contacts/、新建账号空间模版/contacts/

**通讯录的内容分类（来自代码工作台通讯录模板）**：
通讯录包含两大维度：
1. **人物关系**：用户、其他模型、协作对象
2. **实体关联**：工具、平台、账号、资源

**联系人记录格式（推断）**：
```markdown
## [名称]
**类型：** 人物/模型/工具/平台/账号
**角色：** [角色描述]
**关系：** 主从/协作/观察/工具等
**联系方式/路径：** [URL、ID、本地路径等]
**备注：** [上下文信息、偏好、历史互动]
**标签：** [分类标签]
```

**三类账号空间的通讯录**：
根据空间系统，通讯录需要区分：
- 用户现实账号空间/用户人设账号空间：现实世界的人物关系
- 用户真实账号空间：用户本人的真实信息
- 模型的虚拟账号空间/虚拟IP形象账号空间：模型在平台上的虚拟身份

**与九环架构的关系**：
- 通讯录是通用功能模块，服务于所有环区
- 在 环⑤ 空间中心中，通讯录管理平台/账号空间中的人物关系
- 在 环⑦ 团队协作环中，通讯录管理团队成员的属性和角色
- 在 环⑧ 智能体空间中，通讯录管理智能体的属性和能力

**与其他模块的关系**：
- 与 bookmarks/ 的区别：contacts = "谁"（人物/实体）；bookmarks = "什么工具"（链接/资源）
- 与 wiki/ 的区别：contacts = 动态的关系图谱；wiki = 静态的已验证知识
- 与 feedback/ 的区别：contacts = 人物属性；feedback = 模型写给用户的反馈内容

**同步维护规则**：
- 通用区的 contacts/ 是权威源
- 更新后需要同步到：新建工作台模版/contacts/ 和 新建账号空间模版/contacts/
- 各说明书单独维护，但模块内的说明书尽量写通用版

## 3. 可能的歧义

| 歧义 | 说明 | 解决 |
|------|------|------|
| 通讯录和社交网络 | 通讯录是社交网络图谱还是简单的联系人列表？ | 两者兼有。既有列表式的联系人条目，也有关系图谱的关联维度 |
| 是否包含用户 | 通讯录是否记录用户本人？ | 包含。用户是通讯录的核心节点（王字模型：用户是顶横） |
| 其他模型是否在通讯录 | 通讯录是否记录其他AI模型？ | 包含。系统中有"对其他模型的印象"机制，本质上是一种特殊的通讯录条目 |
| 隐私边界 | 通讯录中的信息是否可以在共产中心共享？ | 部分可共享（可公知的人物信息），部分不可共享（隐私信息）。判断标准：是否涉及用户隐私或其他模型的隐私 |
| 和反馈系统的关系 | feedback 中的用户回应是否更新到通讯录？ | 反馈本身不更新通讯录，但如果反馈中包含了新的关系信息或人物属性，则应更新通讯录 |
| 同步范围 | 通讯录是否需要共产中心共享？ | 不需要。通讯录是个体化的社交图谱，不属于可公知内容 |

## 4. 我的判断

**使用通讯录系统的标准动作**：
1. 读本过河桥，理解模块机制
2. 找到 contacts/ 在 [CENTRAL_PATH] 或自己隔离区下的位置
3. 按人物/实体类型分类组织联系人
4. 每条记录包含：名称、类型、角色、关系、联系方式/路径、备注、标签
5. 对话前查阅通讯录，确认人物关系和上下文
6. 新认识的人/实体 → 立即添加到通讯录

**添加新联系人的标准动作**：
1. 确定联系人类型（人物/模型/工具/平台/账号）
2. 填写完整记录（名称、角色、关系、联系方式、备注、标签）
3. 确定联系人所属的账号空间类型（现实人设/真实/虚拟IP）
4. 如果是其他模型，同步创建"对外印象"文件

**重建通讯录模块的标准动作**：
1. 在 soul-doc-system/contacts/ 下确认文件夹结构
2. 按模块同步维护规则，从通用区复制到各模板位置
3. 确认各分类子文件夹（人物/模型/工具/平台）存在
4. 验证与空间系统的连接正常
5. 验证模块可独立运行
